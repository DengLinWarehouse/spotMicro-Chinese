# 备份策略说明

## 1. 结论

把香橙派上的 `ros_noetic_ws` 保存到远程仓库，这个思路 **可行**，而且适合作为后续其他机器狗环境的标准做法。

但要注意：

- 远程仓库保存的应当是 **可恢复的源码环境**，不是当前机器的完整运行态快照。
- 真正值得进入 Git 的内容，是 `src/`、补丁、依赖清单、恢复脚本和说明文档。
- `build/`、`devel/`、`install/` 是机器相关、编译相关目录，不适合提交。

## 2. 为什么不建议直接提交 build/devel/install

原因主要有 4 个：

1. 这些目录体积大，推送和拉取都慢。
2. 它们依赖当前机器的编译器、Python、CMake、系统库路径。
3. 换一台香橙派或系统微调后，旧产物很容易失效。
4. 一旦把这些目录混进版本库，后面排错会非常困难。

## 3. 对本项目最合适的备份对象

### 必须备份

- `ros_noetic_ws/src/`
- `spotmicro_ws/src/`
- 对第三方包做过的本地补丁
- 依赖安装清单
- 当前环境说明文档
- 恢复脚本

### 建议额外备份

- `tar.gz` 源码压缩包
- `~/.bashrc` 中与 ROS 相关的 source 顺序说明
- 关键系统信息：Ubuntu 版本、内核、Python 路径、CMake 版本、GCC 版本

### 不建议提交到 Git

- `build/`
- `devel/`
- `install/`
- `~/.ros/log`
- 大体积临时压缩包

## 4. 针对当前 Orange Pi 实际过程的建议

根据本次真实部署过程，建议统一采用双工作区：

- `~/Desktop/SpotMicro/ros_noetic_ws`
- `~/Desktop/SpotMicro/spotmicro_ws`

备份时也按这两个工作区分别保存，不要重新混回一个目录。

## 5. 推荐的长期方案

### 方案 A：Git 远程仓库 + 本地压缩包

最推荐。

- Git 远程仓库负责日常版本管理
- `archives/` 负责阶段性全量源码快照

### 方案 B：仅做整目录打包

可以应急，但不利于长期维护。

### 方案 C：系统镜像 + Git 双保险

最稳，适合之后批量复制到多台香橙派。

## 6. 远程仓库恢复时的正确思路

恢复流程应当是：

1. 安装系统依赖
2. 拉取备份仓库
3. 还原 `ros_noetic_ws/src`
4. 重新编译 `ros_noetic_ws`
5. 还原 `spotmicro_ws/src`
6. 重新编译 `spotmicro_ws`
7. 校验 `roscore`、`rospack`、关键包查找结果

而不是：

- 直接复制旧的 `build/` / `devel/`
- 直接把旧环境原样覆盖到新机器

## 7. 与本次实际问题对应的备份价值

本次部署过程中已经暴露出一些典型风险：

- Conda 污染 ROS Python 解释器
- `defusedxml` 缺失导致 `roscore` 无法启动
- `CATKIN_ENABLE_TESTING` 未关闭时触发 gmock/gtest 问题
- 单个包需要手工补丁才能适配 Ubuntu 22.04 / Noetic / C++17
- SpotMicro 业务包不应混入 ROS base 源码工作区

这些内容如果只存在于终端记录里，很容易丢失；一旦写入备份仓库文档和脚本，后续迁移就会稳定很多。

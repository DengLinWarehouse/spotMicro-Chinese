# 自动探索转向步态逐渐混乱问题记录与修复方案

## 1. 问题概述

在 `auto_explore_mapping` 自动探索建图实机测试中，机器人在检测到前方障碍并准备转向时，往往会先保持一小段时间的有序步态，然后四条腿的时序逐渐变乱，最终表现为：

1. 某一侧腿被持续压低
2. 身体在转向过程中出现明显扭摆
3. 步态从“还能走”逐渐演变为“越走越乱”
4. 严重时会侧翻

该问题不是单纯的“速度过快”或“避障阈值不合理”，而是**自动转向命令进入后，底层步态节拍在运行中被动态改写，导致相位连续性被破坏**。

---

## 2. 现场现象记录

### 2.1 触发条件

- 机器人处于自动探索建图模式
- 前方检测到障碍物，自动探索器从 `FORWARD` 进入 `TURN_LEFT` / `TURN_RIGHT`
- 转向开始后的前几个 gait cycle 仍基本正常
- 继续转向后，腿序逐渐失配

### 2.2 实机表现

- 转向开始时还能维持步态
- 随着转向持续，步态节拍逐渐“散掉”
- 左右腿支撑相位不再稳定
- 某一侧腿压得越来越低
- 最终出现侧翻

### 2.3 风险判断

该问题属于高风险实机问题：

- 风险等级：高
- 影响对象：舵机、连杆、3D 打印件、整机姿态稳定性
- 可能后果：
  - 舵机堵转或高负载发热
  - 连杆/支架冲击
  - 机身侧翻
  - 后续标定被扰乱

---

## 3. 根因分析

## 3.1 关键结论

当前版本 `spot_micro_motion_cmd` 在 `walk` 状态中，每个控制周期都会根据当前速度命令重新生成一套 gait 参数。

问题在于：

1. `ticks_` 是连续累加的
2. `phase_length` / `swing_ticks` 会随命令变化而变化
3. 相位索引用 `ticks_ % phase_length`
4. 当 `phase_length` 在转向过程中变化时，相位会发生跳变

结果就是：

- 前几步还能走
- 一旦转向命令持续作用，phase timing 被不断改写
- swing / stance 的相位映射逐渐失真
- 四条腿的步态时序开始混乱

## 3.2 对应代码位置

核心位置在：

- `software_orangepiaipro/spot_micro_motion_cmd/src/smfsm/spot_micro_walk.cpp`

关键逻辑：

```cpp
SpotMicroNodeConfig gait_smnc = buildAdaptiveGaitConfig(smnc_, cmd);
updatePhaseData(gait_smnc);
```

其中：

- `buildAdaptiveGaitConfig(...)` 会根据当前命令幅值动态修改：
  - `swing_ticks`
  - `stance_ticks`
  - `phase_ticks`
  - `phase_length`
- `updatePhaseData(...)` 又会用新的 `phase_length` 去解释旧的 `ticks_`

这在理论上能带来“速度越大，步频越高”的效果，但在当前四足实机平台上，会破坏 gait phase continuity。

## 3.3 为什么“前几步正常，后面越来越乱”

这是该问题的典型特征。

原因是：

- 刚进入转向时，旧 phase 和新 phase 还没有完全错开
- 随着连续若干周期重算 gait 参数
- 相位误差不断积累
- 最终从“局部可接受”演变成“全局错乱”

所以现场上看起来像：

- 不是一下子崩
- 而是慢慢乱掉

---

## 4. 采用的解决方案

## 4.1 方案目标

当前阶段的目标不是追求“动态自适应步频”，而是优先保证：

1. 转向过程 gait 相位连续
2. 自动探索中的步态可重复
3. 方案可快速回滚

## 4.2 主方案：临时关闭 walk 状态中的自适应 cadence

采取的工程策略：

- 在 `SpotMicroWalkState::handleInputCommands(...)` 中
- 暂时不再使用 `buildAdaptiveGaitConfig(smnc_, cmd)`
- 改为直接使用固定的 `smnc_`

也就是：

```cpp
// SpotMicroNodeConfig gait_smnc = buildAdaptiveGaitConfig(smnc_, cmd);
SpotMicroNodeConfig gait_smnc = smnc_;
```

### 这样做的直接效果

- `phase_length` 固定
- `swing_ticks` 固定
- `ticks_ % phase_length` 保持一致解释方式
- 转向过程中不会因为 gait timing 被实时改写而导致腿序跳变

---

## 5. 为什么采用“注释保留原逻辑”的修改方式

本次修改明确要求：

- 优先使用注释方式修改
- 保留原核心程序
- 便于快速回滚

因此代码会采用下面这种形式：

```cpp
// NOTE(2026-04-15): Real robot tests show that changing cadence while ticks_
// keeps accumulating can break gait phase continuity during obstacle turns.
// Keep the original adaptive path commented here for easy rollback:
// SpotMicroNodeConfig gait_smnc = buildAdaptiveGaitConfig(smnc_, cmd);
SpotMicroNodeConfig gait_smnc = smnc_;
```

这样做的优点：

1. 原始设计思路仍在现场代码中保留
2. 后续如果要恢复 adaptive cadence，可直接取消注释
3. 当前版本更适合实机稳定性验证

---

## 6. 为什么暂时不优先采用其他方案

### 方案 A：继续调导航层参数

不作为主方案。

原因：

- 导航层只是在发 `cmd_vel`
- 真正导致“越转越乱”的是 gait timing 连续性问题
- 只调 `turn_speed` / `side_balance_gain` 不能根治

### 方案 B：保留 adaptive cadence，但对 cadence 做平滑

理论更优，但当前不优先。

原因：

- 需要重新设计 cadence 状态变量
- 需要保证 phase continuity
- 需要更多实机验证

### 方案 C：切换到专门的转向 gait

后续可考虑，但当前不优先。

原因：

- 实现量更大
- 需要重新定义转向时 body shift 和触地序列
- 当前最紧急的是先把自动探索转向稳定下来

## 7. 实施后的验证建议

## 7.1 上机前

- 重新编译 `spot_micro_motion_cmd`
- 确认自动探索速度保持低速
- 确认有人旁站
- 确认急停和 `safe_stop_robot.sh` 可用

## 7.2 实机验证顺序

1. 原地 / 半扶持验证 `stand -> walk`
2. 开阔区低速前进
3. 接近障碍后观察转向
4. 重复 3~5 次转向场景

## 7.3 重点观察指标

- 转向开始后，步态是否仍保持有序
- 是否还会出现“先正常，后混乱”
- 某一侧腿是否持续被压低
- 是否仍有侧翻倾向

如需回滚本次修复：

   - `software_orangepiaipro/spot_micro_motion_cmd/src/smfsm/spot_micro_walk.cpp`
2. 恢复原逻辑：

```cpp

```


4. 重新编译 `spot_micro_motion_cmd`

---

## 9. 当前结论

本问题的本质不是“前方障碍避障策略不够聪明”，而是：

**自动转向过程中，底层 gait cadence 动态变化破坏了相位连续性，导致四条腿的步态逐渐混乱。**

因此，当前最合理的工程路线是：

- 先关闭 walk 态中的 adaptive cadence
- 用固定节拍恢复实机步态稳定性
- 等自动探索转向闭环稳定后，再考虑恢复更高级的 cadence 设计

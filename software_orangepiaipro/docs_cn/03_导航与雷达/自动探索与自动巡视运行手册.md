# SpotMicro 自动探索与自动巡视运行手册

> 适用范围：`software_orangepiaipro` 当前新增的第一版自动探索建图 / 自动巡视实现。
>
> 目标：把“能启动、能停下、能接管、能看状态”的最小闭环讲清楚，避免上机时继续靠记忆拼命令。

---

## 1. 当前落地了什么

本轮代码已经落地以下组件：

1. `spot_micro_navigation/scripts/laser_wander_explorer.py`
   - 基于雷达的低速避障漫游
   - 状态机：`FORWARD / TURN_LEFT / TURN_RIGHT / ESCAPE / STOP`
2. `spot_micro_navigation/scripts/cmd_vel_arbiter.py`
   - 自动速度与手动速度仲裁
   - 规则：手动优先，手动超时后自动恢复
3. `spot_micro_navigation/scripts/cmd_vel_safety_gate.py`
   - 统一速度限幅
   - 雷达超时停车
   - 近距离硬停车
   - 巡视模式下可选 AMCL 协方差保护
4. `spot_micro_navigation/launch/auto_explore_mapping.launch`
   - 自动探索建图主 launch
5. `spot_micro_navigation/launch/auto_patrol.launch`
   - 自动巡视主 launch
6. `scripts/start_auto_explore_mapping.sh`
   - 自动探索建图一键启动脚本
7. `scripts/start_auto_patrol.sh`
   - 自动巡视一键启动脚本
8. `spot_micro_navigation/config/robot_mode_config.yaml`
   - 模式总配置入口
9. `scripts/common_run_on_all_cores.sh`
   - 统一把启动链路重进程到 `CPU0-3`
   - 避免自动建图 / 自动巡视继续继承错误的 `0-2` 绑核

## 3. 启动前必须确认的条件

## 3.1 低层运动链路已经单独验证通过

至少确认下面几项已经在手动模式下跑过：

1. `spot_micro_motion_cmd` 能正常启动
2. 键盘控制能让机器人 `stand`
3. 手动低速前进 / 转向不会立即失稳
4. RPLidar 能稳定发布 `/scan`

如果这些都没单独验证过，不建议直接上自动模式。

## 3.2 工作区环境

启动脚本默认会尝试寻找：

1. `/opt/ros/noetic/setup.bash`
2. `~/Desktop/SpotMicro/ros_noetic_ws/devel/setup.bash`
3. `~/catkin_ws/devel/setup.bash`
4. `~/Desktop/SpotMicro/spotmicro_ws/devel/setup.bash`

在 Orange Pi AI Pro 当前这套环境里，实际应以：

```bash
source ~/Desktop/SpotMicro/ros_noetic_ws/devel/setup.bash
```

作为基础 ROS 工作区；脚本现在已经兼容这个默认路径。

如果你的实际工作区不在这两个位置，请先设置：

```bash
export SPOTMICRO_SETUP_SCRIPT=/你的工作区/devel/setup.bash
```

如果基础工作区也不在默认位置，还可以单独设置：

```bash
export SPOTMICRO_BASE_SETUP_SCRIPT=/你的基础工作区/devel/setup.bash
```

如果系统没有 `/opt/ros/noetic/setup.bash`，但你本机 ROS 是自己编译在别的位置，还可以显式指定：

```bash
export SPOTMICRO_ROS_SETUP_SCRIPT=/你的ROS基础环境/setup.bash
```

## 3.3 CPU 绑核修复已内置到启动脚本

现在的两个启动脚本，不再只是简单 `roslaunch`。

它们会先通过：

```text
software_orangepiaipro/scripts/common_run_on_all_cores.sh
```

完成以下动作：

1. 检查当前 shell 的 CPU affinity
2. 若发现还是 `0-2`，自动重进程到 `0-3`
3. 清理 Conda / `PYTHONPATH` / `LD_LIBRARY_PATH` 污染
4. 重新 source ROS + SpotMicro 工作区
5. 再启动对应的 `tmux` pane 命令

注意：

1. `common_run_on_all_cores.sh` 已经负责准备运行环境
2. 后续 pane 命令不应再额外使用 `bash -lc` 重新进入登录 shell
3. 否则像 Conda 自动激活这类登录环境逻辑会再次执行，可能把刚清理好的 ROS Python 环境重新污染

这意味着：

1. 即使你当前登录 shell 还是 `0-2`
2. 只要通过这两个脚本启动
3. 自动建图 / 自动巡视链路也会尽量被拉回到 `0-3`

如果平台本身做了 cpuset 分组，例如：

- `CtrlCPU = 0-2`
- `AICPU = 3`

那么脚本现在会优先读取当前 cpuset 允许集合，自动把目标 CPU 集合改成该会话真正可用的范围，而不是一味强切 `0-3`。

也就是说，在 Orange Pi AI Pro 这类平台上，后续主 pane 看到：

```text
[INFO] target cpu set: 0-2
```

是正常现象。

如果你确实要手动覆盖目标 CPU 集合，仍然可以在启动前指定：

```bash
export SPOTMICRO_TARGET_CPU_SET=0-2
```

或在其他平台上指定：

```bash
export SPOTMICRO_TARGET_CPU_SET=0-3
```

只改这一项：

然后重新运行：


### 4.2 修改自动保存参数

改这里：

```yaml
auto_explore_mapping:
  autosave_enabled: true
  autosave_interval_sec: 120.0
  autosave_directory: ~/Desktop/SpotMicro/maps/autosave
  autosave_prefix: hector_autosave
  autosave_keep_last: 10
```

### 4.3 切换巡视是否启用 AMCL

改这里：

```yaml
auto_patrol:
  use_map_server: true
  use_amcl: false
```

建议：

1. 第一版巡视先 `use_amcl: false`
2. 只有当 `map -> odom` 已经稳定、`/amcl_pose` 协方差可控时，再切 `true`

## 6. 手动接管怎么用

当前设计目标不是“纯自动无人值守”，而是：

1. 默认自动运行
2. 人手一接管，马上听人
3. 松开后自动恢复

### 6.1 接管原理

`cmd_vel_arbiter.py` 当前规则：

1. 最近 `0.6 s` 内收到过 `/cmd_vel_manual`
   - 采用手动速度
2. 手动超时
   - 恢复采用 `/cmd_vel_auto`

### 6.2 怎么看当前是谁在控车

查看：

```bash
rostopic echo /spot_micro/cmd_vel_arbiter/source
```

你会看到类似：

```text
manual
auto
auto_disabled
idle
```

## 7. 自动模式怎么停

当前建议的停机优先级如下：

## 7.1 最直接：Ctrl+C 停整个 tmux pane

这是最可靠的人工停机方式。

## 7.2 关闭自动模式

```bash
rostopic pub -1 /spot_micro/auto_mode/enable std_msgs/Bool "data: false"
```

作用：

1. 自动探索节点停止输出有效自动速度
2. 仲裁器不再放行自动速度

## 7.3 发送自动探索 stop

```bash
rostopic pub -1 /spot_micro/auto_explore/stop std_msgs/Bool "data: true"
```

作用：

1. 自动探索节点切到 `STOP`
2. 自动模式保持关闭状态，直到你重新 enable

## 7.4 手动接管并发送零速度

适合短暂停车与姿态修正。

## 9. 当前默认参数的工程含义

## 9.1 自动探索节点

文件：

```text
spot_micro_navigation/config/wander_explorer.yaml
```

关键默认值：

1. `cruise_speed: 0.06`
2. `turn_speed: 0.22`
3. `forward_safe_dist: 0.60`
4. `turn_trigger_dist: 0.42`
5. `stuck_dist: 0.30`

当前实现里：

1. 前方足够空旷时按巡航速度前进
2. 接近障碍但还没到转向阈值时，会自动降速
3. 太近时直接进入转向或脱困

## 9.2 安全门

文件：

```text
spot_micro_navigation/config/safety_gate.yaml
```

关键默认值：

1. `front_stop_distance: 0.35`
2. `hard_stop_distance: 0.20`
3. `max_vx: 0.12`
4. `max_wz: 0.20`

含义：

1. 正前方普通避障距离在 `0.35 m`
2. 更近的 `0.20 m` 属于硬停车距离
3. 所有通过安全门的速度都会二次限幅

## 10. 推荐实机调试顺序

## 第 1 步：只看雷达

确认：

1. `/scan` 帧率稳定
2. 正前方扇区没有明显方向反转

## 第 2 步：只开安全门，不开自动探索

确认：

1. 手动速度能通过安全门
2. 近距离障碍会停下

## 第 3 步：开自动探索，但机器人离障碍远一点

确认：

1. 能前进
2. 看到障碍会停并转
3. 不会立刻高速旋转

## 第 4 步：短时建图 2~3 分钟

确认：

1. 地图持续更新
2. autosave 目录开始出现地图快照

## 第 5 步：再做 5~10 分钟连续运行

重点看：

1. 是否在角落抖动
2. 是否频繁重复绕圈
3. 是否出现长时间转向不恢复

## 12. 最重要的实机安全提醒

1. 第一次上自动模式时，机器人四周至少留出 `1.5 m x 1.5 m` 的空域
2. 必须有人在旁边，手能立刻接管或断电
3. 默认速度已经是低速，但仍不适合在人腿旁边直接试
4. 如果步态本身还不稳，不要指望自动避障层替你兜底
5. 一旦出现“连续快速原地转圈”或“靠近墙体仍反复前冲”，立即停机，先调参数再继续

# 解除 0-2 绑核并重新启动建图操作手册

> 适用对象：当前 SpotMicro + Orange Pi AIpro + ROS Noetic + Hector Mapping 场景。
>
> 目标：让原本只能运行在 `CPU0-2` 的关键建图进程，恢复为可使用 `CPU0-3`，从而验证第四核参与调度后，系统负载与建图稳定性是否改善。

---

## 1. 背景说明

前期排查已经明确发现：

1. 香橙派 4 个核都在线：

```bash
cat /sys/devices/system/cpu/online
```

输出：

```text
0-3
```

2. 但关键 ROS 进程都被限制到了 `0-2`：

```bash
taskset -pc <PID>
```

典型结果：

```text
pid 9102's current affinity list: 0-2
pid 8986's current affinity list: 0-2
pid 8985's current affinity list: 0-2
```

3. 当前用于启动 ROS 的 shell 本身也是 `0-2`：

```bash
echo $$
taskset -pc $$
```

输出类似：

```text
pid 10419's current affinity list: 0-2
```

因此，如果不先解除 shell 的 `0-2` 绑定，后面从这个终端启动的 ROS 进程仍然会继续继承 `0-2`。

---

## 2. 本手册要解决的问题

本手册解决的是：

1. 如何把当前 shell 从 `0-2` 改成 `0-3`
2. 如何在新的 4 核可用 shell 中重新启动建图链
3. 如何验证新的 ROS 关键进程已经不再只使用前三核
4. 如何对比放开前后的负载与稳定性

---

## 3. 推荐操作原则

建议遵循以下原则：

1. 不要在旧的 `0-2` shell 里继续启动新的建图链路。
2. 优先新开一个 `0-3` shell，再在里面启动 ROS。
3. 启动后一定要重新验证关键 PID 的 affinity。
4. 先做临时验证，不要先急着改系统配置。

---

## 4. 操作流程总览

推荐流程如下：

```text
确认当前 shell 是 0-2
  -> 新开一个 0-3 shell
  -> source ROS 环境
  -> 重新启动机器人底层与建图节点
  -> 检查关键 PID 是否已变为 0-3
  -> 在高负载时观察效果
```

---

## 5. 详细操作步骤

## 5.1 先确认当前 shell 确实是 `0-2`

在你当前终端执行：

```bash
echo $$
taskset -pc $$
```

如果输出类似：

```text
pid 10419's current affinity list: 0-2
```

说明：

1. 当前终端已经被绑到前三核
2. 不适合继续从这里直接启动新的建图链

---

## 5.2 推荐做法：新开一个四核可用 shell

执行：

```bash
taskset -c 0-3 bash
```

这样会进入一个新的 bash。

然后在这个新 bash 中执行：

```bash
echo $$
taskset -pc $$
```

理想输出应为：

```text
pid xxxx's current affinity list: 0-3
```

如果是这样，说明这个新 shell 已经允许使用四个核心。

---

## 5.3 在新 shell 中重新加载 ROS 环境

推荐先清理常见环境后再 source：

```bash
conda deactivate 2>/dev/null || true
unset PYTHONPATH
unset PYTHONHOME
unset LD_LIBRARY_PATH
source ~/Desktop/SpotMicro/ros_noetic_ws/devel/setup.bash
source ~/Desktop/SpotMicro/spotmicro_ws/devel/setup.bash
```

说明：

1. 避免旧环境污染；
2. 使用已经确认较干净的 ROS 启动方式；
3. 让本次验证更可控。

---

## 5.4 在新 shell 中重新启动建图链

建议至少重新启动以下两部分：

### 机器人底层运动控制

```bash
roslaunch spot_micro_motion_cmd motion_cmd.launch
```

### Hector 建图

另开一个同样已经是 `0-3` 的 shell，执行：

```bash
taskset -c 0-3 bash
conda deactivate 2>/dev/null || true
unset PYTHONPATH
unset PYTHONHOME
unset LD_LIBRARY_PATH
source ~/Desktop/SpotMicro/ros_noetic_ws/devel/setup.bash
source ~/Desktop/SpotMicro/spotmicro_ws/devel/setup.bash
roslaunch spot_micro_navigation slam_hector_mapping.launch
```

说明：

1. 关键是这两个 launch 都必须从新的 `0-3` shell 里启动；
2. 不能只把其中一个改成 `0-3`；
3. 否则链路仍然可能部分继承旧绑核限制。

---

## 5.5 找出新启动后的关键 PID

重新启动后，执行：

```bash
ps -ef | grep -E "hector_mapping|spot_micro_motion_cmd|i2cpwm_board" | grep -v grep
```

记下新的 PID，例如：

1. `hector_mapping`
2. `spot_micro_motion_cmd_node`
3. `i2cpwm_board_node`

---

## 5.6 验证新 PID 是否已经变成 `0-3`

执行：

```bash
taskset -pc <hector_mapping_pid>
taskset -pc <spot_micro_motion_cmd_pid>
taskset -pc <i2cpwm_board_pid>
```

### 期望结果

应该看到：

```text
pid xxxx's current affinity list: 0-3
```

如果还是：

```text
0-2
```

则说明：

1. 仍然有更上游的环境在传递绑核；
2. 需要继续检查上层父进程链。

---

## 6. 放开绑核后的负载验证

一旦确认关键进程已经是 `0-3`，就进入效果验证阶段。

## 6.1 查看整体 CPU 负载

在建图运行中执行：

```bash
top
```

然后按：

```text
1
```

观察：

1. `Cpu0`
2. `Cpu1`
3. `Cpu2`
4. `Cpu3`

### 目标

观察 CPU3 是否开始参与建图负载。

如果 CPU3 也开始被明显使用，说明：

1. 绑核解除已生效；
2. 第 4 核开始参与调度。

---

## 6.2 查看热点线程当前落在哪些核

执行：

```bash
ps -eLo pid,tid,psr,pcpu,stat,comm --sort=-pcpu | head -n 30
```

重点看：

1. `psr`
2. `pcpu`
3. `comm`

### 目标

确认热点线程不再只出现在 `0/1/2`，而是有可能调度到 `3`。

---

## 6.3 对比放开前后的负载现象

建议对比以下几项：

1. `hector_mapping` 的 CPU 占用
2. 系统整体 `load average`
3. CPU3 是否参与
4. SSH / 终端响应速度
5. 建图是否更流畅
6. 机器人动作是否更稳定

### 期望改善

理论上可能带来：

1. 前三核压力下降
2. 第四核开始承担部分调度
3. 系统整体更平衡
4. 卡顿概率降低

### 需要注意

解除绑核不是万能修复：

1. `hector_mapping` 本身还是高 CPU 节点
2. `i2cpwm_board` / I2C 控制链若存在阻塞，仍可能拖慢系统
3. XRDP / RViz 负担仍然可能很大

也就是说：

**解除绑核是必要优化，但不是唯一优化。**

---

## 7. 如果不想开新 shell，也可以直接修改当前 shell

如果你只是临时验证，也可以直接在当前 shell 上执行：

```bash
taskset -pc 0-3 $$
taskset -pc $$
```

确认已经变成：

```text
0-3
```

然后再在这个 shell 中启动 ROS。

### 但为什么更推荐“新开 shell”

因为新开 shell 更清楚：

1. 可以避免旧终端里已有残留进程干扰
2. 更容易区分“旧的 0-2 会话”和“新的 0-3 会话”
3. 更适合后续写成固定启动流程

---

## 8. 如果新启动后仍然是 `0-2` 怎么办

如果你已经从看似新的 shell 启动，但新 PID 仍然是 `0-2`，说明：

1. 更上层终端链路仍在传递 `0-2`
2. 当前 shell 虽然表面变了，但父链依旧有限制

这时建议继续查：

```bash
echo $$
taskset -pc $$
pstree -sap $$
```

并检查上游进程：

```bash
taskset -pc <上游bash_pid>
taskset -pc <ptyHost_pid>
taskset -pc <cursor_server_pid>
```

目标是找到最早一层开始变成 `0-2` 的进程。

---

## 9. 推荐的长期做法

在后续正式一键启动方案中，建议直接把“放开四核”纳入启动流程。

当前仓库里已经开始按这个思路落地，核心脚本是：

```text
software_orangepiaipro/scripts/common_run_on_all_cores.sh
software_orangepiaipro/scripts/start_auto_explore_mapping.sh
software_orangepiaipro/scripts/start_auto_patrol.sh
```

推荐思路：

1. 统一启动脚本最开始先进入：

```bash
taskset -c 0-3 bash
```

或者在脚本自身内部确保：

```bash
taskset -pc 0-3 $$
```

2. 然后再 source ROS 环境
3. 再启动建图、自动探索、自动巡视等节点

当前实现又向前多做了一步：

1. 不只修复最外层 shell
2. 还会把 `tmux` pane 中的 `roslaunch / rosrun / rostopic` 命令逐个包到四核启动包装层里

因此后续在 Orange Pi 上，更推荐直接使用：

```bash
bash ~/Desktop/SpotMicro/spotMicro-Chinese/software_orangepiaipro/scripts/start_auto_explore_mapping.sh
```

或：

```bash
bash ~/Desktop/SpotMicro/spotMicro-Chinese/software_orangepiaipro/scripts/start_auto_patrol.sh
```

这样可以避免以后每次手工忘记解除 `0-2` 绑核，也更不容易被旧 `tmux` / 终端父链再次拖回 `0-2`。

---

## 10. 本轮操作的推荐验收标准

本手册执行完成后，建议以下指标作为“绑核解除成功”的标准：

### 验收标准 1：当前启动 shell 为 `0-3`

```bash
taskset -pc $$
```

输出应包含：

```text
0-3
```

### 验收标准 2：新启动关键 PID 也为 `0-3`

```bash
taskset -pc <PID>
```

输出应包含：

```text
0-3
```

### 验收标准 3：建图时 CPU3 开始有实际参与

```bash
top
```

按 `1` 后，观察 `Cpu3` 不再长期近乎闲置。

### 验收标准 4：系统卡顿较之前减轻

至少应观察：

1. 终端更流畅
2. SSH 更稳定
3. 建图过程更平稳

**当前建图卡顿问题中，解除 `0-2` 绑核是一个已经确认、且非常值得优先验证的优化动作。建议先在 `0-3` shell 中重新启动机器人底层和 Hector 建图链，再对比 CPU3 是否开始参与，以及系统整体稳定性是否改善。**

# 方案 A：脚本级永久解除 0-2 绑核优化设计

> 适用对象：SpotMicro + Orange Pi AIpro + ROS Noetic + 机器人一键启动场景。
>
> 目标：把“解除 `0-2` 绑核”的操作，从一次性临时手工命令，升级为长期可复用、对用户透明、可验证、可回退的脚本级永久方案。

---

## 1. 方案背景

## 1.1 当前仓库中的实际落地点

本方案已经开始在仓库中落地，当前对应文件为：

```text
software_orangepiaipro/scripts/common_run_on_all_cores.sh
software_orangepiaipro/scripts/start_auto_explore_mapping.sh
software_orangepiaipro/scripts/start_auto_patrol.sh
software_orangepiaipro/scripts/read_robot_mode_config.py
software_orangepiaipro/spot_micro_navigation/config/robot_mode_config.yaml
```

其中实际实现与最初设计相比，有两个关键细化：

1. 不只修正“外层启动 shell”
2. 还会把 `tmux` 中每个关键 pane 的命令都单独包一层四核启动包装脚本

这样做的原因是：

1. 你之前的 `roslaunch` 是从 Cursor / ptyHost 派生链启动的
2. 这条链本身就可能已经继承 `0-2`
3. 如果只修外层脚本，而 `tmux` pane 继续继承旧 affinity，问题仍可能复发

所以当前实现采用的是：

```text
tmux pane command
  -> bash software_orangepiaipro/scripts/common_run_on_all_cores.sh
      -> bash -lc "roslaunch / rosrun / rostopic ..."
```

这比“只在最外层执行一次 `taskset`”更稳。

## 1.2 当前实现已经兼容 cpuset 平台限制

后续排查确认，Orange Pi AI Pro 当前系统不是简单的“终端误绑到 `0-2`”，而是平台主动做了 cpuset 分组：

```text
/sys/fs/cgroup/cpuset/CtrlCPU/cpuset.cpus = 0-2
/sys/fs/cgroup/cpuset/AICPU/cpuset.cpus  = 3
```

而交互 shell 默认处于：

```text
cpuset:/CtrlCPU
```

这意味着：

1. 当前会话真实可用 CPU 集合就是 `0-2`
2. 直接强切 `0-3` 会失败
3. 失败原因不是脚本错误，而是 cpuset 交集之后仍然只有 `0-2`

因此当前 `common_run_on_all_cores.sh` 已进一步优化为：

1. 若用户显式设置 `SPOTMICRO_TARGET_CPU_SET`，优先使用该值
2. 若未显式设置，则先读取当前进程所在 cpuset cgroup 的 `cpuset.cpus`
3. 只有在读取不到 cpuset 限制时，才退回默认值 `0-3`

这样在普通 Linux 机器上，脚本仍会尽量放开到 `0-3`；
而在 Orange Pi AI Pro 这种 `CtrlCPU/AICPU` 分组平台上，脚本会自动适配到当前会话真正可用的 CPU 集合，避免一启动就报错退出。

前期排查已经确认以下事实：

1. 香橙派 `cpu0-3` 全部在线；
2. 但当前 ROS 关键进程被限制在 `0-2`；
3. 当前 shell 本身也是 `0-2`；
4. 从该 shell 启动的：
   - `hector_mapping`
   - `spot_micro_motion_cmd`
   - `i2cpwm_board`
   等关键进程，都会继承 `0-2`；
5. 因此第四核虽然在线，但无法参与关键机器人进程调度。

此前已经验证：

```bash
taskset -pc $$
```

输出类似：

```text
pid xxxx's current affinity list: 0-2
```

同时：

```bash
taskset -pc <hector_mapping_pid>
taskset -pc <spot_micro_motion_cmd_pid>
taskset -pc <i2cpwm_board_pid>
```

也都显示：

```text
0-2
```

因此必须有一个长期有效的工程方案，确保以后每次启动机器狗 ROS 系统时，都能自动放开到：

```text
0-3
```

---

## 2. 为什么采用“脚本级永久方案”

这里的“永久”并不是指：

1. 修改 Linux 内核
2. 修改 systemd 全局 CPUAffinity
3. 修改系统所有用户会话

而是指：

**以后凡是通过机器人专用启动脚本启动 ROS，就自动保证使用 `0-3`，用户不再需要手动执行解绑核命令。**

这比系统级永久修改更适合当前阶段，原因如下：

### 2.1 风险更低

1. 不改系统全局配置
2. 不影响与机器人无关的其他进程
3. 不会把排障复杂度扩散到整个系统

### 2.2 更贴合实际使用方式

你后续本来就希望使用统一的一键启动脚本，例如：

1. 自动探索建图
2. 自动巡视

那么最自然的做法，就是把“解绑核”固定写进这些脚本链路中。

### 2.3 容易验证与回退

如果脚本级方案不理想：

1. 改脚本即可
2. 回滚脚本即可
3. 不会污染系统层

---

## 3. 方案目标

脚本级永久方案需要满足以下目标：

1. 用户以后不再手动输入：
   ```bash
   taskset -c 0-3 bash
   ```
2. 统一启动脚本自动进入 `0-3`
3. 新启动的关键 ROS 进程自动继承 `0-3`
4. 启动脚本能自动验证 affinity 是否正确
5. 启动脚本能在异常时明确报错
6. 后续自动探索建图与自动巡视都共用这套机制

---

## 5. 为什么不能只在脚本里简单写一条 `taskset`

很多时候会想直接在脚本最前面写：

```bash
taskset -pc 0-3 $$
```

虽然这条命令有作用，但它并不是最优工程方案。

### 5.1 存在的问题

1. 没有检查当前 affinity 是否已经正确
2. 没有失败提示
3. 没有统一封装
4. 不利于多个启动脚本复用
5. 不方便后续扩展验证逻辑

### 5.2 更好的方式

推荐使用：

**自检 + 自重启到 `0-3` + 防递归 + 启动后验证**

这比单条 `taskset` 更稳，也更适合长期维护。

---

## 6. 通用包装脚本的设计要求

`common_run_on_all_cores.sh` 建议具备以下能力。

## 6.1 功能 1：读取当前 shell affinity

例如通过：

```bash
taskset -pc $$
```

得到当前 affinity。

## 6.2 功能 2：若不是 `0-3`，自动重进程化

推荐逻辑：

1. 如果当前已经是 `0-3`
   - 继续执行
2. 如果当前不是 `0-3`
   - 自动用：
     ```bash
     taskset -c 0-3 bash "$0" ...
     ```
     重新执行自己

## 6.3 功能 3：防止无限递归

建议用环境变量，例如：

```bash
SPOTMICRO_AFFINITY_REEXEC=1
```

如果已经重进程过一次，但 affinity 仍不对：

1. 直接报错
2. 停止继续执行

## 6.4 功能 4：清理环境污染

建议脚本中统一做：

```bash
conda deactivate 2>/dev/null || true
unset PYTHONPATH
unset PYTHONHOME
unset LD_LIBRARY_PATH
```

这样能减少环境问题对验证结果的影响。

## 6.5 功能 5：统一 source ROS 工作空间

建议统一加载：

```bash
source ~/Desktop/SpotMicro/ros_noetic_ws/devel/setup.bash
source ~/Desktop/SpotMicro/spotmicro_ws/devel/setup.bash
```

## 6.6 功能 6：执行下游命令

包装脚本不直接绑定某一种模式，而是执行调用方传入的启动命令。

例如：

```bash
common_run_on_all_cores.sh bash -lc "roslaunch ..."
```

## 6.7 功能 7：打印关键状态

建议至少打印：

1. 当前 shell PID
2. 当前 shell affinity
3. 当前执行模式

方便调试。

## 7. 推荐的通用包装脚本逻辑

以下是推荐逻辑结构（示意，不是最终实现文件）：

```bash
#!/usr/bin/env bash
set -euo pipefail

TARGET_CPU_SET="0-3"

current_affinity() {
  taskset -pc $$ 2>/dev/null | awk -F': ' '{print $2}'
}

ensure_all_cores() {
  local current
  current="$(current_affinity || true)"

  if [[ "${current}" == "${TARGET_CPU_SET}" ]]; then
    return 0
  fi

  if [[ "${SPOTMICRO_AFFINITY_REEXEC:-0}" == "1" ]]; then
    echo "[ERROR] 已尝试切换到 ${TARGET_CPU_SET}，但当前 affinity 仍为: ${current}"
    exit 1
  fi

  echo "[INFO] 当前 shell affinity 为: ${current:-unknown}，重新进入 ${TARGET_CPU_SET}"
  export SPOTMICRO_AFFINITY_REEXEC=1
  exec taskset -c "${TARGET_CPU_SET}" bash "$0" "$@"
}

prepare_ros_env() {
  conda deactivate 2>/dev/null || true
  unset PYTHONPATH
  unset PYTHONHOME
  unset LD_LIBRARY_PATH

  source ~/Desktop/SpotMicro/ros_noetic_ws/devel/setup.bash
  source ~/Desktop/SpotMicro/spotmicro_ws/devel/setup.bash
}

main() {
  ensure_all_cores "$@"
  prepare_ros_env

  echo "[INFO] shell pid: $$"
  echo "[INFO] shell affinity: $(current_affinity)"

  "$@"
}

main "$@"
```

### 该逻辑的优点

1. 自动自检
2. 自动切换到 `0-3`
3. 防止无限重启
4. 可被多个模式脚本复用

## 8. 模式脚本如何复用该包装层

后续所有机器人主入口脚本，建议统一通过 `common_run_on_all_cores.sh` 启动。

## 8.1 自动探索建图脚本

当前实现不再直接依赖脚本可执行位，而是显式通过 `bash` 调用包装脚本。示意：

```bash
#!/usr/bin/env bash
set -euo pipefail

BASE_DIR=~/Desktop/SpotMicro/spotMicro-Chinese/software_orangepiaipro/scripts

bash "${BASE_DIR}/common_run_on_all_cores.sh" bash -lc "
  echo '[INFO] 启动自动探索建图模式'
  roslaunch spot_micro_navigation auto_explore_mapping.launch
"
```

## 8.2 自动巡视脚本

当前实现同样通过 `bash` 显式调用公共包装脚本。示意：

```bash
#!/usr/bin/env bash
set -euo pipefail

BASE_DIR=~/Desktop/SpotMicro/spotMicro-Chinese/software_orangepiaipro/scripts

bash "${BASE_DIR}/common_run_on_all_cores.sh" bash -lc "
  echo '[INFO] 启动自动巡视模式'
  roslaunch spot_micro_navigation auto_patrol.launch
"
```

这样以后：

1. 用户不需要关心 `taskset`
2. 也不需要知道 shell 当前是不是 `0-2`
3. 只执行统一启动命令即可

## 9. 为什么还要结合配置文件

你已经确认后续采用：

**配置文件 + 启动脚本读取配置文件**

所以脚本级解绑核方案，不应孤立设计，而应与配置文件机制统一。

推荐配置文件：

```text
spot_micro_navigation/config/robot_mode_config.yaml
```

建议管理内容：

1. `common.map_yaml`
2. `scan_topic`
3. `cmd_vel` 相关话题
4. 自动探索建图参数
5. 自动巡视参数

### 为什么这很重要

因为后续脚本的职责就变成了：

1. 确保当前运行在 `0-3`
2. 读取配置文件
3. 启动对应 ROS 模式

这样“解绑核”和“模式启动”就是一体化设计，不会分裂成多套入口。

## 10. 建议加入启动后自验证

这是优化后的方案 A 里非常重要的一层。

仅仅让 shell 变成 `0-3` 还不够，最好在脚本中增加自动验证：

### 10.1 验证当前 shell

启动后打印：

```bash
taskset -pc $$
```

期望看到：

```text
0-3
```

### 10.2 验证关键进程

脚本启动 ROS 后，自动检查：

1. `hector_mapping`
2. `spot_micro_motion_cmd_node`
3. `i2cpwm_board_node`

例如：

```bash
sleep 3
for name in hector_mapping spot_micro_motion_cmd_node i2cpwm_board_node; do
  pid=$(pgrep -f "$name" | head -n 1 || true)
  if [[ -n "$pid" ]]; then
    echo "[INFO] $name pid=$pid"
    taskset -pc "$pid"
  fi
done
```

### 10.3 为什么这一步重要

因为它能立即回答：

1. 解绑核是否真的成功传递给子进程
2. 哪个关键节点仍然没放开
3. 问题是在 shell 层还是 launch 层

## 11. 建议加入失败保护

为了让方案长期可用，建议加上以下保护机制。

## 11.1 `taskset` 命令存在性检查

```bash
command -v taskset >/dev/null 2>&1 || {
  echo "[ERROR] 未找到 taskset，请先安装 util-linux"
  exit 1
}
```

## 11.2 重进程失败即退出

若已经尝试重进程化，但 affinity 仍不是 `0-3`：

1. 不要继续启动 ROS
2. 直接报错退出

## 11.3 配置文件检查

启动模式脚本建议检查：

1. 配置文件是否存在
2. 地图文件是否存在
3. 配置项是否完整

## 11.4 启动脚本不要静默失败

发生以下情况时都应明确提示：

1. affinity 切换失败
2. source 工作空间失败
3. 地图路径不存在
4. 关键 launch 命令不存在

## 12. 推荐与 tmux 结合

如果后续要做“一条命令启动全部功能”，建议把该方案与 `tmux` 结合。

例如：

### 自动探索建图模式

```text
pane 1: motion_cmd.launch
pane 2: slam_hector_mapping.launch
pane 3: 自动探索节点 / 日志
pane 4: 键盘接管节点
```

### 自动巡视模式

```text
pane 1: 底层控制
pane 2: 自动巡视节点
pane 3: 日志观察
pane 4: 键盘控制
```

这样一来：

1. 脚本先统一确保 `0-3`
2. 再在 `tmux` 中启动所有子流程
3. 所有 pane 启动的 ROS 进程都会继承正确 affinity

这会比手工多终端启动更稳定、更可重复。

## 13. 推荐实施顺序

建议按以下顺序逐步落地，而不是一次性把所有功能都塞进脚本。

### 第 1 步：实现 `common_run_on_all_cores.sh`

目标：

1. 只验证 shell 能正确切到 `0-3`
2. 不急着启动所有 ROS 功能

### 第 2 步：先包装一个简单启动脚本

例如只包装：

1. `motion_cmd.launch`
2. `slam_hector_mapping.launch`

验证：

1. 新进程是否变成 `0-3`
2. 建图是否更稳定

### 第 3 步：加入配置文件读取

目标：

1. 切换地图不靠命令行
2. 启动脚本统一读取 `robot_mode_config.yaml`

### 第 4 步：加入启动后自检

目标：

1. 自动打印关键 PID affinity
2. 一眼就能看出是否成功

### 第 5 步：再纳入 tmux / 一键启动

目标：

1. 自动探索建图一键启动
2. 自动巡视一键启动

## 14. 这个方案为什么适合当前阶段

这是当前最合适的“长期方案”，因为它：

1. 不动系统全局
2. 不需要先查清 Cursor 为什么默认是 `0-2`
3. 只对机器人启动流程生效
4. 用户体验最接近“永久解决”
5. 后续容易直接并入一键启动体系

换句话说：

它不是去“修所有终端”，而是去“保证机器人启动流程始终正确”。

对当前工程目标来说，这已经足够。

### 验收标准 1：用户不再手动敲 `taskset`

以后只需要执行：

```bash
bash ~/Desktop/SpotMicro/spotMicro-Chinese/software_orangepiaipro/scripts/start_auto_explore_mapping.sh
```


### 验收标准 2：脚本自动把 shell 放到 `0-3`

脚本输出中能看到：

```text
[INFO] shell affinity: 0-3
```

### 验收标准 3：新启动关键 PID 也是 `0-3`

```text
hector_mapping -> 0-3
```

至少应观察到：

1. 前三核不再独占全部关键负载
2. 建图链更稳定
3. 终端响应更流畅

## 16. 最终建议

对当前 SpotMicro 项目而言，推荐把“解除 `0-2` 绑核”做成：

**脚本级永久方案，而不是系统级永久方案。**

具体实施方式是：

1. 增加一个通用四核启动包装脚本
2. 所有机器人启动脚本统一走这个包装层
3. 包装层优先重进程化到“当前 cpuset 会话允许的最大 CPU 集合”
4. 包装层统一 source ROS 环境
5. 模式脚本继续负责具体业务启动
6. 启动后自动验证关键 PID 的 affinity

这样以后你在日常使用中就不需要再手动解绑核了。

---

## 17. 一句话总结

**方案 A 的最佳优化方式，不是把一条 `taskset` 临时命令记住，而是建立一个统一的“4 核启动包装层”，让以后所有机器人专用启动脚本都自动运行在 `0-3`，并且在启动后自动验证关键 ROS 进程是否真正继承了四核 affinity。**

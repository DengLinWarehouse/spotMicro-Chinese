# SpotMicro 运动调试复盘

## 1. 调试目标

- 让 SpotMicro 在 Orange Pi + ROS Noetic overlay 环境下稳定运行运动主链路。
- 解决“只能原地踏步、不能明显向前行走”的问题。
- 建立可重复的工作区维护方式，避免源码、编译结果、运行结果不一致。
- 在不大规模重做 12 路舵机标定的前提下，恢复可用步态。
- 找出当前剩余的运动问题，为后续导航和雷达接入打基础。

## 2. 最终结论

- 机器狗已经从“原地踏步”恢复到“可以向前行走”。
- `8 相 gait` 与 `4 相 gait` 目前都可以向前运动。
- 当前实机表现上，`8 相 gait 优于 4 相 gait`。
- 原先怀疑的“仍在按 180° 舵机参数工作”并不是这次“原地踏步”的当前主因。
- 本次最关键的软件修复，是“不同 gait 下理论步幅归一化逻辑”修正。
- 当前剩余最重要的运动问题是：`左前腿偏弱`。

## 3. 问题分类总结

### 3.1 工作区源码与实际运行源码不一致

#### 问题描述

Orange Pi 上的 `spotmicro_ws/src` 与仓库中的 `software_orangepiaipro` 一度不是严格统一的一份源码。这会导致“改了代码但运行结果没变化”或“运行生效了但仓库没同步”的问题。

#### 典型现象

- 改动 YAML、脚本、C++ 文件后，运行行为与预期不一致。
- 后续还发生过 `spotmicro_ws/src` 内容误删，需要恢复。

#### 根因分析

- 工作区没有统一采用“源代码仓库为唯一事实来源”的模式。
- overlay 中既有源码副本，又有外部 clone 包，容易漂移。

#### 解决办法

- 将 Orange Pi 工作区改为：核心包由 `spotmicro_ws/src` 软链接到 `software_orangepiaipro`。
- 新增链接脚本：`software_orangepiaipro/scripts/link_spotmicro_workspace.sh`。
- 同步更新部署文档和操作文档，明确后续只维护 `software_orangepiaipro`。

#### 结果

- 现在源码维护路径更加清晰。
- 后续修改只需改一处，避免“运行代码”和“仓库代码”分裂。

### 3.2 扩展包依赖过多，导致主链路编译失败

#### 问题描述

一开始编译时把 SLAM、gmapping、joy 等扩展包和运动主链路一起编译，这些扩展包依赖较多，在当前 Orange Pi 环境下导致频繁失败。

#### 典型现象

`catkin_make` 过程中依次出现多种依赖错误：

- 缺 `joy`
- GMock/GTest 相关错误
- `boost_signals` 缺失
- `hector_geotiff` 缺失

#### 根因分析

- 当前目标是先跑通机器狗运动主链路，而不是一次性把导航/SLAM 全部打通。
- 将所有包混编，导致问题耦合、排障效率低。

#### 解决办法

使用 `CATKIN_IGNORE` 暂时屏蔽非核心包，先保证运动主链路可编译、可运行。临时忽略的包包括：

- `spot_micro_joy`
- `gmapping_ROS`
- `openslam_gmapping`
- `spot_micro_launch`

#### 结果

- 成功把调试范围收缩到运动主链路。
- 编译复杂度显著下降，为后续定位运动问题创造条件。

### 3.3 Conda 环境污染 ROS Python 运行与编译环境

#### 问题描述

编译过程中一度使用了 Conda Python，而不是系统 Python，导致 ROS Noetic 的 Python 依赖无法正常解析。

#### 典型现象

- CMake 日志中出现：`Using PYTHON_EXECUTABLE: /usr/local/miniconda3/bin/python3`
- 后续出现：`No module named 'catkin_pkg'`

#### 根因分析

- Conda 修改了 `PATH`、`PYTHONPATH`、`PYTHONHOME`。
- ROS 的 Python 工具链是按系统 Python 组织的，Conda 环境不兼容当前 overlay 配置。

#### 解决办法

每次构建或运行 ROS 主链路前先执行：

- `conda deactivate`
- `unset PYTHONPATH`
- `unset PYTHONHOME`

编译时显式指定：

- `-DPYTHON_EXECUTABLE=/usr/bin/python3`

#### 结果

- Python 环境恢复稳定。
- ROS 编译流程不再被 Conda 干扰。

### 3.4 测试依赖链路不完整，引发 GMock/GTest 错误

#### 问题描述

默认编译测试时，系统会尝试构建 GMock/GTest，当前环境缺失相关资源，导致编译中断。

#### 典型现象

出现 `/usr/src/gmock` 不存在或 ExternalProject 无法处理等错误。

#### 根因分析

- 当前目标是跑机器人，不是跑包测试。
- 测试依赖不完整并不影响运动主链路。

#### 解决办法

编译时关闭测试：

- `catkin_make -DCATKIN_ENABLE_TESTING=OFF -DPYTHON_EXECUTABLE=/usr/bin/python3`

#### 结果

- 成功绕过无关测试依赖。
- 主链路编译得以继续推进。

### 3.5 ROS Python 脚本缺少执行权限

#### 问题描述

`spot_micro_keyboard_command` 和 `servo_move_keyboard` 的 Python 脚本最初不可执行。

#### 典型现象

`rosrun` 提示已经找到脚本，但脚本不是可执行文件。

#### 根因分析

仓库中脚本文件未设置执行权限。

#### 解决办法

为脚本添加可执行权限：

- `spot_micro_keyboard_command/scripts/*.py`
- `servo_move_keyboard/scripts/*.py`

#### 结果

`rosrun` 可以继续尝试执行对应节点。

### 3.6 Python 脚本 shebang 指向无效解释器

#### 问题描述

即使脚本具备执行权限，仍无法启动。

#### 典型现象

报错：`/usr/bin/python: bad interpreter: No such file or directory`

#### 根因分析

- 脚本首行使用了旧式 shebang：`#!/usr/bin/python`
- Ubuntu 20.04/22.04 默认没有该解释器路径

#### 解决办法

将脚本首行统一改为：

- `#!/usr/bin/env python3`

并再次确认执行权限。

#### 结果

- `spotMicroKeyboardMove.py` 成功运行。
- 手动键盘控制链路恢复正常。

### 3.7 机器狗最初只能原地踏步，不能明显向前行走

#### 问题描述

机器狗最初虽然有步态动作，但水平推进不足，整体接近原地踏步。

#### 典型现象

- 键盘控制下，步态循环正常，但前进趋势弱。
- 用户观察上表现为“能迈腿但不怎么往前走”。

#### 最初怀疑

一度怀疑是否仍在沿用 180° 舵机逻辑，导致运动幅度计算不匹配。

#### 进一步分析

当前运行配置里，系统并不只是简单停留在旧的 180° 默认值状态，因此，这个猜测不是此次现象的主要根因。

#### 真正根因

- 不同步态，尤其 8 相与 4 相切换时，理论步幅没有统一归一化。
- 相位数、摆动时间、支撑时间变化后，实际前进距离没有保持一致。
- 导致某些 gait 下虽然动作存在，但推进量不足。

#### 解决办法

在 `spot_micro_motion_cmd` 中引入 stride reference 机制。新增配置参数：

- `stride_reference_num_phases: 8`
- `stride_reference_overlap_time: 0.0`
- `stride_reference_swing_time: 0.36`

同时在控制代码中增加 gait 无关的步幅规划换算逻辑。

#### 修改文件

- `software_orangepiaipro/spot_micro_motion_cmd/include/spot_micro_motion_cmd/spot_micro_motion_cmd.h`
- `software_orangepiaipro/spot_micro_motion_cmd/src/spot_micro_motion_cmd.cpp`
- `software_orangepiaipro/spot_micro_motion_cmd/src/smfsm/spot_micro_walk.cpp`
- 以及 `software/` 对应同名文件

#### 结果

- 这是本次运动恢复的核心软件修复。
- `4 相 gait` 不再退化为原地踏步。

### 3.8 8 相 gait 和 4 相 gait 验证结果不一致，但均可前进

#### 问题描述

需要确认修复后不仅默认 gait 可用，而且切换 gait 后仍能维持前进能力。

#### 验证方法

- 先验证 8 相成功前进。
- 再仅修改 gait 相关参数切换至 4 相，不改其他站姿/步幅参数。

#### 4 相测试中仅修改的内容

- `num_phases`
- 四条腿的 `*_contact_phases`
- `overlap_time`
- `swing_time`
- `body_shift_phases`
- 三个 body balance shift

#### 结果

- 8 相可以前进。
- 4 相也可以前进。
- 但用户实测观察为：8 相效果更好。

#### 结论

- stride normalization 修复已经生效。
- 4 相不再是“只能原地踏步”。
- 但当前这台机器的推荐默认 gait 仍应保留为 8 相。

### 3.9 4 相 gait 不适合直接作为当前默认 gait

#### 问题描述

虽然 4 相修好后可以走，但并不代表它就是当前最优解。

#### 现象

用户明确反馈：4 相也能前进，但 8 相运动效果更好。

#### 根因分析

4 相步态对当前机械结构、舵机响应、机体抖动更敏感，在没有继续做更深层局部优化前，贸然切成 4 相会降低整体表现。

#### 解决办法

- 将 8 相作为当前默认成功基线。
- 4 相作为实验 gait 保留，后续如需研究再单独优化。

#### 结果

当前基线配置应以“8 相前进成功版本”为准。

### 3.10 左前腿偏弱仍是遗留问题

#### 问题描述

在整体已经能前进后，仍观察到左前腿相对偏弱。

#### 现象

左前腿相较右前腿或其他腿，动作幅度、响应、负载表现存在差异。

#### 当前判断

这更像是单腿/单关节问题，而不是全局 gait 规划问题。

#### 优先怀疑项

- `LF_2` / `LF_3` 与 `RF_2` / `RF_3` 的差异
- 舵机安装角度误差
- 连杆阻力或摩擦
- `center` / `range` 配置不理想
- 个别舵机本体性能偏弱

#### 已制定的解决思路

保持 8 相 gait 不动，避免再扩大变量范围；使用 `servo_move_keyboard` 做以下单舵机对比：

- `LF_1` vs `RF_1`
- `LF_2` vs `RF_2`
- `LF_3` vs `RF_3`

#### 当前状态

这是运动调试阶段当前最重要的未完成问题，还需要继续做局部对比验证。

## 4. 本次调试中最有效的方法论

### 4.1 先跑通主链路，再处理扩展链路

先把运动主链路打通，再回头处理 joy、gmapping、SLAM、导航等扩展问题。这样可以避免多类问题叠加，降低排障难度。

### 4.2 先修环境，再修运动

Conda、Python、权限、脚本解释器这些基础问题不解决，后面的运动测试都会失真。环境问题先清干净，是提高调试效率的关键。

### 4.3 每次只改一类变量

例如测试 4 相 gait 时，只改 gait 相关项，不同时改站姿、步幅、舵机参数。这样才能判断变化真正来自哪里。

### 4.4 及时保留成功基线

8 相成功配置必须保留，一旦实验失败，可以快速回退，不至于把“已知可行状态”再次弄丢。

### 4.5 系统级问题收敛后，转向单腿级问题

先解决“能否稳定前进”，再细化到“哪条腿表现不好”，这是更高效的调试路径。

## 5. 当前阶段性成果

- 运动主链路可编译、可运行。
- 键盘控制节点可正常启动。
- 机器狗可以 `stand / idle / walk`。
- 8 相 gait 能稳定向前。
- 4 相 gait 也能向前。
- 已形成较清晰的 Orange Pi 工作区维护模型。
- 已将“原地踏步”问题从系统层面基本解决。

## 6. 当前遗留问题

- 左前腿偏弱，尚未完成根因定位。
- 导航、雷达、SLAM 依赖链仍需独立处理，不应与运动主链路混调。
- 未来若要提升地图质量和导航效果，还需继续优化：
  - 雷达外参
  - odom 策略
  - 低速控制一致性

## 7. 后续建议

### 7.1 短期建议

- 保持 `8 相 gait` 为默认运动基线。
- 暂停大范围 gait 改动。
- 集中做左前腿与右前腿的单舵机对比测试。
- 先判断是：
  - 幅度偏小
  - 响应迟缓
  - 方向不对
  - 机械阻力过大

### 7.2 中期建议

在左前腿问题定位完成后，再决定是否微调：

- `LF_2`
- `LF_3`

对应的 `center / range / center_angle_deg`

### 7.3 长期建议

运动主链路稳定后，再单独恢复并处理：

- `spot_micro_launch`
- `rplidar_ros`
- `hector_slam`

不建议把运动调试和导航调试重新混在一起进行。

## 8. 最终总结

本次运动调试的真正突破，不在于“某一次临时参数改动”，而在于：

- 清理了工程环境问题
- 建立了正确的 Orange Pi 工作区维护方式
- 修复了 gait 切换下的步幅归一化逻辑
- 确认了当前实机最优基线为 8 相 gait

当前系统已经从“原地踏步”进入“可实际前进”的阶段。后续运动方向应从“全局 gait 能否走”转向“局部腿部执行质量优化”，首要目标是解决左前腿偏弱。

# 机器狗网页应用层功能实现逻辑详细分析

> 文档定位：对 `机器狗网页应用层三层架构方案框架.md` 的功能实现逻辑进行逐项展开。  
> 文档目的：供方案评审使用，只讨论 **功能实现逻辑、状态关系、边界条件、风险与审核要点**，**不涉及具体代码编写**。  
> 当前约束：局域网访问、无视频流、无物理急停断电按钮、自动巡航按随机漫游定义。

---

## 1. 结论先行

这套网页应用层若要真正“好用”，核心不在页面按钮本身，而在于以下 4 件事是否设计正确：

1. **所有页面功能都要落到统一应用状态机上**
2. **所有运动相关动作都要经过后端任务管理器，而不是页面直通 ROS**
3. **开始、急停、安全收车三者的逻辑边界必须严格分离**
4. **地图、模式、运行状态必须始终有唯一真值，不能靠页面当前显示猜测**

因此，本详细分析文档采用“逐功能审查”的方式，每项功能统一说明：

1. 功能目标
2. 输入与输出
3. 前置条件
4. 主流程
5. 异常与边界
6. 审核重点

---

## 2. 功能逻辑设计总原则

在详细展开单个功能前，先固定全局原则。

### 2.1 单一真值原则

所有关键状态都必须在后端任务管理器中有唯一真值，前端只负责显示和触发操作。

推荐由后端统一维护的关键状态包括：

- `system_online`
- `selected_mode`
- `runtime_state`
- `armed`
- `estop_latched`
- `fault_active`
- `selected_map_id`
- `selected_map_name`
- `speed_level`
- `latest_manual_cmd_valid`
- `map_preview_status`
- `ros_runtime_health`

如果这些状态由页面自行缓存并推断，将导致以下问题：

1. 页面显示与 ROS 实际状态不一致
2. 刷新页面后状态丢失
3. 多次点击后进入不可预测状态

### 2.2 先状态、后按钮原则

页面按钮不是直接功能入口，而是“请求状态转换”的入口。

例如：

- 点击“手动建图”并不等于立即切换完成
- 点击“开始”并不等于机器人马上动起来
- 点击“停止”也不能笼统等于急停

每个按钮都应先被翻译为：

```text
用户意图 -> 后端判定前置条件 -> 状态迁移 -> ROS 层执行 -> 状态回写页面
```

### 2.3 先归零、再切换原则

凡是会影响运动行为的功能切换，都必须经过统一的“归零与脱离旧模式”流程。

典型动作包括：

- 模式切换
- 从自动转手动
- 开始前复位
- 安全收车
- 故障恢复

### 2.4 风险优先级原则

在所有功能中，优先级从高到低应为：

```text
急停 > 故障停车 > 安全收车 > 模式切换 > 开始/使能 > 普通参数设置 > 页面显示
```

## 3. 全局对象与状态定义

为方便逐项审核，先定义网页应用层应存在的几个核心对象。

### 3.1 模式对象

模式对象表示“用户希望系统运行在哪种功能模式下”。

建议字段：

- 模式编号
- 模式名称
- 是否需要地图
- 是否需要 SLAM
- 是否允许手动输入
- 是否允许自动输入
- 是否显示地图预览
- 启动前检查项列表

建议模式集合：

1. 手动控制
2. 手动建图
3. 自动探索建图
4. 自动漫游巡航

### 3.2 运行状态对象

运行状态表示“系统当前处于哪一阶段”。

建议状态集合：

- `DISCONNECTED`
- `READY_DISARMED`
- `STARTING`
- `ARMED_READY`
- `RUNNING_MANUAL`
- `RUNNING_MANUAL_MAPPING`
- `RUNNING_AUTO_EXPLORE`
- `RUNNING_AUTO_PATROL`
- `SAFE_STOPPING`
- `ESTOP_LATCHED`
- `FAULT`

### 3.3 地图对象

地图对象表示当前已保存地图的元信息，而不是页面上的一张图片。

建议最少包含：

- 地图唯一 ID
- 显示名称
- 底层地图 YAML 路径
- 预览图路径
- 创建时间
- 最近更新时间
- 来源模式
- 是否可用于巡航

### 3.4 会话对象

会话对象表示“本次网页控制连接与本次运行任务”的生命周期。

建议至少区分：

- 页面连接会话
- 当前模式运行会话
- 当前任务启动会话

这样做的原因是：

1. 页面刷新不应该直接把 ROS 任务全停掉
2. 但页面断连超时后，又必须能触发手动控制归零

## 4. Layer 1：网页前端功能实现逻辑

本层只负责交互，不负责决策。  
前端的所有动作都只能视为“请求”，不能视为“执行成功”。

## 5. 功能 F01：连接与系统状态区

### 5.1 功能目标

让用户始终知道：

1. 页面是否连上后端
2. 后端是否连上 ROS 运行时
3. 当前模式是什么
4. 是否已使能
5. 是否处于急停锁定
6. 当前地图名是什么
7. 当前是否存在故障

### 5.2 输入与输出

输入：

- 后端定时推送的状态快照

输出：

- 前端状态文字
- 图标/颜色提示
- 按钮可用/禁用状态

### 5.3 前置条件

- 前端与后端 WebSocket 或轮询链路已建立
- 后端能够汇总应用状态

### 5.4 主流程

1. 页面加载后，立即请求当前系统状态
2. 若后端可达，则显示“应用层在线”
3. 若后端同时确认 ROS 运行时可达，则显示“机器人在线”
4. 根据后端返回的 `runtime_state` 渲染页面主状态
5. 根据 `armed / estop_latched / fault_active` 决定其他控件可用性

### 5.5 异常与边界

1. 前端在线但 ROS 离线
   - 页面可显示，但开始按钮必须禁用
2. 状态刷新超时
   - 页面进入“状态未知”，禁止继续发新控制
3. 页面状态与后端状态短时抖动
   - 前端不能自行平滑推断，只显示后端最新真值

### 5.6 审核重点

1. 页面是否把“前端在线”误认为“机器人在线”
2. 当前地图名是否始终来自后端真值
3. 急停锁定和普通未使能是否有清晰区分

## 6. 功能 F02：虚拟摇杆输入

### 6.1 功能目标

让用户通过网页进行低速、方向明确、可预测的手动运动控制。

这里必须特别强调：

1. 本摇杆 **不是全向平面速度摇杆**
2. 本摇杆 **不表达横移**
3. 本摇杆 **不直接表达速度大小**
4. 本摇杆表达的是 **运动方向意图 + 转向意图**

也就是说：

- 摇杆向上：表示向前运动
- 摇杆向下：表示向后运动
- 摇杆向右：表示向右转向，不表示向右横移
- 摇杆向左：表示向左转向，不表示向左横移
- 摇杆处于右上：表示“向前运动的同时逐渐向右转弯”
- 摇杆处于左上：表示“向前运动的同时逐渐向左转弯”
- 摇杆处于右下：表示“向后运动的同时逐渐向右转弯”
- 摇杆处于左下：表示“向后运动的同时逐渐向左转弯”

因此，该摇杆更接近“车式/差速式方向控制语义”，而不是“全向底盘二维平移语义”。

### 6.2 输入与输出

输入：

- 触摸或鼠标拖动产生的二维摇杆位置

输出：

- 一个“手动控制意图对象”，而不是直接底层速度

该对象至少包含：

- 前后运动方向量
- 左右转向量
- 转向变化程度
- 采样时间戳
- 来源会话 ID

这里建议明确：

1. `前后运动方向量`
   - 表示前进、后退或中立
   - 不直接等价于最终线速度数值
2. `左右转向量`
   - 表示左转、右转趋势
   - 不等价于横向平移
3. `转向变化程度`
   - 表示需要转得多急、多缓
   - 该量可以影响角速度或转弯曲率
   - 但不应直接抬高线速度

### 6.3 前置条件

- 当前模式允许手动输入
- 系统未处于急停锁定
- 当前已使能
- 后端判定机器人可接受手动控制

### 6.4 主流程

1. 页面持续采集摇杆位置
2. 对摇杆位置做归一化处理
3. 页面只输出“方向与转向意图”，不输出最终运动速度
4. 页面将归一化后的控制意图发送给后端
5. 后端判断当前模式和状态是否允许接收手动输入
6. 后端结合当前速度档位，将摇杆意图映射为“固定速度上限下的线速度 + 角速度”组合
7. 后端以固定频率输出，不依赖页面每次事件都能及时到达

建议在后端采用如下解释逻辑：

1. 纵向分量决定：
   - 前进
   - 后退
   - 停止
2. 横向分量决定：
   - 左转趋势
   - 右转趋势
   - 转向强弱
3. 最终线速度大小主要由：
   - 当前速度档位
   - 当前模式上限
   - 安全门限
   共同决定
4. 摇杆位移不直接决定“机器人跑得更快”，而是决定“机器人朝什么方向走、转多少”

### 6.5 推荐控制语义

为避免后续实现时被误解为“二维速度分解”，建议明确采用以下控制语义：

| 摇杆区域 | 语义解释 | 不应解释为 |
|----------|----------|------------|
| 上 | 直行前进 | 增大速度 |
| 下 | 直行后退 | 增大倒车速度 |
| 左 | 原地或近原地左转 | 向左横移 |
| 右 | 原地或近原地右转 | 向右横移 |
| 左上 | 前进并逐渐左转 | 左前方斜移 |
| 右上 | 前进并逐渐右转 | 右前方斜移 |
| 左下 | 后退并逐渐左转 | 左后方斜移 |
| 右下 | 后退并逐渐右转 | 右后方斜移 |

其中：

1. “上/下”决定前后运动主方向
2. “左/右”决定转向偏置
3. “斜向”表示前后运动与转向的组合

### 6.6 与速度设置的关系

虚拟摇杆与速度设置必须彻底解耦：

1. 摇杆负责“往哪走、怎么转”
2. 速度档位负责“最多走多快”

因此同一个速度档下：

- 摇杆在上方：直行
- 摇杆在右上：以相同速度档前进，但同时加右转趋势
- 摇杆在右侧：原地或低速右转

而不是：

- 摇杆推得更远就跑得更快

若后期确实想加入“摇杆位移同时调节转向激烈程度”，也只能影响角速度或转弯曲率，不建议影响线速度档位。

### 6.7 为什么不能让前端直接发速度

原因：

1. 浏览器事件频率不稳定
2. 页面切后台时事件可能暂停
3. 网络短抖动会导致速度更新不连续
4. 前端不适合承担安全节拍控制
5. 当前摇杆语义本身就是“方向意图”，并不是“最终速度向量”

### 6.8 异常与边界

1. 页面断连
   - 后端应在超时后自动归零手动输入
2. 摇杆回中失败
   - 后端看门狗必须能在超时后归零
3. 用户在自动模式下拖动摇杆
   - 若模式允许手动接管，则切到手动优先
   - 若模式不允许，则应忽略并明确反馈
4. 用户误以为左右方向可以横移
   - 页面文案与图示必须提前说明“左右=转向，不是横移”
5. 用户误以为摇杆推得越远就越快
   - 页面文案必须明确“速度由速度档单独控制”

### 6.9 审核重点

1. 摇杆输入是不是“方向/转向意图”而不是“二维平移速度”
2. 后端是否固定频率输出而非事件驱动输出
3. 页面断连时是否一定归零
4. 左右方向是否被明确解释为“转向”而非“横移”
5. 速度是否始终由独立速度档控制

## 7. 功能 F03：速度设置

### 7.1 功能目标

让用户以较低认知负担调整运动快慢，同时避免连续滑条带来的误触和过冲。

### 7.2 输入与输出

输入：

- 速度加号
- 速度减号

输出：

- 当前速度档位
- 对摇杆映射或自动模式上限的约束参数

### 7.3 建议逻辑

速度设置应采用 **离散档位** 而不是连续滑条。

推荐原因：

1. 更容易记忆
2. 更容易限制
3. 更适合低速机器人
4. 更容易和模式联锁

### 7.4 主流程

1. 页面展示当前档位
2. 用户点击加减后，向后端发送“调整档位”请求
3. 后端检查当前是否允许调整
4. 后端更新 `speed_level`
5. 所有后续运动控制都以新的 `speed_level` 为上限

### 7.5 与模式的关系

不同模式下速度档位不应完全等价。

例如：

- 手动控制：速度档位决定手动控制上限
- 手动建图：速度档位应比手动控制更保守
- 自动探索建图：速度档位应限制自动漫游上限
- 自动漫游巡航：速度档位应作为漫游速度上限，而非页面实时控制速度

### 7.6 异常与边界

1. 用户切换模式后保留高档位
   - 建议模式切换后自动回落到安全默认档
2. 用户未开始前将速度调高
   - 允许设置，但不得输出运动
3. 急停锁定时改速度
   - 允许改配置，但不得解除锁定

### 7.7 审核重点

1. 速度档位是否为离散档
2. 模式切换后是否回落安全档
3. 速度设置是否仅是“上限约束”，而不是直接驱动运动

## 8. 功能 F04：模式切换

### 8.1 功能目标

让用户通过有限模式入口完成主要任务，同时确保切换过程安全、可追踪、不会残留旧任务。

### 8.2 输入与输出

输入：

- 用户点击某个模式按钮

输出：

- 后端进入对应 `selected_mode`
- 若切换成功，状态区显示新模式
- 若切换失败，停留原模式并给出失败原因

### 8.3 核心原则

模式切换不等于“启动运行”，只等于“选择当前要运行的系统配置”。

这点非常关键：

1. 用户可以先选模式
2. 再做地图选择、速度设置等准备
3. 最后再按开始/使能

### 8.4 主流程

建议统一流程如下：

```text
接收模式切换请求
-> 判断是否处于急停锁定/故障态
-> 若允许切换，则先执行脱离旧模式
-> 清零速度
-> 停止旧模式专属任务
-> 检查新模式依赖
-> 载入新模式配置
-> 更新 selected_mode
-> 进入 READY_DISARMED
```

### 8.5 为什么不能热切

热切会带来以下问题：

1. 自动探索还在发自动速度
2. 旧地图任务还在占用资源
3. 安全门参数可能仍是旧模式配置
4. 页面显示已切换，但 ROS 还没切过去

### 8.6 异常与边界

1. 运行中切模式
   - 必须先停运动，再切换
2. 依赖未满足
   - 例如巡航模式未选地图，则拒绝切换或切换后保持未就绪
3. 模式切换中再次点击其他模式
   - 后端只保留最后一次有效请求，或直接拒绝重复点击

### 8.7 审核重点

1. 模式切换是否与“开始”解耦
2. 模式切换是否总是先归零
3. 失败时是否保持旧状态不被破坏

## 9. 功能 F05：地图显示区

### 9.1 功能目标

在无视频流条件下，为用户提供最核心的环境反馈与任务确认能力。

### 9.2 输入与输出

输入：

- 后端提供的地图预览图片或快照
- 预览更新时间
- 当前模式

输出：

- 页面地图图像
- 地图更新时间标识
- 地图状态提示

### 9.3 逻辑定位

地图显示区不是地图编辑器，也不是 RViz 替代品。  
它在第一阶段只承担三个作用：

1. 证明建图正在进行
2. 证明当前选中的地图是什么
3. 在无视频流条件下提供最低限度环境反馈

### 9.4 显示逻辑

建议逻辑如下：

- 手动控制：默认不显示
- 手动建图：显示实时或准实时快照
- 自动探索建图：显示实时或准实时快照
- 自动漫游巡航：显示当前加载地图快照

这里需要进一步明确一个关键设计点：

**“网页预览快照”与“地图存档文件”不能视为同一类数据。**

建议逻辑上拆成两条链：

1. **运行态预览链**
   - 目的：给网页地图显示区看“当前地图大概长什么样”
   - 特点：覆盖写、低频刷新、轻量、允许降采样
   - 不承担正式归档责任
2. **地图存档链**
   - 目的：给后续地图选择、命名、巡航加载使用
   - 特点：按保存策略生成正式地图文件
   - 数量受控，必须有保留策略

如果把网页每次显示的预览都当成正式地图文件保存，会产生大量无意义快照，迅速占满存储空间。

### 9.5 预览快照的生命周期设计

针对你提出的“会产生大量地图快照、占用存储空间”的问题，建议采用以下设计：

#### 方案原则

1. **网页地图显示使用“最新预览图”机制，不为每次刷新都生成独立存档**
2. **正式地图存档只在明确事件下生成**
3. **预览图与正式地图文件分开管理**

#### 建议的保存策略

运行态预览：

- 只保留：
  - `latest_preview.png`，或
  - 最近 `3~5` 张循环覆盖预览图
- 刷新频率建议：
  - `1~2 s/次` 即可
- 目的：
  - 给用户看当前建图趋势
  - 不追求逐帧保真

正式地图存档：

- 仅在以下场景保存：
  1. 用户主动点击“保存地图”
  2. 自动探索建图触发定时自动保存
  3. 模式退出前执行一次最终保存
- 保留策略建议：
  - 自动保存仅保留最近 `5~10` 份
  - 手动命名确认后的正式地图永久保留，直到用户删除

#### 不推荐的做法

不推荐：

1. 地图显示区每刷新一次就保存一份新的 `.pgm/.yaml`
2. 把网页预览图和地图正式文件混在同一目录、同一命名策略下
3. 依赖用户手工定期清理临时预览图

#### 推荐的目录语义

逻辑上建议分开：

- `previews/`
  - 只放网页显示用预览图
  - 允许覆盖写
- `maps/`
  - 只放正式地图存档
  - 走命名、索引、保留策略

这样用户审核时也更容易区分：

1. 页面上看到的是“预览”
2. 真正能选中加载的是“正式地图”

### 9.6 地图图片格式转换逻辑

针对你提到的第二个问题：

**`.pgm` 是否需要转换格式后再在网页中显示？**

我的明确建议是：

1. **正式地图底层文件继续保留 `.pgm + .yaml` 作为真值**
2. **网页显示时应由后端转换为更适合浏览器显示的格式**
3. **第一阶段优先推荐转换为 `.png`**

#### 为什么不建议网页直接显示 `.pgm`

原因如下：

1. 浏览器对 `.pgm` 的原生支持并不稳定，也不适合作为标准网页资源格式
2. `.pgm` 更适合作为 ROS 地图存档与算法输入，不适合作为用户界面图片格式
3. 网页显示往往还需要：
   - 缩放
   - 叠加时间戳
   - 叠加模式信息
   - 标记“预览过期”
4. 这些需求都更适合在后端先转换成标准网页图片格式后再显示

#### 为什么优先推荐 PNG

推荐 `.png` 的原因：

1. 无损，适合地图这种黑白/灰度结构图
2. 浏览器兼容性好
3. 便于叠加文本和标识
4. 后续若要压缩显示尺寸也容易处理

#### 建议的格式策略

建议分层如下：

| 用途 | 推荐格式 | 说明 |
|------|----------|------|
| 正式地图存档 | `.pgm + .yaml` | 作为 ROS 地图真值 |
| 网页运行预览 | `.png` | 供前端显示 |
| 缩略图列表预览 | `.png` 或 `.webp` | 供地图选择列表显示 |

其中第一阶段最稳妥的做法是：

- 存档仍是 `.pgm + .yaml`
- 预览统一转成 `.png`

这样逻辑最清晰，也最不容易把“运行预览”与“正式地图”混淆。

#### 转换时机建议

建议在后端完成以下逻辑：

1. 收到地图更新或检测到地图文件变化
2. 判断是否需要刷新网页预览图
3. 若需要，则把当前地图转换或导出为 `preview.png`
4. 前端只拿 `preview.png` 显示

也就是说：

- 前端不碰 `.pgm`
- 前端只看标准图片资源

### 9.7 异常与边界

1. 地图长时间未更新
   - 页面应标为“预览过期”
2. 当前模式不需要地图
   - 地图区隐藏或折叠
3. 地图预览生成失败
   - 不得影响运动主链，仅作为状态异常提示
4. 预览图目录膨胀
   - 必须通过覆盖写或循环保留策略限制数量
5. 正式地图过多
   - 自动保存地图必须具备保留上限和清理策略
6. `.pgm` 转换失败
   - 不影响正式地图真值，但页面应显示“预览不可用”

### 9.8 审核重点

1. 地图区是否只承担“反馈”而非“控制”
2. 是否显示更新时间
3. 预览失败时是否不会影响主功能
4. 是否区分“网页预览快照”和“正式地图存档”
5. 预览图是否采用覆盖写或有限循环保留，而非无限累积
6. 网页显示是否采用 `.png` 等标准图片格式，而不是前端直接处理 `.pgm`

### 9.9 地图保存按钮与保存时命名的补充设计

你指出的这个问题是成立的：

既然系统已经有：

1. 手动建图
2. 自动探索建图
3. 地图显示区

那么应用层里必须存在一个明确的 **地图保存入口**，否则用户只能看到地图在生成，却无法以“面向使用”的方式把当前地图固化下来。

因此，建议在第一阶段补充一个明确功能：

- **地图保存按钮**

并将其逻辑定义为：

- 用户点击保存
- 系统弹出命名输入框
- 用户输入地图名称并确认
- 后端将“当前地图真值”保存为一份正式地图存档
- 保存成功后，把该地图加入可选地图列表

#### 地图保存按钮的定位

地图保存按钮不是地图显示区的一部分“装饰”，而是建图闭环中必不可少的出口。

其作用是：

1. 结束一次阶段性建图成果
2. 把当前运行态地图转为可复用地图资产
3. 进入后续地图选择、巡航加载链路

如果没有这个按钮，用户会遇到以下问题：

1. 只能看到地图在长，但不知道怎样把它正式保存
2. 自动保存出来的文件命名混乱，不适合作为用户使用入口
3. 当前地图和正式可选地图之间缺少明确转换动作

#### 地图保存按钮的显示逻辑

建议显示规则如下：

| 模式 | 是否显示“保存地图”按钮 | 说明 |
|------|------------------------|------|
| 手动控制 | 否 | 当前不建图 |
| 手动建图 | 是 | 用户可随时保存当前建图结果 |
| 自动探索建图 | 是 | 用户可在探索过程中保存阶段性地图 |
| 自动漫游巡航 | 否 | 当前以使用地图为主，不是产出地图 |

进一步建议：

1. 只有在后端确认“当前已有有效地图数据”时，保存按钮才可点击
2. 若地图尚未形成有效内容，按钮可显示但应禁用，并提示“暂无可保存地图”

#### 点击保存后的主流程

建议流程如下：

```text
用户点击“保存地图”
-> 后端检查当前模式是否允许保存
-> 后端检查当前是否有有效地图数据
-> 页面弹出命名输入框
-> 用户输入地图名称并确认
-> 后端校验名称合法性
-> 后端执行正式地图保存
-> 保存成功后更新地图索引
-> 主状态区更新“当前地图名称”
-> 地图选择列表新增该地图
```

#### 为什么“点击保存时命名”是合理设计

这是第一阶段最符合用户直觉的方式，原因如下：

1. 用户刚完成一段建图，最清楚这张地图应叫什么
2. 命名发生在保存动作当下，用户不容易忘记
3. 这样可以避免后期一堆默认文件名再回头批量整理

#### 命名与正式保存的关系

这里要明确两点：

1. 用户输入的是“地图显示名”
2. 后端保存的是“正式地图记录”

也就是说，保存动作完成后，系统至少应同时得到：

1. 一份正式地图文件
2. 一条地图元数据记录
3. 一个用户可理解的显示名称

#### 是否应在保存前暂停建图

第一阶段建议不要把“保存地图”设计成必须停下机器人才能保存。

更合理的逻辑是：

1. 点击保存时，对当前地图状态做一次正式快照保存
2. 建图主链可以继续运行
3. 保存成功后，当前预览继续更新

这样用户可以：

1. 在探索中途做阶段性保存
2. 最后再保存最终版本

但需要明确：

- “保存当前地图”不等于“结束建图模式”

#### 自动保存与手动保存的关系

建议明确区分：

1. **自动保存**
   - 用于容灾和阶段性保底
   - 名称可以带时间戳
   - 数量受保留策略控制
2. **手动保存**
   - 用于用户确认的正式版本
   - 允许用户输入地图名称
   - 应进入正式地图列表

因此，“保存地图按钮”对应的应是：

- **手动正式保存**

而不是简单触发一次自动保存。

#### 保存成功后的系统表现

保存成功后建议有以下反馈：

1. 页面提示“地图保存成功”
2. 当前地图名称显示为新命名结果
3. 地图选择列表中出现该地图
4. 若后续进入自动漫游巡航，可直接选择该地图

#### 异常与边界

1. 用户在非建图模式点击保存
   - 应禁止或不显示该按钮
2. 当前地图数据无效
   - 拒绝保存，并提示原因
3. 名称为空
   - 不允许提交
4. 名称重复
   - 可允许，但建议提示可能混淆
5. 保存失败
   - 不得影响当前建图运行，但必须提示失败

#### 审核重点

1. 是否明确补充了“保存地图按钮”
2. 保存动作是否只在建图模式开放
3. 点击保存时是否允许直接输入地图名称
4. 保存后是否进入正式地图列表，而不是只生成临时快照
5. 保存失败时是否不会中断运动主链

## 10. 功能 F06：地图选择

### 10.1 功能目标

让用户在巡航相关模式前明确选择将要使用的地图，并使该选择成为系统真值的一部分。

### 10.2 输入与输出

输入：

- 用户打开地图选择框
- 用户选择某张地图
- 用户点击确认

输出：

- `selected_map_id`
- `selected_map_name`
- 地图预览图切换

### 10.3 主流程

1. 页面请求后端获取地图列表
2. 后端返回地图元数据集合
3. 页面展示列表与预览
4. 用户选择后，页面先显示待确认状态
5. 用户点击确认
6. 后端更新系统当前选图真值
7. 主状态区同步显示地图名称

### 10.4 为什么要有“确认”步骤

因为地图切换会影响后续运行模式，属于关键配置动作。  
如果用户在列表中随手点了一下就立刻切换，容易造成误选。

### 10.5 异常与边界

1. 当前处于运行中
   - 建议运行中禁止切换地图
2. 地图文件损坏
   - 应在后端过滤为“不可用地图”
3. 无地图可选
   - 巡航模式不得进入可开始状态

### 10.6 审核重点

1. 地图选择是否以“确认后生效”为原则
2. 运行中是否禁止切换地图
3. 地图列表是否由后端而不是前端扫描文件夹

## 11. 功能 F07：地图命名

### 11.1 功能目标

让用户用可理解的名称标识地图，而不影响底层地图文件可用性。

### 11.2 核心逻辑

地图命名应只改“显示名”，不直接修改运行时文件真值。

推荐分离关系：

- 底层文件名：稳定、不随意改
- 显示名称：可编辑、给用户看

### 11.3 主流程

1. 用户选择某张地图
2. 用户输入显示名称
3. 后端校验名称合法性
4. 后端只修改地图元数据中的 `display_name`
5. 页面刷新列表和主显示区

### 11.4 为什么不能优先靠重命名文件

原因：

1. 可能破坏 `yaml` 与 `pgm` 的引用关系
2. 可能破坏已有脚本中的默认路径
3. 可能导致页面名称和实际加载路径混乱

### 11.5 异常与边界

1. 名称为空
   - 拒绝提交或回退默认名
2. 名称重复
   - 可允许，但应提示可能混淆
3. 当前地图正在使用
   - 可允许改显示名，但不得改底层路径

### 11.6 审核重点

1. 是否明确区分显示名与文件路径
2. 是否禁止前端绕过后端直接改文件
3. 改名后主状态区是否同步更新

## 12. 功能 F08：地图名称显示框

### 12.1 功能目标

### 12.2 实现逻辑

### 12.3 原因

1. 用户点了列表但没确认
2. 页面刷新后丢失
3. 地图已被后端切换但页面没同步

### 12.4 审核重点

1. 显示框是否绑定后端真值
2. 未选图时是否明确显示“未选择地图”
3. 地图切换失败时是否保持旧值不变

## 13. 功能 F09：开始/使能

### 13.1 功能目标

将系统从“已选择模式、已完成配置但不允许运动”切换到“允许运行当前模式”。

### 13.2 关键定义

开始/使能不是“强制开跑”，而是“允许本模式进入执行态”。

### 13.3 前置条件

按模式不同，至少应检查：

- 系统在线
- 不处于急停锁定
- 不处于故障态
- 已选择模式
- 模式依赖满足
- 如需要地图，则地图已选且可读
- 如需要 SLAM 或自动链路，则对应运行时就绪

### 13.4 主流程

建议流程：

```text
接收开始请求
-> 检查当前状态是否允许
-> 检查模式依赖
-> 若依赖满足，进入 STARTING
-> 执行模式对应启动流程
-> 若成功，armed = true
-> 进入 ARMED_READY 或 RUNNING_*
```

### 13.5 为什么不应直接“一按就走”

因为在以下情况中，开始应只表示“进入准备就绪”：

1. 手动控制但摇杆仍在中位
2. 自动模式需要先启动依赖组件
3. 页面速度档虽非 0，但仍需先过安全检查

### 13.6 异常与边界

1. 用户连续点开始
   - 后端应防抖，只保留首个有效请求
2. 启动中失败
   - 回退到 `READY_DISARMED`
3. 已处于运行态再次点开始
   - 应无效或仅提示“已开始”

### 13.7 审核重点

1. 开始是否总是先做依赖检查
2. 开始是否不等于绕过安全门
3. 启动失败时是否安全回退

## 14. 功能 F10：急停

### 14.1 功能目标

在检测到危险或用户紧急干预时，以最高优先级尽快切断运动指令链。

### 14.2 关键定义

急停不是“优雅停机”，而是“最快进入停止链路”。

### 14.3 主流程

建议逻辑：

```text
急停触发
-> 立刻锁存 estop_latched = true
-> 禁止新模式切换和新开始请求
-> 立即停自动输出
-> 持续发送零速度
-> 记录急停来源与时间
-> 状态进入 ESTOP_LATCHED
```

### 14.4 为什么要锁存

不锁存会出现以下危险：

1. 网络抖动后误恢复
2. 用户误触后又再次开始
3. 页面刷新后状态丢失

### 14.5 与安全收车的边界

急停只负责：

1. 停止运动
2. 阻止继续运行
3. 进入人工干预状态

急停不应默认执行：

1. 站稳动作
2. 趴下动作
3. 新模式切换

因为这些动作会消耗时间，且在急停语义下不应优先。

### 14.6 当前系统的现实限制

由于没有物理急停断电按钮：

1. 软件急停只能切断控制指令链
2. 无法在控制器或供电层实现真正的硬切
3. 因此速度上限和测试环境必须保守

### 14.7 异常与边界

1. 急停时页面断连
   - 急停一旦被后端接收，就必须独立完成
2. 已在安全收车流程中急停
   - 急停应立即抢占
3. 急停后如何恢复
   - 必须人工复位，不能自动恢复

### 14.8 审核重点

1. 急停是否最高优先级
2. 是否有锁存机制
3. 是否明确不等于“站立后趴下”

---

## 15. 功能 F11：安全收车

### 15.1 功能目标

在任务正常结束时，让机器人按尽量平稳的顺序停止并回到安全姿态。

### 15.2 关键定义

安全收车是“正常结束流程”，不是“最高优先级紧急动作”。

### 15.3 主流程

建议逻辑：

```text
接收安全收车请求
-> 禁止自动输出
-> 清零手动/自动速度
-> 等待运动归零
-> 请求站立稳定
-> 请求 idle / lie down
-> 进入 READY_DISARMED
```

### 15.4 与现有基础能力的关系

该流程可以复用现有 `safe_stop_robot.py` 逻辑，但应用层需明确：

1. 何时允许执行
2. 执行中页面如何显示
3. 失败后如何回退

### 15.5 异常与边界

1. 机器人当前姿态异常
   - 收车可能失败，应进入 `FAULT`
2. 收车过程中用户再次点开始
   - 应拒绝，直到收车结束
3. 收车过程中触发急停
   - 急停必须抢占

### 15.6 审核重点

1. 收车流程是否单独于急停
2. 收车是否有执行中状态显示
3. 收车失败时是否进入故障态而不是假装成功

---

## 16. 功能 F12：手动控制模式

### 16.1 功能目标

提供最低复杂度、最快可理解的人工操控模式。

### 16.2 主逻辑

手动控制模式只做两件事：

1. 接受网页摇杆和速度档位
2. 通过后端映射为手动速度输入

不承担：

1. 地图显示
2. 自动探索
3. 自动任务调度

### 16.3 开始后的运行逻辑

1. 用户先选“手动控制”
2. 后端进入 `READY_DISARMED`
3. 用户按开始
4. 后端进入 `ARMED_READY`
5. 摇杆有有效输入时，进入 `RUNNING_MANUAL`
6. 摇杆回中后，保持 `ARMED_READY` 或低速空闲态

### 16.4 审核重点

1. 手动模式是否尽可能简单
2. 是否不依赖地图
3. 未开始前是否绝不运动

---

## 17. 功能 F13：手动建图模式

### 17.1 功能目标

让用户边手动操控边建图，并通过地图快照观察建图进展。

### 17.2 主逻辑

在手动控制基础上增加：

1. SLAM 运行链
2. 地图预览
3. 地图保存来源标记

### 17.3 建议流程

```text
选择手动建图
-> 切换到建图依赖配置
-> 启动或确认 SLAM 运行
-> 进入 READY_DISARMED
-> 用户开始/使能
-> 手动速度控制生效
-> 地图预览同步刷新
```

### 17.4 地图保存逻辑边界

本模式主要是“产生地图”，不是“选择地图”。  
因此：

1. 地图生成与预览可以实时存在
2. 地图最终命名与确认保存可以作为单独动作处理

### 17.5 审核重点

1. 建图模式是否明确依赖雷达/SLAM 就绪
2. 地图预览是否不影响运动主链
3. 建图产物是否与地图管理逻辑分离

---

## 18. 功能 F14：自动探索建图模式

### 18.1 功能目标

让机器人在低速受限条件下自动随机探索环境，同时持续生成地图。

### 18.2 主逻辑

本模式的关键不是“让机器人自己动”，而是：

1. 自动速度从自动模块发出
2. 手动输入可作为接管通道
3. 安全门统一裁决

### 18.3 典型运行链

```text
Explorer -> /cmd_vel_auto
手动输入 -> /cmd_vel_manual
Arbiter -> 统一仲裁
SafetyGate -> 安全限幅
MotionCmd -> 执行运动
SLAM -> 地图更新
```

### 18.4 页面实现逻辑

页面在该模式下主要承担：

1. 启动/停止探索任务
2. 显示当前运行状态
3. 显示地图快照
4. 允许必要时手动接管

### 18.5 手动接管逻辑

需要明确约束：

1. 手动接管只应暂时抢占自动速度
2. 手动输入超时后，是否恢复自动，由后端策略决定
3. 页面必须明确告知“当前由谁控制”

### 18.6 审核重点

1. 自动探索是否仍受统一安全门约束
2. 手动接管逻辑是否清晰
3. 页面是否能看出当前控制源

---

## 19. 功能 F15：自动巡航模式（随机漫游）

### 19.1 功能目标

在已选择地图的前提下，让机器人进入基于当前环境约束的随机漫游运行。

### 19.2 特别说明

当前“自动巡航”并非固定路线、固定航点任务。  
因此其功能逻辑必须避免误导用户。

建议在设计文案上使用：

- `自动漫游巡航`

而不是容易让人误解为固定路径巡逻的“自动巡航”。

### 19.3 本模式的逻辑定位

本模式本质上是：

1. 加载已选地图
2. 以当前地图为运行参考
3. 进行随机漫游式自动运行

它不是：

1. 精准路线跟随
2. 固定航点巡逻
3. 闭环任务编排

### 19.4 为什么仍然需要选地图

虽然是随机漫游，但选地图仍有价值：

1. 确认当前运行环境属于哪张已知地图
2. 页面能显示当前地图名称和预览
3. 后续升级到更严格定位/规划模式时可以复用接口

### 19.5 异常与边界

1. 用户以为会沿固定路线走
   - 必须通过界面文案避免误解
2. 地图与当前环境不匹配
   - 应允许开始前检查并提醒风险
3. 用户未选地图
   - 应禁止进入可开始状态

### 19.6 审核重点

1. 模式文案是否避免误导
2. 选地图是否是硬前置条件
3. 是否明确不是固定航点模式

---

## 20. 功能 F16：后端任务管理器

### 20.1 功能目标

作为三层架构中的唯一中枢，负责把页面操作翻译为受控的 ROS 行为。

### 20.2 核心职责

1. 维护单一真值状态
2. 执行模式切换编排
3. 管理开始、急停、安全收车
4. 管理地图当前选择
5. 管理页面会话和手动控制心跳
6. 汇总运行状态并反馈给前端

### 20.3 不应承担的职责

1. 不应重写底层运动控制
2. 不应替代现有 ROS 安全门
3. 不应把前端事件直接透传到底层

### 20.4 内部建议模块

建议逻辑上拆成：

1. 状态机管理模块
2. 模式调度模块
3. 地图管理模块
4. 手动控制输入管理模块
5. 安全动作管理模块
6. 状态汇总与推送模块

### 20.5 审核重点

1. 后端是否成为唯一控制中枢
2. 是否避免页面直通 ROS
3. 是否避免把所有逻辑塞进单一脚本难以维护

---

## 21. 功能 F17：ROS 安全接口层

### 21.1 功能目标

利用现有 ROS 链路承接网页应用层控制，而不是重造底层运动控制。

### 21.2 关键逻辑

后端任务管理器与 ROS 层之间的关系应是：

- 后端：决定“做什么”
- ROS 层：负责“如何安全地执行”

### 21.3 与现有组件的建议映射

建议复用关系如下：

1. 手动控制意图 -> `/cmd_vel_manual`
2. 自动运行意图 -> `/cmd_vel_auto`
3. 自动/手动仲裁 -> `cmd_vel_arbiter.py`
4. 安全限速与停车 -> `cmd_vel_safety_gate.py`
5. 启停姿态管理 -> `nav_mode_manager.py`
6. 正常收车 -> `safe_stop_robot.py`

### 21.4 审核重点

1. 是否尽量复用现有安全链
2. 后端是否越过安全门直接控制 `/cmd_vel`
3. 各模式是否都走统一安全接口

---

## 22. 功能 F18：看门狗与心跳

### 22.1 功能目标

在无视频流、局域网网页控制条件下，确保网络抖动或页面断开不会导致机器人继续收到陈旧控制。

### 22.2 必须存在的心跳

至少需要：

1. 页面到后端的连接心跳
2. 手动控制输入有效期
3. 后端到 ROS 关键状态检查

### 22.3 主流程

1. 页面连接建立后定期上报在线状态
2. 手动控制请求必须带时间戳
3. 后端若在规定时间内未收到有效手动输入，应自动归零
4. 若后端发现 ROS 关键组件异常，应进入 `FAULT`

### 22.4 为什么是必须项

没有看门狗时，以下问题无法接受：

1. 手动控制页面断开后机器人还在继续走
2. 自动模式组件死掉后页面仍显示“运行中”
3. 页面刷新后旧状态被误用

### 22.5 审核重点

1. 是否对手动输入设置超时
2. 是否对状态链路设置超时
3. 故障超时是否一定导致安全回退

---

## 23. 功能 F19：故障检测与故障态

### 23.1 功能目标

当底层状态与页面预期不一致、关键数据失效或任务启动失败时，系统不能装作正常，必须显式进入故障态。

### 23.2 推荐触发条件

故障可由以下情形触发：

1. 模式启动失败
2. 关键 topic 长时间无数据
3. 地图读取失败
4. 安全收车失败
5. 应用状态与 ROS 实际状态不一致

### 23.3 进入故障态后的逻辑

建议行为：

```text
停止新任务启动
-> 清零运动输出
-> 记录故障原因
-> 页面显式显示 FAULT
-> 等待人工处理
```

### 23.4 故障态与急停态区别

- `ESTOP_LATCHED`：人为紧急停止
- `FAULT`：系统自检或运行异常触发的停止

这两者不能混为一谈。

### 23.5 审核重点

1. 是否有明确故障态
2. 是否区分急停与故障
3. 故障态下是否禁止继续开始

---

## 24. 功能 F20：页面刷新与会话恢复

### 24.1 功能目标

保证页面刷新、浏览器重开、平板临时掉线时，系统行为可预测。

### 24.2 主逻辑

页面刷新后应重新拉取后端真值，而不是默认重置所有状态。

需要明确区分：

1. 页面视图丢失
2. 后端任务仍在运行
3. 手动控制输入已中断

### 24.3 建议逻辑

1. 页面刷新后重新获取状态快照
2. 若系统当前在自动模式运行，可恢复显示
3. 若系统当前在手动模式运行，但页面输入已中断，则后端应已归零
4. 若当前处于急停或故障态，页面必须立即显示锁定状态

### 24.4 审核重点

1. 页面刷新后是否总是拉后端真值
2. 手动控制是否不会因页面重连而自动恢复
3. 锁定态是否能正确恢复显示

为了让你逐一审核更高效，建议按以下顺序评审：

1. `F10 急停`
2. `F11 安全收车`
3. `F09 开始/使能`
4. `F04 模式切换`
5. `F16 后端任务管理器`
6. `F17 ROS 安全接口层`
7. `F18 看门狗与心跳`
8. `F19 故障检测与故障态`
9. `F02 虚拟摇杆输入`
10. `F03 速度设置`
11. `F12 手动控制模式`
12. `F13 手动建图模式`
13. `F14 自动探索建图模式`
14. `F15 自动巡航模式（随机漫游）`

原因：

1. 先审高风险安全逻辑
2. 再审系统控制逻辑
3. 最后审显示与交互便利性

## 26. 当前推荐结论

基于当前框架文档和仓库现状，功能实现逻辑上我给出的明确建议是：

1. **先把“开始、急停、安全收车、模式切换、故障态”定死，再讨论页面排版**
2. **先把后端任务管理器定义清楚，再讨论前端控件细节**
3. **自动巡航阶段必须在文案上明确为“随机漫游”，不能让用户误解为固定路径巡逻**
4. **地图显示区只做反馈，不做复杂编辑**
5. **在没有物理急停之前，软件逻辑必须始终保持保守策略**

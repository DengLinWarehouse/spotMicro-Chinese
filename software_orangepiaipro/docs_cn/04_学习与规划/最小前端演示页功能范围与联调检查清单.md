# 最小前端演示页功能范围与联调检查清单

> 文档定位：承接 `机器狗网页应用层开发流程文档.md` 与 `后端任务管理器模块拆分与开发任务清单.md`，用于定义前端第一阶段“最小功能演示页”的边界。  
> 文档目标：明确最小前端演示页应该做什么、不做什么、如何与后端联调、每一步要验证什么。  
> 核心原则：**这个页面是联调工具，不是正式产品界面。**

---

## 1. 结论先行

最小前端演示页的任务只有一个：

- **用最小成本验证前后端与 ROS 控制链是否真实打通**

因此第一阶段前端不追求：

1. 视觉完整度
2. 页面美化
3. 完整交互动效
4. 所有功能一次到位
5. 最终产品级体验

第一阶段前端只需要证明以下事情：

1. 页面能拉取后端真值状态
2. 页面能触发模式切换
3. 页面能触发开始 / 急停 / 安全收车
4. 页面能发送最小手动摇杆意图
5. 页面能显示地图预览占位结果
6. 页面刷新后能重新恢复状态显示

如果这些都还没稳定，就不应该进入完整前端开发。

---

## 2. 最小前端演示页的定位

### 2.1 它是什么

它是一个“前后端联调面板”，作用类似：

1. 验证器
2. 调试台
3. 状态观察面板
4. 最小手动控制入口

### 2.2 它不是什么

它不是：

1. 最终交付 UI
2. 完整视觉设计稿落地版
3. 产品级移动端交互页
4. 地图管理的最终体验页

### 2.3 为什么必须先做它

因为当前系统的主要风险在：

1. 后端状态机是否稳定
2. 状态同步是否正确
3. 按钮动作是否真的对应底层行为
4. 页面断连/刷新后系统是否仍然安全

这些问题必须在最小前端阶段暴露和修正。

---

## 3. 最小前端演示页的设计原则

### 3.1 只做验证必须项

每增加一个页面元素，都要问自己：

- 这个元素能否帮助验证前后端控制链？

如果不能，就不属于第一阶段。

### 3.2 前端不做状态推断

页面只显示后端给出的真值，不自行推断：

1. 当前是不是在运行
2. 当前是不是已使能
3. 当前是不是急停
4. 当前地图是不是已经切换成功

### 3.3 前端不直接做业务编排

页面只负责发请求，不负责：

1. 判断 ROS 是否就绪
2. 判断模式切换是否应先归零
3. 判断急停是否应抢占收车
4. 判断是否允许保存地图

这些都必须由后端决定。

### 3.4 页面必须“朴素可观察”

为了联调效率，页面应优先满足：

1. 状态清楚
2. 按钮直接
3. 反馈明确
4. 容易发现问题

而不是“好看”。

---

## 4. 最小前端演示页必须包含的功能

建议第一阶段只包含 7 个功能区块。

| 区块编号 | 区块名称 | 是否必须 | 用途 |
|----------|----------|----------|------|
| P01 | 系统状态区 | 是 | 看后端真值状态 |
| P02 | 模式切换区 | 是 | 验证模式选择与切换 |
| P03 | 动作按钮区 | 是 | 验证开始/急停/收车 |
| P04 | 最小摇杆区 | 是 | 验证手动控制链 |
| P05 | 地图预览区 | 是 | 验证地图图片链路 |
| P06 | 地图名称显示区 | 是 | 验证地图真值同步 |
| P07 | 调试反馈区 | 是 | 显示请求结果与错误信息 |

---

## 5. P01：系统状态区

### 5.1 必须显示的字段

最少显示：

1. 当前运行状态 `runtime_state`
2. 当前模式 `selected_mode`
3. 是否已使能 `armed`
4. 是否急停锁定 `estop_latched`
5. 是否故障 `fault_active`
6. 当前地图名 `selected_map_name`
7. 当前控制源 `current_control_source`
8. 后端在线状态
9. ROS 在线状态

### 5.2 为什么必须有这些字段

因为最小演示页的第一职责是验证“页面看到的，是否就是后端真值”。

### 5.3 不需要做的事情

第一阶段不需要：

1. 图标美化
2. 仪表盘式状态板
3. 高级可视化动画

### 5.4 联调检查点

1. 页面初次加载是否能正确拿到当前状态
2. 模式切换后该区是否更新
3. 急停后是否立即显示锁定态
4. 页面刷新后是否恢复显示后端真值

---

## 6. P02：模式切换区

### 6.1 必须提供的按钮

第一阶段只需要 4 个按钮：

1. 手动控制
2. 手动建图
3. 自动探索建图
4. 自动漫游巡航

### 6.2 功能目标

验证：

1. 页面能不能请求模式切换
2. 后端能不能返回成功/失败
3. 切换结果能不能同步回页面

### 6.3 页面逻辑要求

1. 点击按钮后只表示“发起切换请求”
2. 当前高亮状态必须由后端真值决定
3. 切换失败时不能直接假装页面已切换成功

### 6.4 第一阶段不做的事情

1. 模式说明弹窗
2. 引导式帮助动画
3. 复杂模式图标设计

### 6.5 联调检查点

1. 点击模式后，请求是否发送成功
2. 切换成功时页面是否显示新模式
3. 切换失败时页面是否保留旧模式显示
4. 切换过程中是否可见“处理中”状态

---

## 7. P03：动作按钮区

### 7.1 必须包含的按钮

第一阶段必须有：

1. 开始 / 使能
2. 急停
3. 安全收车

### 7.2 功能目标

验证高风险动作链是否已从前端打通到后端，再从后端打通到 ROS。

### 7.3 页面逻辑要求

1. `开始/使能`
   - 只发起请求
   - 页面不能自己假设成功
2. `急停`
   - 必须独立、显眼
   - 结果必须立即反馈
3. `安全收车`
   - 应与急停明显区分
   - 过程状态应可观察

### 7.4 第一阶段不做的事情

1. 二次确认弹窗的复杂设计
2. 产品级视觉安全提示
3. 动画化状态切换效果

### 7.5 联调检查点

1. 开始请求是否能改变状态
2. 急停是否立即进入锁存态
3. 收车是否进入 `SAFE_STOPPING`
4. 收车中急停是否能抢占

---

## 8. P04：最小摇杆区

### 8.1 功能目标

验证网页到后端的“方向/转向意图”链路是否正常，而不是做成完整产品级操控器。

### 8.2 摇杆语义必须固定

摇杆含义必须与既定设计一致：

1. 上：前进
2. 下：后退
3. 左：左转
4. 右：右转
5. 斜向：前后运动 + 转向组合

并且必须明确：

1. 不是横移
2. 不是二维平移
3. 不直接决定速度大小

### 8.3 第一阶段最小要求

第一阶段摇杆只需要做到：

1. 能拖动
2. 能回中
3. 能发送方向/转向意图
4. 页面能显示当前摇杆原始值或归一化值

### 8.4 第一阶段不做的事情

1. 高级阻尼手感
2. 动画过渡优化
3. 多手势支持
4. 视觉精修

### 8.5 联调检查点

1. 摇杆是否能正确区分前进与右转组合
2. 页面断连后是否停止输入
3. 摇杆回中后后端是否归零
4. 自动模式中手动接管语义是否符合后端逻辑

---

## 9. P05：地图预览区

### 9.1 功能目标

验证后端是否能把地图预览正确提供给前端显示。

### 9.2 第一阶段最小要求

只需要做到：

1. 能显示一张当前预览图
2. 能显示更新时间
3. 能在不同模式下正确显示或隐藏

### 9.3 必须遵守的边界

1. 地图区只负责反馈，不负责控制
2. 前端不直接处理 `.pgm`
3. 前端只显示后端转换后的标准图片资源，例如 `.png`

### 9.4 第一阶段不做的事情

1. 地图缩放
2. 地图拖拽
3. 地图标注
4. 地图编辑

### 9.5 联调检查点

1. 手动建图时是否能看到预览
2. 自动探索建图时是否能看到预览
3. 自动漫游巡航时是否能看到已选地图
4. 预览失效时是否有清晰提示

---

## 10. P06：地图名称显示区

### 10.1 功能目标

验证后端地图真值是否能被页面稳定显示。

### 10.2 第一阶段最小要求

只需要做到：

1. 显示当前地图名称
2. 未选图时显示“未选择地图”
3. 模式变化后该字段不乱跳

### 10.3 第一阶段不做的事情

1. 地图列表弹窗
2. 地图命名弹窗
3. 地图管理完整交互

这些属于后续功能页，不属于最小演示页的必须项。

### 10.4 联调检查点

1. 地图保存后名称是否能同步
2. 地图切换后名称是否能同步
3. 页面刷新后名称是否能恢复

---

## 11. P07：调试反馈区

### 11.1 功能目标

让联调人员明确知道每次动作是成功、失败还是被拒绝，以及失败原因是什么。

### 11.2 必须显示的内容

建议显示：

1. 最近一次请求类型
2. 最近一次请求结果
3. 后端返回的错误信息或禁止原因
4. 最近一次状态更新时间

### 11.3 为什么必须有这个区块

如果没有调试反馈区，联调时会出现：

1. 页面没反应，不知道是前端没发出去还是后端拒绝了
2. 模式没切成功，不知道是依赖不满足还是动作失败
3. 急停没结果，不知道是状态没回流还是动作没触发

### 11.4 第一阶段不做的事情

1. 完整日志系统
2. 可搜索日志面板
3. 日志导出

### 11.5 联调检查点

1. 每次动作后是否有可见反馈
2. 失败原因是否来自后端而不是前端猜测
3. 页面是否能区分“请求已发出”和“动作已完成”

---

## 12. 第一阶段明确不做的功能

为了避免最小演示页膨胀，以下功能明确不属于第一阶段。

### 12.1 地图相关不做项

1. 地图列表管理页
2. 地图命名弹窗的正式体验版
3. 地图删除功能
4. 地图缩略图网格浏览
5. 地图编辑

### 12.2 页面交互不做项

1. 响应式精修
2. 页面动画
3. 品牌风格设计
4. 高保真图标系统

### 12.3 控制相关不做项

1. 高级手柄适配
2. 多用户会话支持
3. 公网访问
4. 视频流

### 12.4 自动模式相关不做项

1. 固定航点巡逻 UI
2. 高级任务编排页
3. 复杂路线编辑器

## 13. 最小前端与后端的接口关系

最小前端阶段只需要两类接口能力：

### 13.1 状态读取类

至少包括：

1. 获取当前状态快照
2. 获取当前地图名称
3. 获取最近动作结果

### 13.2 动作触发类

至少包括：

1. 选择模式
2. 开始 / 使能
3. 急停
4. 安全收车
5. 提交摇杆意图

### 13.3 图片资源类

至少包括：

1. 获取地图预览图
2. 获取地图预览更新时间

### 13.4 第一阶段不需要的接口

可以后置：

1. 完整地图管理 API
2. 完整日志 API
3. 历史任务 API

## 14. 最小前端开发顺序建议

建议最小前端按下面顺序开发。

### Step 1：静态页面骨架

先把 7 个区块摆出来，不接后端。

目标：

1. 确认信息布局足够清楚
2. 便于后续逐块接入

### Step 2：状态区接后端

先只接状态读取，不接动作。

目标：

1. 验证页面能正确显示后端真值
2. 验证页面刷新后的恢复逻辑

### Step 3：动作按钮接后端

先接：

1. 模式切换
2. 开始
3. 急停
4. 安全收车

目标：

1. 验证动作链是否打通
2. 验证结果反馈区是否可用

### Step 4：摇杆区接后端

目标：

1. 验证最小手动控制链
2. 验证断连和回中逻辑

### Step 5：地图预览区接后端

目标：

1. 验证地图图片链
2. 验证不同模式下的显示逻辑

### Step 6：补强禁用逻辑与错误反馈

目标：

1. 验证不允许动作时前端显示是否一致
2. 验证失败时页面反馈是否清楚

---

## 15. 最小前端联调步骤

推荐按以下顺序联调。

### 联调步骤 1：状态快照

验证内容：

1. 页面初次加载能否获取状态
2. 页面刷新后能否重新获取状态
3. 页面显示的状态是否与后端一致

成功标准：

1. 无需手工刷新多次
2. 关键字段显示正确

### 联调步骤 2：模式切换

验证内容：

1. 手动控制模式切换
2. 手动建图模式切换
3. 自动探索建图模式切换
4. 自动漫游巡航模式切换

成功标准：

1. 页面高亮与后端真值一致
2. 切换失败时有明确提示

### 联调步骤 3：开始 / 急停 / 安全收车

验证内容：

1. 开始后状态是否变化
2. 急停后是否锁存
3. 收车中是否可观察
4. 收车中急停是否抢占

成功标准：

1. 状态流转符合设计
2. 页面反馈与后端一致

### 联调步骤 4：最小摇杆

验证内容：

1. 向上、向下、向左、向右是否语义正确
2. 右上/左上是否是前进加转向
3. 回中后是否归零
4. 页面断连后是否归零

成功标准：

1. 摇杆语义不跑偏
2. 无“横移”误解

### 联调步骤 5：地图预览

验证内容：

1. 建图模式下是否能看到预览
2. 巡航模式下是否能看到当前地图
3. 预览更新时间是否正确
4. 预览失效时是否有提示

成功标准：

1. 图片显示正常
2. 预览状态清楚

## 16. 最小前端验收清单

第一阶段前端完成后，至少要通过以下清单。

### 16.1 状态显示

- 能显示运行状态
- 能显示当前模式
- 能显示是否已使能
- 能显示是否急停锁定
- 能显示当前地图名
- 能显示后端与 ROS 在线状态

### 16.2 动作按钮

- 模式按钮可发送请求
- 开始按钮可发送请求
- 急停按钮可发送请求
- 安全收车按钮可发送请求
- 每个动作后有结果反馈

### 16.3 摇杆

- 可拖动
- 可回中
- 可发送方向/转向意图
- 与速度档解耦的语义未被破坏

### 16.4 地图预览

- 可显示预览图
- 可显示更新时间
- 不同模式下显示/隐藏正确

### 16.5 页面恢复能力

- 页面刷新后可恢复状态显示
- 断连不会导致旧状态假装仍然有效

---

## 17. 最小前端常见失败模式

在联调中尤其要防以下问题。

### 17.1 页面先行假设成功

错误表现：

- 点了模式按钮，页面先切高亮，后端实际失败

正确做法：

- 页面只根据后端真值更新

### 17.2 页面本地缓存覆盖后端真值

错误表现：

- 页面刷新后显示的还是旧地图名、旧模式

正确做法：

- 刷新后重新拉后端状态

### 17.3 摇杆被做成二维平移

错误表现：

- 用户以为右上是右前斜移

正确做法：

- 页面文案明确为“前进 + 右转”

### 17.4 地图预览和正式地图混淆

错误表现：

- 页面直接拿正式 `.pgm` 当预览

正确做法：

- 页面只显示后端转换后的标准图片资源

---

## 18. 当前最推荐的前端第一阶段边界

结合现有设计文档，我给出的明确建议是：

1. **最小前端只做联调，不做完整产品**
2. **最小前端必须能看状态、能发动作、能测摇杆、能看地图预览**
3. **地图选择、地图命名、地图保存的完整交互可以后置到正式前端**
4. **最小前端一旦能稳定验证后端逻辑，就应停止继续堆功能，转入完整前端设计**

---

## 19. 下一步建议

在这份文档确认通过后，最自然的下一步有两个方向：

1. `状态机转移表详细设计`
2. `正式前端页面模块与交互流程设计`

如果目标是先落地后端，我更建议先补：

- `状态机转移表详细设计`

因为它会直接影响后端实现和最小前端联调判断。

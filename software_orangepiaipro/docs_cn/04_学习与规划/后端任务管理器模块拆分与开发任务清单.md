# 后端任务管理器模块拆分与开发任务清单

> 文档定位：承接 `机器狗网页应用层三层架构方案框架.md`、`机器狗网页应用层功能实现逻辑详细分析.md`、`机器狗网页应用层开发流程文档.md`，把“后端先搭骨架、再细化功能”的开发路线具体落成模块划分与任务清单。  
> 文档目标：明确后端任务管理器应拆成哪些模块、模块边界是什么、谁依赖谁、先做什么后做什么，以及每项任务完成后如何验收。  
> 文档边界：本文件只做设计与开发任务分解，不给出具体代码实现。

---

## 1. 结论先行

后端任务管理器必须被当成整个网页应用层的“唯一控制中枢”，而不是一个简单的 HTTP 转发层。

明确结论：

1. 后端必须掌握 **单一真值状态**
2. 后端必须负责 **模式切换编排**
3. 后端必须负责 **开始 / 急停 / 安全收车 / 故障回退**
4. 后端必须隔离前端与 ROS，避免页面直接碰到底层控制链
5. 后端应采用 **模块化拆分**，而不是把所有逻辑堆进一个大脚本

如果后端不这样设计，后续最容易出现的问题是：

1. 页面状态和 ROS 实际状态不一致
2. 模式切换残留旧任务
3. 急停逻辑被普通页面流程干扰
4. 地图、会话、控制逻辑彼此耦合，后期难维护

---

## 2. 后端任务管理器的总体职责

后端任务管理器负责把“用户操作”转译成“受控、可验证、可回退的系统动作”。

它必须统一承担以下职责：

1. 管理系统运行状态真值
2. 管理模式真值
3. 统一接收并处理动作请求
4. 管理与 ROS 的交互边界
5. 管理手动输入与心跳超时
6. 管理地图元数据真值
7. 向前端提供统一状态视图

---

## 3. 后端模块拆分总览

建议后端逻辑上至少拆成 9 个模块。

| 模块编号 | 模块名称 | 主要职责 | 第一阶段优先级 |
|----------|----------|----------|----------------|
| M01 | StateManager | 统一保存系统真值状态 | 最高 |
| M02 | ModeManager | 模式选择、模式切换编排 | 最高 |
| M03 | ActionManager | 开始、急停、安全收车等动作入口 | 最高 |
| M04 | RosRuntimeAdapter | 与 ROS topic / launch / 脚本的统一适配层 | 最高 |
| M05 | HealthMonitor | 运行状态、故障、关键链路健康检查 | 高 |
| M06 | SessionWatchdog | 页面会话、手动输入心跳、超时归零 | 高 |
| M07 | ManualControlManager | 摇杆意图接收、速度档约束、手动输入状态 | 中高 |
| M08 | MapRegistry | 地图索引、命名、选择、保存元数据 | 中 |
| M09 | StatusGateway | 向前端输出统一状态快照与事件 | 中 |

建议理解为：

```text
前端 -> StatusGateway / ActionManager
               |
               v
   StateManager <-> ModeManager
               <-> HealthMonitor
               <-> SessionWatchdog
               <-> ManualControlManager
               <-> MapRegistry
               <-> RosRuntimeAdapter
```

## 4. 模块 M01：StateManager

### 4.1 模块目标

作为系统状态唯一真值的持有者，统一保存并更新所有关键状态。

### 4.2 必须管理的状态

建议至少包括：

- `runtime_state`
- `selected_mode`
- `armed`
- `estop_latched`
- `fault_active`
- `fault_reason`
- `selected_map_id`
- `selected_map_name`
- `speed_level`
- `last_manual_input_at`
- `ros_connected`
- `ui_connected`
- `current_control_source`

### 4.3 应负责的事情

1. 提供状态读接口
2. 提供受控状态写接口
3. 保证状态变更具备一致性
4. 对外输出统一状态快照

### 4.4 不应负责的事情

1. 不直接调用 ROS
2. 不直接执行业务动作
3. 不直接解析前端请求

### 4.5 开发任务

1. 定义状态字段集合
2. 定义状态初始值
3. 定义状态更新规则
4. 定义状态快照结构
5. 定义锁存态与回退态规则

### 4.6 验收标准

1. 任意时刻能输出完整状态快照
2. 急停、故障、模式、地图名等关键状态不会出现冲突值
3. 状态更新路径清晰、单一

---

## 5. 模块 M02：ModeManager

### 5.1 模块目标

管理模式选择与模式切换，保证切换过程中始终遵守“先归零、再脱离旧模式、再进入新模式”的规则。

### 5.2 应负责的事情

1. 处理模式选择请求
2. 检查模式切换前置条件
3. 执行模式切换编排
4. 将切换结果写回 StateManager

### 5.3 不应负责的事情

1. 不直接执行急停
2. 不直接做地图存档
3. 不直接处理页面连接问题

### 5.4 模式切换的推荐统一流程

```text
收到模式切换请求
-> 检查是否允许切换
-> 请求 ActionManager / RosRuntimeAdapter 执行旧模式脱离
-> 清零运动相关输出
-> 加载新模式配置
-> 更新 selected_mode
-> 状态回到 READY_DISARMED
```

### 5.5 开发任务

1. 固定四种模式定义
2. 为每种模式定义依赖项
3. 定义切换前检查规则
4. 定义切换失败回退规则
5. 定义切换中并发请求处理策略

### 5.6 验收标准

1. 模式切换不等于开始运行
2. 切换过程中不会保留旧模式输出
3. 切换失败时旧状态不被破坏

---

## 6. 模块 M03：ActionManager

### 6.1 模块目标

统一管理高风险动作入口，确保优先级、可观测性和回退逻辑一致。

### 6.2 应负责的动作

至少包括：

1. `start / arm`
2. `estop`
3. `safe_stop`
4. `fault_reset`
5. `map_save_request`

### 6.3 优先级规则

必须遵循：

```text
estop > fault handling > safe_stop > mode switch > start
```

### 6.4 应负责的事情

1. 接收动作请求
2. 判断动作是否允许执行
3. 调用 RosRuntimeAdapter 执行底层动作
4. 调用 StateManager 更新状态
5. 返回结构化执行结果

### 6.5 不应负责的事情

1. 不直接生成网页状态样式
2. 不直接处理地图预览图文件
3. 不应成为所有业务逻辑的杂糅层

### 6.6 开发任务

1. 定义动作枚举与输入结构
2. 定义动作执行结果结构
3. 定义急停锁存逻辑
4. 定义安全收车执行中状态
5. 定义开始失败回退逻辑

### 6.7 验收标准

1. 急停优先级最高
2. 收车和开始不能互相并发打架
3. 所有动作均返回结构化结果，不靠日志判断成功失败

---

## 7. 模块 M04：RosRuntimeAdapter

### 7.1 模块目标

将后端业务层与 ROS 实际实现隔离开，形成一个统一适配边界。

### 7.2 为什么必须单独拆

如果把 ROS topic、脚本调用、启动链、状态查询直接散落在各个模块里，会导致：

1. 后期很难维护
2. 测试替身难做
3. 模式切换、急停等动作对 ROS 的操作方式不一致

### 7.3 应负责的事情

1. 统一封装 ROS topic 发布
2. 统一封装 launch 或脚本触发
3. 统一提供 ROS 运行状态读取
4. 将 ROS 状态转换成后端可理解的结构

### 7.4 建议覆盖的能力

至少应抽象出：

1. 发布手动控制意图
2. 请求开始链路
3. 请求急停链路
4. 请求安全收车链路
5. 查询关键 topic / 节点状态
6. 查询地图链运行健康状态

### 7.5 开发任务

1. 定义 ROS 适配接口
2. 定义最小能力子集
3. 统一错误返回格式
4. 抽象出“请求成功”和“实际动作成功”区别

### 7.6 验收标准

1. 业务模块不直接操作 ROS 原语
2. 适配层可被替换或模拟
3. 后端能通过统一接口完成最小闭环

---

## 8. 模块 M05：HealthMonitor

### 8.1 模块目标

对关键运行条件做健康监测，并在异常时触发故障态。

### 8.2 应负责的事情

1. 监测 ROS 是否在线
2. 监测关键 topic 是否超时
3. 监测模式依赖是否满足
4. 监测动作执行结果是否异常
5. 生成故障原因

### 8.3 第一阶段重点监测项

建议先盯住：

1. 手动输入是否超时
2. 地图预览链是否可用
3. 建图模式是否有有效地图数据
4. 自动模式依赖是否在线
5. 急停后是否确实进入锁存态

### 8.4 开发任务

1. 定义健康检查项列表
2. 定义故障等级与触发规则
3. 定义故障态进入条件
4. 定义故障信息回写 StateManager 的方式

### 8.5 验收标准

1. 关键依赖失效时系统不会假装正常
2. 故障态和急停态有清晰区分
3. 故障原因可追踪

---

## 9. 模块 M06：SessionWatchdog

### 9.1 模块目标

处理“网页是临时连接、机器人是持续运行系统”之间的节奏差异，避免断连后保留陈旧控制。

### 9.2 应负责的事情

1. 记录页面连接状态
2. 管理前端会话生命周期
3. 管理手动输入超时
4. 超时后通知 ManualControlManager 归零

### 9.3 第一阶段必须覆盖的风险

至少要防住：

1. 页面刷新后旧手动输入残留
2. 网页关闭后机器人仍在走
3. 会话切换后旧请求还在生效

### 9.4 开发任务

1. 定义会话标识机制
2. 定义输入有效期
3. 定义超时归零动作
4. 定义页面断连后的状态处理规则

### 9.5 验收标准

1. 页面断连后不会持续保留手动输出
2. 新旧会话不会互相污染
3. 会话超时能触发安全回退

---

## 10. 模块 M07：ManualControlManager

### 10.1 模块目标

管理摇杆意图、速度档位与手动控制源状态，把前端输入翻译成“可交给 ROS 安全链处理”的手动控制意图。

### 10.2 应负责的事情

1. 接收方向/转向意图
2. 校验当前模式是否允许手动输入
3. 结合速度档位形成手动控制目标
4. 在超时或不允许时归零
5. 记录当前控制源状态

### 10.3 必须遵守的语义

1. 摇杆不表示横移
2. 摇杆不直接决定速度档
3. 速度档与摇杆语义解耦

### 10.4 开发任务

1. 定义手动意图输入结构
2. 定义速度档位到上限的映射
3. 定义手动输入允许条件
4. 定义归零规则

### 10.5 验收标准

1. 手动控制语义与设计文档一致
2. 不允许页面直接提交最终底层速度真值
3. 超时和非法模式下会自动归零

---

## 11. 模块 M08：MapRegistry

### 11.1 模块目标

管理地图的“可用身份”，使地图从文件变成可被系统识别的业务对象。

### 11.2 应负责的事情

1. 维护地图索引
2. 保存地图元数据
3. 管理显示名
4. 管理当前选中地图
5. 区分正式地图与预览图

### 11.3 不应负责的事情

1. 不直接画网页图片
2. 不直接承担建图运行逻辑
3. 不直接做动作调度

### 11.4 第一阶段优先功能

1. 地图列表获取
2. 地图选择
3. 地图保存登记
4. 地图命名

### 11.5 开发任务

1. 定义地图元数据结构
2. 定义保存后如何注册地图
3. 定义名称校验规则
4. 定义地图选择真值更新规则

### 11.6 验收标准

1. 地图显示名与底层路径分离
2. 保存后的地图能进入正式列表
3. 当前地图真值始终明确

---

## 12. 模块 M09：StatusGateway

### 12.1 模块目标

向前端暴露统一、稳定、可预测的状态视图，避免前端自己拼凑多个来源的状态。

### 12.2 应负责的事情

1. 输出统一状态快照
2. 输出模式、地图、动作执行结果
3. 提供前端可直接消费的状态结构
4. 必要时输出状态变更事件

### 12.3 不应负责的事情

1. 不决定动作是否合法
2. 不维护底层真值
3. 不直接驱动 ROS

### 12.4 开发任务

1. 定义状态快照结构
2. 定义动作返回结构
3. 定义前端读取频率/推送机制边界

### 12.5 验收标准

1. 前端不需要自行拼接复杂状态
2. 状态字段命名统一、可读
3. 前端刷新后可直接恢复当前状态显示

---

## 13. 模块依赖关系

建议依赖关系如下：

```text
StatusGateway -> StateManager

ModeManager -> StateManager
ModeManager -> RosRuntimeAdapter
ModeManager -> HealthMonitor

ActionManager -> StateManager
ActionManager -> RosRuntimeAdapter
ActionManager -> HealthMonitor

ManualControlManager -> StateManager
ManualControlManager -> RosRuntimeAdapter
ManualControlManager -> SessionWatchdog

SessionWatchdog -> StateManager

HealthMonitor -> StateManager
HealthMonitor -> RosRuntimeAdapter

MapRegistry -> StateManager

所有业务模块 -> StateManager
```

关键原则：

1. `StateManager` 是核心依赖，不反向依赖业务模块
2. `RosRuntimeAdapter` 是能力依赖，不持有业务真值
3. `StatusGateway` 只读状态，不直接决策

---

## 14. 第一阶段必须做的模块子集

如果要先完成“后端骨架”，第一阶段不需要 9 个模块都做满，但建议至少把以下 6 个模块先立起来：

1. `M01 StateManager`
2. `M02 ModeManager`
3. `M03 ActionManager`
4. `M04 RosRuntimeAdapter`
5. `M06 SessionWatchdog`
6. `M09 StatusGateway`

原因：

1. 这 6 个模块已经足够支撑最小闭环
2. `HealthMonitor`、`ManualControlManager`、`MapRegistry` 可先留简化版

## 15. 后端开发任务清单

下面给出建议的开发任务拆分顺序。

---

## 16. Task 01：统一状态模型

### 16.1 目标

先把系统的状态词汇统一，否则后续所有模块都会各自定义一套状态。

### 16.2 内容

1. 定义状态枚举
2. 定义模式枚举
3. 定义动作枚举
4. 定义控制源枚举
5. 定义故障类型枚举

### 16.3 依赖

无

### 16.4 完成标准

后端所有模块都引用同一套状态定义。

---

## 17. Task 02：StateManager 骨架

### 17.1 目标

提供状态读写和快照能力。

### 17.2 内容

1. 状态初始化
2. 状态更新接口
3. 状态快照接口
4. 锁存态字段支持

### 17.3 依赖

- Task 01

### 17.4 完成标准

能稳定读出统一状态快照。

---

## 18. Task 03：动作与结果模型

### 18.1 目标

统一开始、急停、安全收车、保存地图等动作的输入输出结构。

### 18.2 内容

1. 动作请求结构
2. 动作结果结构
3. 错误码结构
4. 禁止原因结构

### 18.3 依赖

- Task 01

### 18.4 完成标准

所有动作请求都有统一返回格式。

---

## 19. Task 04：RosRuntimeAdapter 最小能力

### 19.1 目标

先建立最小 ROS 适配层，不要求一次覆盖所有功能。

### 19.2 最小能力建议

1. 查询 ROS 在线状态
2. 请求开始链路
3. 请求急停链路
4. 请求安全收车链路
5. 发布手动控制占位输入

### 19.3 依赖

- Task 01
- Task 03

### 19.4 完成标准

后端可不依赖前端，单独对 ROS 发出最小控制请求。

---

## 20. Task 05：ActionManager 骨架

### 20.1 目标

先让高风险动作有统一入口。

### 20.2 内容

1. 开始请求入口
2. 急停请求入口
3. 安全收车入口
4. 动作执行结果回写 StateManager

### 20.3 依赖

- Task 02
- Task 03
- Task 04

### 20.4 完成标准

动作入口已存在，且结果能回写状态。

---

## 21. Task 06：ModeManager 骨架

### 21.1 目标

先支持模式选择与简单切换，不急于做所有模式依赖细节。

### 21.2 内容

1. 模式选择入口
2. 模式切换合法性检查
3. 简化版切换回退逻辑

### 21.3 依赖

- Task 02
- Task 04

### 21.4 完成标准

能完成“选模式 -> 状态切到 READY_DISARMED”的最小链路。

---

## 22. Task 07：StatusGateway 最小接口

### 22.1 目标

给最小前端演示页提供可消费的状态读取能力。

### 22.2 内容

1. 当前状态接口
2. 当前模式接口
3. 当前地图名接口
4. 当前动作结果读取能力

### 22.3 依赖

- Task 02

### 22.4 完成标准

最小前端可以只靠这个模块完成状态显示。

---

## 23. Task 08：SessionWatchdog 最小能力

### 23.1 目标

先把“页面断连后不要持续输出旧控制”这个关键问题堵住。

### 23.2 内容

1. 页面会话标识
2. 手动输入超时
3. 超时归零通知

### 23.3 依赖

- Task 02
- Task 04

### 23.4 完成标准

页面断连或输入超时后，后端能进入手动归零路径。

---

## 24. Task 09：ManualControlManager 最小能力

### 24.1 目标

先支持最小摇杆联调，不急于把控制曲线一次做到最细。

### 24.2 内容

1. 接收方向/转向意图
2. 校验模式是否允许手动控制
3. 结合速度档进行最小映射
4. 超时归零

### 24.3 依赖

- Task 02
- Task 04
- Task 08

### 24.4 完成标准

最小前端摇杆能打通方向/转向控制链。

---

## 32. 不推荐的开发顺序

以下顺序不推荐：

1. 先做地图功能，再补状态机
2. 先做完整摇杆交互，再补会话超时
3. 先做完整前端，再定义状态快照
4. 先把模式切换写死在前端，再让后端跟着适配

原因很简单：

1. 会造成控制逻辑分散
2. 返工概率极高
3. 安全逻辑后置，风险不可接受

# 机器狗网页应用层骨架阶段实际测试反馈总结

> 文档日期：2026-04-17  
> 文档定位：总结“机器狗网页应用层骨架设计完成后”的实际联调反馈、问题闭环情况、当前验证边界与下一阶段建议。  
> 文档边界：本文件只记录已经发生的实际测试、日志现象、话题联调结果与工程判断，不展开新的功能设计细节。

---

## 1. 结论先行

截至 2026-04-17，机器狗网页应用层骨架阶段已经完成了“网页最小演示页 + 后端骨架 + ROS 最小控制链”的第一轮联调，结论如下：

1. **前端最小演示页已经可用**  
   页面可以正常打开，中文显示正常，状态区、模式切换、速度档位、开始、安全收车、急停、虚拟摇杆等基础交互均有反馈。

2. **后端骨架已经成为当前应用层的状态汇总入口**  
   HTTP API 可访问，页面刷新后能重新读取后端真值，而不是回到假的初始界面状态。

3. **开始 / 安全收车 / 急停 / 站立 / 行走的控制链路已经软件层打通**  
   在 `dry_run=False` 条件下，后端能够向 `/stand_cmd`、`/walk_cmd`、`/idle_cmd` 发布控制指令，`spot_micro_motion_cmd_node` 已确认订阅这些话题。

4. **手动摇杆控制链已从“部分打通”升级为“软件闭环打通”**  
   经过补丁修正后，网页摇杆输入可以同时发布到 `/cmd_vel_manual` 与 `/cmd_vel`，并且通过 `rostopic echo` 已看到符合预期的 `Twist` 数据变化。

5. **当前仍不能宣布“实机运动通过”**  
   由于测试期间机器狗电池没电，尚未完成实机负载运动验证，因此目前只能确认“软件链路通过”，不能确认“整机动作稳定通过”。

---

## 2. 本阶段测试目标

本阶段不是做完整功能验收，而是验证以下最小闭环是否成立：

```text
网页操作
-> 后端状态机 / HTTP API
-> ROS 应用层节点
-> 运动控制相关话题
-> 底层运动节点接收
```

具体检查点包括：

1. 页面是否能打开并正确显示状态
2. 页面操作是否能驱动后端状态变化
3. 后端是否能通过 HTTP API 对外提供真实状态
4. 后端是否能将启动、收车、急停等动作下发到 ROS
5. 摇杆是否能形成手动速度指令输出
6. 页面刷新后是否仍能恢复后端真值

---

   - 页面行为

## 4. 已完成的实际测试与反馈

## 4.1 前端页面与状态显示测试

用户反馈结果如下：

1. 页面可以完整打开
2. 状态区有内容，且内容正确
3. 页面无空白块、无错位、无明显报错
4. 请求反馈区可以显示成功/失败信息
5. 页面操作过程中无明显卡顿

工程判断：

1. 最小演示前端已经满足联调用途
2. 当前前端可以作为后端功能验证面板继续使用
3. 暂时不需要优先开发完整视觉版前端，应继续以后端功能闭环为主

---

## 4.2 模式切换与状态回显测试

用户反馈结果如下：

1. 模式切换按钮点击后有反应
2. 当前模式会更新
3. 页面刷新后，能够重新显示后端真值
4. 页面刷新后，不会回到假的默认初始状态

工程判断：

1. 前端并非纯本地假状态，而是已经依赖后端状态接口
2. 后端“唯一真值中枢”的骨架方向是正确的

## 4.3 开始 / 安全收车 / 急停测试

用户反馈结果如下：

1. 开始按钮交互正常
2. 安全收车交互正常
3. 急停交互正常

后续 ROS 联调观察结果：

1. `/stand_cmd` 有实际发布

3. `/idle_cmd` 有实际发布
4. `spot_micro_motion_cmd_node` 已订阅上述话题

工程判断：

3. 但这里仍是“控制链软件通过”，不是“机体动作通过”

---

## 4.4 速度档位测试

用户反馈结果如下：

1. 速度档位加减正常
2. 页面状态能反映速度档位变化

结合后续话题观察结果：

1. 在 1 档时，前进速度输出为 `linear.x = 0.03`
2. 输出量级与配置 `speed_levels_mps: [0.0, 0.03, 0.06, 0.09, 0.12]` 一致

工程判断：

1. 当前“方向由摇杆给出、速度由档位给出”的设计已经落实到数据输出层
2. 这符合前面功能逻辑分析中的设计目标

---

## 4.5 虚拟摇杆测试

用户反馈结果如下：

1. 摇杆可以拖动
2. 摇杆数值会变化
3. 右上角行为符合“前进 + 右转”的设计理解
4. 松手后摇杆会自动回中

后续 ROS 话题验证结果：

在 `dry_run=False` 条件下，`rostopic echo /cmd_vel` 与 `rostopic echo /cmd_vel_manual` 已观测到如下现象：

1. 中立时输出全 0
2. 向上推动时：
   - `linear.x = 0.03`
   - `angular.z = 0.0`
3. 推向右上角时：
   - `linear.x = 0.03`
   - `angular.z` 从 `0.028...` 逐步增大到约 `0.115`
4. 松手后再次回到全 0

工程判断：

1. 当前摇杆逻辑符合既定设计：  
   **摇杆位移表达的是运动方向变化，而不是速度幅值变化。**
2. 当前输出符合“向前运动时逐渐右转”，而不是“横移”或“斜向平移”
3. 手动控制主链已经实现基础可用

## 5. 联调过程中发现的问题与闭环情况

本阶段不是一次性通过，而是经历了几轮典型工程问题暴露与修复。

| 问题项 | 实际现象 | 原因判断 | 处理结果 |
|--------|----------|----------|----------|
| 后端 ROS 包找不到 | `rospack find spot_micro_app_backend` 报未找到 | 工作区链接脚本未包含新包 | 已将 `spot_micro_app_backend` 纳入链接流程并重新编译 |
| 后端节点启动崩溃 | `TypeError: Attempted to reuse key: 'SET_SPEED_LEVEL'` | 枚举定义重复 | 已修复 |
| 局域网无法访问网页 | 本机 `curl 127.0.0.1` 可用，但 Windows 端访问失败 | `ufw` 未放行 8765 端口 | 已放行，局域网可访问 |
| 网页状态正常但 ROS 无运动话题输出 | `/cmd_vel`、`/cmd_vel_manual` 无数据 | 后端仍处于 `dry_run=True` | 已改为 `dry_run=False` |
| 手动摇杆初期未闭环 | `/cmd_vel_manual` 发布但底层未消费 | 当前最小系统未启用完整 arbiter/safety-gate | 已新增阶段性直桥：手动模式下同步发往 `/cmd_vel` |

工程判断：

1. 这些问题都属于典型的系统集成问题，而不是方案方向错误
2. 当前大部分关键阻塞项已经闭环
3. 剩余最大不确定性已经从“软件集成”转移为“实机行为”

## 6. 当前验证结论分级

为避免误判，需要把“软件通过”和“实机通过”分开。

### 6.1 已通过项

以下项目可以判定为**已通过**：

1. 前端最小演示页访问与交互
2. 中文界面显示
3. 后端 HTTP API 基础可用
4. 后端状态真值回显
5. 模式切换基础链路
6. 开始 / 安全收车 / 急停动作入口
7. `/stand_cmd`、`/walk_cmd`、`/idle_cmd` 到底层订阅节点的链路
8. 摇杆到 `/cmd_vel_manual` 的链路
9. 摇杆到 `/cmd_vel` 的阶段性桥接链路
10. 页面刷新后的状态恢复

### 6.2 部分通过项

以下项目可以判定为**部分通过**：

1. 手动运动控制  
   - 软件层面：通过  
   - 实机层面：未验证

2. 机器人启停流程  
   - 命令链层面：通过

### 6.3 尚未验证项

以下项目目前仍不能宣称通过：

1. 实机站立稳定性
2. 实机低速前进稳定性

4. 实机急停后的姿态安全性

6. 地图显示、地图保存、地图命名、地图选择
7. 手动建图、自动探索建图、自动漫游巡航

## 7. 关键工程判断

### 7.1 当前骨架方向是正确的

从测试反馈看，三层架构的推进方向是正确的：

```text
网页前端
-> 后端任务管理器
-> ROS 话题层
```

说明当前没有走向“网页直接碰 ROS”的混乱路径。

### 7.2 最小前端策略是正确的

如果一开始就开发完整前端，这几轮修复会非常痛苦。  
现在先用最小演示页把：

1. 状态定义
2. 控制入口
3. HTTP 接口
4. ROS 联调

先跑通，工程成本明显更低。

### 7.3 当前最大的剩余风险已经变成安全与实机

下一阶段最大的风险不再是网页显示，而是：

1. 电池恢复后实机是否按预期动作
2. 没有物理急停断电按钮时的现场安全风险
3. 当前直桥 `/cmd_vel` 方案只是阶段性联调手段，不应长期替代完整安全链

---

## 8. 当前存在的风险与注意事项

| 风险项 | 严重度 | 发生概率 | 触发条件 | 缓解措施 |
|--------|--------|----------|----------|----------|
| 无物理断电急停 | 高 | 中 | 实机异常动作、软件急停失效 | 首次实机测试必须有人值守并准备人工断电；尽快补充硬件急停 |
| 直桥 `/cmd_vel` 绕过完整仲裁 | 中 | 高 | 后续多控制源并行时 | 当前仅用于最小联调；后续回归 `cmd_vel_arbiter + safety_gate` |
| 电池没电导致无法完成实机闭环 | 中 | 高 | 当前阶段 | 先完成软件验证，待电池恢复后再做悬空与落地测试 |
| `rostopic info /cmd_vel` 曾出现 `Publishers: None` | 低 | 中 | 查询时机与 ROS 注册状态不同步 | 以 `rostopic echo` 是否实际收到消息作为更高优先级判据 |
| 无视频流 | 中 | 中 | 后续建图或巡航时环境反馈不足 | 依赖地图预览与状态提示，但不能把地图显示区做成控制主链 |

---

## 9. 下一阶段建议

下一阶段不建议立刻扩展很多新界面元素，而是建议按以下顺序推进：

1. **先做实机安全验证**
   - 电池恢复后，先悬空测试
   - 速度只用 1 档
   - 先验证开始、回零、收车、急停
   - 再验证短距离前进和小角度转向

2. **再细化地图相关能力**
   - 地图预览生成
   - 地图保存按钮
   - 地图命名
   - 地图选择与显示
   - 地图快照清理策略

3. **最后再完善完整前端**
   - 当前最小前端已经足够支撑后端功能联调
   - 完整 UI 应放在后端行为稳定之后

---

## 10. 最终总结

本阶段最重要的结果，不是“网页已经做出来了”，而是：

1. **应用层骨架已经真正落到了系统上**
2. **前端、后端、ROS 之间已经形成最小可验证闭环**
3. **手动控制链已经从 UI 行为正确，推进到 ROS 速度话题输出正确**
4. **当前阻塞点已经从软件集成，转移到了实机安全验证**

因此，现阶段最准确的阶段性结论应当是：

> 机器狗网页应用层骨架阶段已经完成，并已通过前端交互测试、HTTP 状态回显测试、ROS 话题联调测试与手动摇杆软件闭环测试；但由于尚未完成带电实机验证，当前结果只能判定为“软件链路通过，实机效果待验证”。

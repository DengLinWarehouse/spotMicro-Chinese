# 机器狗网页应用层三层架构方案框架

> 文档状态：方案框架版，待确认后再细化接口、页面元素、状态转移表和实现清单。  
> 适用对象：当前已具备底层运动能力、导航/建图基础能力，但上机操作复杂、模式切换依赖命令行的 SpotMicro 系统。  
> 当前约束：局域网访问、无视频流、无物理急停断电按钮、自动巡航定义为随机漫游而非固定航点巡逻。

---

## 1. 目标与结论

当前系统的主要问题不是“不能动”，而是“使用复杂、状态不透明、模式切换不安全”。  
本方案的目标是将现有“命令集合式操作”升级为“状态机驱动的应用层操作”。

明确结论：

1. 应采用 **极简网页前端 + 后端任务管理器 + ROS 安全接口层** 的三层架构。
2. 网页前端不直接操作底层 ROS 话题，不直接承担模式切换、安全联锁和流程编排。
3. “开始/停止”不能只做成页面按钮语义，必须绑定到底层运行状态、模式互锁和安全流程。
4. “停止”必须拆分为：
   - **急停**：优先实现 `<100 ms` 进入零速度链路
   - **安全收车**：零速 -> 站稳 -> 趴下
5. 由于当前 **没有物理急停断电按钮**，第一阶段只能作为 **低速、室内、测试人员在场** 的受控系统使用，不能把软件急停视为最终安全方案。

## 4. 三层架构总览

## 4.1 架构分层

```text
[Layer 1] 极简网页前端
  - 摇杆输入
  - 速度档位
  - 模式切换
  - 地图预览
  - 地图选择/命名
  - 开始/急停/安全收车
  - 系统状态显示

           |
           | HTTP / WebSocket
           v

[Layer 2] 后端任务管理器
  - 应用状态机
  - 模式切换编排
  - 任务启动/停止
  - 看门狗与心跳
  - 急停锁存
  - 地图索引管理
  - 页面状态汇总

           |
           | ROS Topic / Service / roslaunch / 脚本调用
           v

[Layer 3] ROS 安全接口层
  - /cmd_vel_manual
  - /cmd_vel_auto
  - cmd_vel_arbiter.py
  - cmd_vel_safety_gate.py
  - nav_mode_manager.py
  - safe_stop_robot.py
  - slam / explorer / patrol runtime
```

## 4.2 各层职责

### Layer 1：网页前端

职责：

1. 提供极简可用的人机界面
2. 只发送高层操作意图
3. 显示当前系统状态和地图预览

禁止承担：

1. 直接控制 ROS 模式状态
2. 直接发布底层高风险指令
3. 直接管理 launch 生命周期

### Layer 2：后端任务管理器

职责：

1. 作为网页与 ROS 的唯一控制中枢
2. 维护应用状态机
3. 统一执行模式切换、急停、安全收车
4. 对前端提供稳定接口
5. 做连接超时归零、故障锁存和任务恢复判定

### Layer 3：ROS 安全接口层

职责：

1. 接收后端的控制意图
2. 复用现有速度仲裁和安全限幅链路
3. 维持运动控制、建图和自动漫游运行
4. 在异常时强制归零并回到安全状态

## 5. 模式定义

为避免“同一个名字对应多种运行内容”，应用层统一采用以下模式定义。

| 模式 | 目的 | 典型运行组件 | 是否允许运动 | 是否显示地图预览 |
|------|------|--------------|--------------|------------------|
| 手动控制 | 纯手动行走/转向 | `motion_cmd` | 是 | 否 |
| 手动建图 | 人工控制机器人边走边建图 | `motion_cmd + SLAM` | 是 | 是 |
| 自动探索建图 | 低速随机探索并持续建图 | `motion_cmd + SLAM + explorer + arbiter + safety_gate` | 是 | 是 |
| 自动巡航 | 已加载地图后随机漫游 | `motion_cmd + map load + auto runtime + safety_gate` | 是 | 是，显示当前地图 |

说明：

1. 当前“自动巡航”按你的要求定义为 **随机漫游**，不是固定路线航点巡逻。
2. 为避免歧义，后续界面文案建议使用：
   - `自动探索建图`
   - `自动漫游巡航`
3. 若未来需要固定路线巡逻，应新增独立模式，不建议复用“自动巡航”名称。

## 6. 应用状态机框架

应用层不再以“当前页面点了哪个按钮”为真值，而以“系统处于什么状态”为真值。

建议状态机如下：

```text
DISCONNECTED
  -> READY_DISARMED
  -> STARTING
  -> ARMED_READY
  -> RUNNING_MANUAL
  -> RUNNING_MANUAL_MAPPING
  -> RUNNING_AUTO_EXPLORE
  -> RUNNING_AUTO_PATROL
  -> SAFE_STOPPING
  -> ESTOP_LATCHED
  -> FAULT
```

### 6.1 状态语义

- `READY_DISARMED`
  - 系统在线，但未允许运动
- `ARMED_READY`
  - 已使能，但尚未进入实际运动输出
- `RUNNING_*`
  - 当前模式正在执行
- `SAFE_STOPPING`
  - 正在执行安全收车序列
- `ESTOP_LATCHED`
  - 急停已触发，必须人工确认后才能恢复
- `FAULT`
  - 后端检测到底层状态不一致、关键 topic 超时或任务启动失败

### 6.2 核心规则

1. 模式切换必须先退回到安全中间态，不能直接热切。
2. 急停优先级高于一切模式与页面操作。
3. 安全收车不是急停。
4. 未使能状态下允许做配置、选图、命名、查看地图，但不允许机器人运动。
5. 一旦进入 `ESTOP_LATCHED`，必须人工复位，不能自动恢复。

## 7. 页面功能框架

本阶段只确定功能框架，不下沉到具体交互细节。

### 7.1 页面核心区块

建议页面只保留以下 6 个区域：

1. **连接与系统状态区**
   - 在线状态
   - 当前模式
   - 是否已使能
   - 是否急停锁定
   - 当前地图名
2. **运动控制区**
   - 虚拟摇杆
   - 速度加减按钮
3. **模式切换区**
   - 手动控制
   - 手动建图
   - 自动探索建图
   - 自动漫游巡航
4. **地图显示区**
   - 仅在手动建图、自动探索建图、自动漫游巡航时显示
5. **地图管理区**
   - 地图选择
   - 地图命名
   - 当前地图名显示
6. **控制执行区**
   - 开始/使能
   - 急停
   - 安全收车

### 7.2 交互原则

1. 页面默认进入未使能状态
2. 速度初始值为 `0`
3. 模式切换后默认不自动开始运动
4. 地图预览不是精细编辑器，而是运行态反馈窗口
5. 危险动作按钮必须明显区分颜色与优先级

## 8. 开始、急停、安全收车的框架定义

这是本方案的关键约束，必须先定清楚。

### 8.1 开始/使能

定义：

- 允许当前模式进入可执行状态
- 不等于“页面一按立即高速运动”

前置条件示例：

- 手动控制：底层控制在线
- 手动建图：雷达与 SLAM 在线
- 自动探索建图：自动探索链路在线
- 自动漫游巡航：已选择地图，地图可读，自动链路在线

### 8.2 急停

定义：

- 最高优先级的立即停止动作
- 目标是尽快让机器人退出运动指令链路

注意：

1. 当前系统无物理急停断电按钮，网页急停只能算“软件急停”
2. 软件急停不能替代硬件急停
3. 第一阶段必须限制机器人速度和测试环境

### 8.3 安全收车

定义：

- 在任务结束或人工结束运行时，让机器人按较平稳流程停机

推荐流程：

```text
disable auto
-> zero cmd_vel
-> stand
-> idle / lie down
```

这与现有 `safe_stop_robot.py` 的职责一致。

## 9. 地图与反馈框架

由于当前不需要视频流，网页端对用户的主要反馈来自：

1. 地图预览
2. 状态文本
3. 模式状态
4. 当前地图名称

### 9.1 地图显示原则

1. 手动建图和自动探索建图时显示地图快照
2. 自动漫游巡航时显示当前已选择地图
3. 地图显示应带更新时间，避免把旧图误认为实时图

### 9.2 地图管理原则

1. 地图文件与显示名称分离
2. 地图命名应通过元数据实现，而不是直接依赖文件重命名
3. 用户确认选图后，页面主状态区应持续显示当前地图名

## 10. 安全约束

当前系统的最大短板是 **无物理急停断电按钮**。  
因此应用层设计必须主动收紧运行边界。

### 10.1 第一阶段强制约束

1. 仅允许局域网内访问
2. 仅允许低速运行
3. 默认速度档从 `0` 起步
4. 人员必须在场
5. 室内空旷环境优先
6. 未使能时任何模式都不得输出运动指令

### 10.2 建议的初始速度边界

建议第一阶段应用层把速度限制在：

- 手动/自动前向速度 `<= 0.10 ~ 0.12 m/s`
- 角速度 `<= 0.18 ~ 0.25 rad/s`

原则：

1. 先以稳定与可控为目标
2. 不追求 UI 上看起来“灵敏”
3. 不允许网页端越过安全门直接放大速度

### 10.3 不可忽视的风险

| 风险项 | 说明 | 当前结论 |
|--------|------|----------|
| 无物理急停 | 软件失效时无法硬切动力链 | 高风险，后续必须补 |
| 浏览器断连 | 手动输入可能丢失 | 必须依赖后端看门狗归零 |
| 模式残留 | 切换时旧任务未停止 | 必须统一由后端管理 |
| 旧地图误用 | 用户不知道当前加载哪张图 | 必须显式显示地图名 |

## 11. 第一阶段推荐落地范围

为降低集成风险，建议第一阶段只做“可用闭环”，不做过度扩展。

### 第一阶段应完成

1. 三层架构基本骨架
2. 四种模式的统一入口
3. 开始/急停/安全收车分离
4. 地图预览基础显示
5. 地图选择与地图名显示
6. 后端状态机与故障锁存

### 第一阶段暂不追求

1. 视频流
2. 公网远控
3. 多用户并发控制
4. 固定航点巡逻
5. 精细地图编辑
6. 复杂任务编排

## 13. 当前建议的决策结论

在你给定的约束下：

1. **采用三层架构是正确且必要的**
2. **局域网部署是合理的第一阶段方案**
3. **无视频流不会阻碍第一阶段落地**
4. **自动巡航按随机漫游定义可以先做，但界面命名建议避免与固定路线巡逻混淆**
5. **无物理急停是当前最大安全短板，必须在文档和实现中持续显式标注**

如果本框架确认通过，下一步建议输出：

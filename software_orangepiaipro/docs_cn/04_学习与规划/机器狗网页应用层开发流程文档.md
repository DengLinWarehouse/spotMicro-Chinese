# 机器狗网页应用层开发流程文档

> 文档定位：基于 `机器狗网页应用层三层架构方案框架.md` 与 `机器狗网页应用层功能实现逻辑详细分析.md`，给出一份可执行的开发推进顺序。  
> 开发策略：**后端先搭骨架，再逐项细化功能；前端先做最小功能演示，用于联调与验证，待后端稳定后再开发完整前端。**  
> 文档边界：本文件只定义开发流程、阶段目标、交付物、验证方法与依赖顺序，不展开具体代码实现。

---

## 1. 结论先行

这套网页应用层不适合采用“先把页面做出来，再倒逼后端适配”的流程。  
正确的推进方式应当是：

1. **先把后端任务管理器做成唯一真值中枢**
2. **先打通后端到 ROS 的最小闭环**
3. **前端第一阶段只做最小演示页，用来验证接口、状态和控制链**
4. **等后端功能和状态机稳定后，再做完整前端**

明确推荐的总体顺序：

```text
需求冻结
-> 后端骨架
-> ROS 适配最小闭环
-> 最小前端演示
-> 后端安全逻辑与模式逻辑细化
-> 地图与任务功能细化
-> 完整前端开发
-> 联调与回归
-> 试运行
```

这样做的原因是：

1. 当前项目的核心风险在状态机、安全联锁、模式切换，而不在界面样式
2. 如果前端先做完整，很容易反复改接口、改状态定义、改按钮逻辑
3. 后端如果没有先稳定，前端越完整，后续重构成本越大

---

## 2. 开发总原则

### 2.1 后端优先原则

后端必须先成为唯一控制中枢，再允许前端逐步接入。

后端优先实现的内容包括：

1. 应用状态机骨架
2. 模式管理骨架
3. 急停/安全收车/开始的动作入口
4. 与 ROS 安全接口层的最小连接
5. 状态汇总与查询接口

### 2.2 最小闭环原则

每一阶段必须形成一个可以验证的闭环，而不是先堆大量未验证模块。

示例：

- 能查询状态
- 能选择模式
- 能开始
- 能急停
- 能看到状态变化

这些比“页面很漂亮”更优先。

### 2.3 风险先行原则

开发顺序要按风险优先级排序：

```text
急停/收车/故障
-> 模式切换
-> 开始/使能
-> 后端状态机
-> ROS 对接
-> 手动控制
-> 地图保存/选择
-> 自动探索/自动巡航
-> 完整前端交互
```

### 2.4 前端延迟美化原则

完整前端必须后置。  
第一阶段前端只做验证，不做风格优化，不做复杂动画，不做页面包装。

---

## 3. 总体开发阶段划分

建议把开发分成 8 个阶段。

| 阶段 | 名称 | 目标 |
|------|------|------|
| Phase 0 | 设计冻结 | 冻结状态定义、功能边界、模式边界 |
| Phase 1 | 后端骨架 | 建立任务管理器和状态机框架 |
| Phase 2 | ROS 最小闭环 | 打通后端到 ROS 的最小安全控制链 |
| Phase 3 | 最小前端演示 | 用简单页面验证接口与状态变化 |
| Phase 4 | 后端核心功能细化 | 完成开始/急停/收车/模式切换/故障态 |
| Phase 5 | 后端地图与任务功能细化 | 完成地图保存、地图索引、自动模式相关逻辑 |
| Phase 6 | 完整前端开发 | 开发正式页面、完整控件与交互 |
| Phase 7 | 联调、回归与试运行 | 做系统级验证与试运行 |

---

## 4. Phase 0：设计冻结

### 4.1 阶段目标

在开始实现前，冻结会影响后续接口的大项定义，避免边开发边改基础语义。

### 4.2 必须冻结的内容

1. 四种模式定义：
   - 手动控制
   - 手动建图
   - 自动探索建图
   - 自动漫游巡航
2. 核心状态集合：
   - `READY_DISARMED`
   - `STARTING`
   - `ARMED_READY`
   - `RUNNING_*`
   - `SAFE_STOPPING`
   - `ESTOP_LATCHED`
   - `FAULT`
3. 三个关键动作定义：
   - 开始/使能
   - 急停
   - 安全收车
4. 摇杆控制语义：
   - 方向与转向意图
   - 不表示横移
   - 不直接表示速度大小
5. 地图管理语义：
   - 预览图与正式地图分离
   - 手动保存与自动保存分离

### 4.3 交付物

1. 已确认的方案框架文档
2. 已确认的功能实现逻辑详细分析文档
3. 一份统一的术语表

### 4.4 验收标准

开发成员对以下问题给出一致答案：

1. 什么叫“已使能”
2. 什么叫“急停”
3. 什么叫“安全收车”
4. 自动巡航当前是不是固定路径
5. 地图保存和地图预览是否是同一回事

---

## 5. Phase 1：后端骨架

### 5.1 阶段目标

先建立“能承载后续所有功能”的后端骨架，但不急于把所有具体功能一次写满。

### 5.2 本阶段只做骨架，不做复杂功能

本阶段重点是“框架到位”，不是“所有功能完成”。

应优先具备的后端模块：

1. 应用状态机模块
2. 模式选择模块
3. 基础动作调度模块
4. 状态查询模块
5. 与 ROS 通信的抽象接口模块
6. 日志与错误记录骨架

### 5.3 本阶段建议的内部模块划分

后端逻辑上建议拆成：

1. `StateManager`
   - 管理运行状态真值
2. `ModeManager`
   - 管理模式切换请求与前置条件
3. `ActionManager`
   - 管理开始、急停、安全收车动作入口
4. `RosBridgeAdapter`
   - 屏蔽前端对 ROS 的直接感知
5. `SessionWatchdog`
   - 管理页面连接和输入超时
6. `MapRegistry`
   - 预留地图索引接口

### 5.4 本阶段不急于完成的内容

1. 地图保存细节
2. 地图缩略图细节
3. 完整网页页面
4. 所有模式的完整启动链

### 5.5 交付物

1. 后端目录结构
2. 统一状态定义
3. 统一错误码/动作返回结构
4. 基础状态查询接口
5. 基础动作入口占位

### 5.6 验收标准

至少可以做到：

1. 查询当前系统状态
2. 设置当前模式
3. 请求开始/急停/安全收车
4. 返回“成功 / 失败 / 不允许”的结构化结果

---

## 6. Phase 2：ROS 最小闭环

### 6.1 阶段目标

在不依赖完整前端的情况下，先让后端与现有 ROS 安全链打通。

### 6.2 本阶段重点

重点不是“网页能不能好看”，而是验证：

1. 后端能不能正确调用 ROS 层
2. 状态能不能从 ROS 回流到后端
3. 急停、开始、收车这些高风险动作能不能闭环

### 6.3 推荐先打通的功能链

按顺序建议先打通：

1. 状态查询
2. 模式选择
3. 开始/使能
4. 急停
5. 安全收车
6. 手动速度最小链

### 6.4 本阶段推荐复用的 ROS 基础

重点复用：

1. `cmd_vel_arbiter.py`
2. `cmd_vel_safety_gate.py`
3. `nav_mode_manager.py`
4. `safe_stop_robot.py`

### 6.5 本阶段建议的验证方式

无需完整前端，可通过以下方式验证：

1. 命令行触发后端接口
2. 极简调试页
3. 手工观察 ROS topic 与状态变化

### 6.6 交付物

1. 后端到 ROS 的接口适配层
2. 状态同步最小闭环
3. 最小动作执行闭环

### 6.7 验收标准

必须实际验证：

1. 后端请求开始 -> 状态变化正确
2. 后端请求急停 -> 进入锁定态
3. 后端请求安全收车 -> 进入收车流程
4. 页面或测试工具断开 -> 手动输入超时归零

---

## 7. Phase 3：最小前端演示

### 7.1 阶段目标

开发一个极简、丑一点也没关系的前端页面，只用于验证前后端接口是否正确、状态是否能显示、动作是否能触发。

### 7.2 这个前端的定位

它不是正式产品界面，而是一个联调工具。

它只需要具备：

1. 状态显示
2. 模式切换
3. 开始按钮
4. 急停按钮
5. 安全收车按钮
6. 最简单的摇杆演示
7. 地图预览占位显示

### 7.3 为什么必须先做最小演示页

原因：

1. 纯后端调试无法充分暴露前后端状态同步问题
2. 纯完整前端开发成本太高，且接口还在变
3. 最小演示页最适合做协议和状态流转验证

### 7.4 本阶段禁止追求的事情

1. 视觉风格
2. 响应式细节
3. 页面动画
4. 完整地图管理体验
5. 所有模式全量完成

### 7.5 最小前端建议页面元素

建议只保留：

1. 当前状态文本
2. 当前模式文本
3. 当前地图名文本
4. 模式按钮
5. 开始/急停/安全收车按钮
6. 一个最简摇杆区
7. 一个地图图片区域

### 7.6 交付物

1. 最小联调前端页面
2. 前后端状态同步演示
3. 最小摇杆验证链

### 7.7 验收标准

必须验证：

1. 页面刷新后状态能重新拉取
2. 点击模式切换后页面状态跟随后端真值变化
3. 点击开始/急停/安全收车有正确反馈
4. 摇杆输入能被后端正确识别为方向/转向意图

---

## 8. Phase 4：后端核心功能细化

### 8.1 阶段目标

在最小前端已经能验证接口的基础上，开始把后端最关键的核心功能做扎实。

### 8.2 本阶段优先细化的功能

按推荐顺序：

1. 急停
2. 安全收车
3. 开始/使能
4. 模式切换
5. 故障态
6. 看门狗与心跳

### 8.3 原因

这些功能决定系统“敢不敢让人用”。  
如果这些没有定稳，前端做得越多，返工越大。

### 8.4 本阶段应解决的问题

1. 急停是否锁存
2. 安全收车是否可观测
3. 开始失败如何回退
4. 模式切换如何先归零再切换
5. 故障态与急停态如何区分
6. 页面断连后如何归零

### 8.5 交付物

1. 稳定的状态机转移逻辑
2. 高优先级动作执行逻辑
3. 故障检测逻辑
4. 会话心跳逻辑

### 8.6 验收标准

必须验证：

1. 急停优先级最高
2. 收车过程中急停可抢占
3. 模式切换过程中禁止并发开始
4. 故障态禁止继续开始
5. 页面断连后不会持续输出旧手动指令

---

## 9. Phase 5：后端地图与任务功能细化

### 9.1 阶段目标

在高风险控制逻辑稳定后，再细化地图与自动任务相关功能。

### 9.2 本阶段优先完成的内容

1. 地图预览链
2. 地图保存按钮逻辑
3. 保存时命名逻辑
4. 地图索引管理
5. 地图选择逻辑
6. 自动探索建图模式逻辑
7. 自动漫游巡航模式逻辑

### 9.3 推荐依赖顺序

建议按以下顺序实现：

```text
地图预览
-> 地图正式保存
-> 保存时命名
-> 地图列表
-> 地图选择
-> 自动探索建图
-> 自动漫游巡航
```

### 9.4 为什么地图相关放在这个阶段

因为地图功能虽然重要，但优先级低于：

1. 急停
2. 收车
3. 模式切换
4. 状态机

如果地图管理先做得很细，而底层状态逻辑仍不稳定，会导致后续修改代价很高。

### 9.5 本阶段要特别注意的点

1. 预览图与正式地图文件分离
2. 自动保存与手动保存分离
3. `.pgm + .yaml` 保持为真值
4. 网页预览采用 `.png`
5. 地图显示名与底层路径分离

### 9.6 交付物

1. 地图预览生成逻辑
2. 地图保存与命名逻辑
3. 地图索引与选择逻辑
4. 自动模式基础任务逻辑

### 9.7 验收标准

必须验证：

1. 建图模式下可看到预览图
2. 点击保存时可命名地图
3. 保存后可进入正式地图列表
4. 选图后主状态区能显示地图名
5. 自动漫游巡航模式只使用已选地图

---

## 10. Phase 6：完整前端开发

### 10.1 阶段目标

在后端状态机、动作逻辑、地图功能都已经稳定后，再开发正式前端。

### 10.2 这个阶段才适合做什么

1. 页面布局优化
2. 极简风格统一
3. 移动端适配
4. 视觉层级优化
5. 按钮禁用/告警表现
6. 地图管理交互优化

### 10.3 为什么完整前端必须后置

因为前端的这些内容高度依赖：

1. 最终状态机定义
2. 最终接口字段
3. 最终错误码结构
4. 最终功能边界

如果这些还没稳定就开发完整前端，会导致：

1. 页面状态判断逻辑反复重写
2. 控件启用/禁用逻辑反复调整
3. 地图相关 UI 返工严重

### 10.4 正式前端应覆盖的完整功能

1. 系统状态区
2. 虚拟摇杆
3. 速度设置
4. 模式切换
5. 地图显示区
6. 地图选择
7. 地图命名
8. 地图保存按钮
9. 开始/急停/安全收车
10. 故障与锁定提示

### 10.5 交付物

1. 正式前端页面
2. 移动端可用交互
3. 完整状态显示与提示逻辑

### 10.6 验收标准

必须验证：

1. 非技术用户也能理解页面当前状态
2. 按钮语义与系统状态一致
3. 危险动作与普通动作视觉上有明显区分
4. 模式、地图、状态显示无歧义

---

## 11. Phase 7：联调、回归与试运行

### 11.1 阶段目标

将后端、ROS、正式前端三者做系统级联调，并在受控条件下试运行。

### 11.2 联调重点

1. 状态同步一致性
2. 页面刷新后恢复显示
3. 页面断连后的归零行为
4. 急停/收车/开始的边界行为
5. 模式切换中的互锁
6. 地图保存/选择链路

### 11.3 回归重点

每次改动后至少回归：

1. 手动控制
2. 手动建图
3. 自动探索建图
4. 自动漫游巡航
5. 地图保存
6. 地图选择
7. 开始
8. 急停
9. 安全收车

### 11.4 试运行边界

由于当前无物理急停断电按钮，试运行必须限制在：

1. 低速
2. 室内
3. 空旷场地
4. 人员在场
5. 随时可手工干预

### 11.5 交付物

1. 联调记录
2. 缺陷清单
3. 回归结果
4. 试运行结论

### 11.6 验收标准

至少达到：

1. 关键动作无逻辑歧义
2. 页面状态与后端状态一致
3. 模式切换可重复执行
4. 急停行为稳定
5. 地图链路完整可用

---

## 12. 后端优先的功能开发顺序建议

如果按“一个一个功能落地”的方式推进，建议后端严格按下面顺序开发：

1. 状态定义与状态管理骨架
2. 开始/急停/安全收车动作入口
3. ROS 适配骨架
4. 故障态与看门狗
5. 模式切换
6. 手动输入接收与超时归零
7. 地图预览链
8. 地图保存与命名
9. 地图索引与地图选择
10. 自动探索建图模式逻辑
11. 自动漫游巡航模式逻辑

不推荐的顺序是：

1. 先做完整前端
2. 先做地图页面美化
3. 先做复杂交互动效
4. 再补后端状态机

---

## 13. 前端开发顺序建议

前端建议分两轮：

### 第一轮：最小功能演示页

只做：

1. 状态显示
2. 模式切换按钮
3. 开始按钮
4. 急停按钮
5. 安全收车按钮
6. 最小摇杆区
7. 地图图片占位

### 第二轮：正式前端

在后端稳定后再做：

1. 页面整体布局
2. 速度设置区
3. 地图选择与命名弹窗
4. 地图保存按钮
5. 异常提示样式
6. 移动端优化

---

## 14. 每阶段推荐的验证方法

| 阶段 | 推荐验证方式 | 核心问题 |
|------|--------------|----------|
| Phase 1 | 单元级逻辑验证 | 状态机骨架是否合理 |
| Phase 2 | 命令行/接口测试 | 后端能否驱动 ROS |
| Phase 3 | 最小前端联调 | 状态和按钮是否对得上 |
| Phase 4 | 动作边界测试 | 急停/收车/开始是否正确 |
| Phase 5 | 地图链路测试 | 预览/保存/命名/选择是否打通 |
| Phase 6 | 前端交互验证 | 页面是否易用、无歧义 |
| Phase 7 | 实机回归 | 全链路是否稳定 |

---

## 15. 推荐里程碑

建议设置 5 个项目里程碑。

### Milestone 1：后端骨架完成

判定标准：

## 16. 当前最推荐的实际推进顺序

结合你当前要求，我给出的最推荐推进顺序是：

1. 先冻结现有两份设计文档
2. 先开发后端骨架
3. 先打通 ROS 最小闭环
4. 先做最小前端演示页
5. 用最小前端把后端高风险逻辑调通
6. 再做地图保存、命名、选择这些中风险功能
7. 后端功能稳定后，再开发完整前端
8. 最后做系统联调与实机试运行

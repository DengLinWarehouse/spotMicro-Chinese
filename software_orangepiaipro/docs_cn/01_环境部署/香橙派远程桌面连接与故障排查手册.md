# 香橙派远程桌面连接与故障排查手册

## 1. 文档说明

本文档基于本项目在香橙派上的实际排障过程整理，结合 `temp/远程桌面连接.txt` 的已验证内容，形成一份可复用的远程桌面连接说明。

适用场景：

- 香橙派使用 Ubuntu 桌面镜像
- Windows 端使用系统自带“远程桌面连接（RDP / mstsc）”
- 需要在香橙派上远程打开桌面、运行 RViz、查看图片或进行图形化操作
- 远程桌面偶发无法连接、3389 端口不通、`xrdp` 服务启动失败

本文重点不是只给出一套命令，而是把“连接过程中遇到的实际问题、判断方法和对应解决办法”系统梳理出来。

---

## 2. 推荐使用方式

在 SpotMicro 项目中，推荐这样分工：

- SSH：运行 `roscore`、雷达驱动、运动控制、建图、导航等核心 ROS 节点
- 远程桌面：运行 RViz、图片查看器、文件管理器等 GUI 程序

原因：

- SSH 更稳定，适合长期保持 ROS 节点运行
- 远程桌面容易受网络或服务状态影响，适合做可视化，不建议承载所有核心流程
- `rviz` 不能在普通无显示的 SSH 终端中直接运行，否则会出现 `could not connect to display`、`Could not load the Qt platform plugin "xcb"` 等错误

---

## 3. 连接远程桌面的基础条件

远程桌面要能正常连接，必须同时满足下面几项：

### 3.1 香橙派端

- 已安装 `xrdp`
- 已安装 `xorgxrdp`
- 已安装桌面环境，例如 `xfce4`
- `xrdp.service` 状态为 `active (running)`
- `xrdp-sesman.service` 状态为 `active (running)`
- 3389 端口处于监听状态
- 防火墙已放行 `3389/tcp`

### 3.2 Windows 端

- 与香橙派处于同一局域网
- 可以 `ping` 通香橙派 IP
- `Test-NetConnection <IP> -Port 3389` 返回 `TcpTestSucceeded : True`

---

## 4. 已验证的标准安装与恢复流程

以下流程适用于：

- 初次配置远程桌面
- `xrdp` 配置已经混乱
- 远程桌面此前能连、现在又突然不能连

所有命令均在香橙派 SSH 终端执行。

### 4.1 查看香橙派当前 IP

```bash
hostname -I
```

示例：

```bash
172.17.0.1 192.168.3.165
```

通常应使用局域网 IP，例如 `192.168.3.165` 作为 Windows 远程桌面的目标地址。

### 4.2 备份旧配置

```bash
TS=$(date +%Y%m%d_%H%M%S)
sudo mkdir -p /tmp/xrdp_backup_$TS
sudo cp -a /etc/xrdp /tmp/xrdp_backup_$TS/ 2>/dev/null || true
```

### 4.3 停止旧服务

```bash
sudo systemctl stop xrdp xrdp-sesman 2>/dev/null || true
```

### 4.4 卸载并清理旧 XRDP

```bash
sudo apt purge -y xrdp xorgxrdp
sudo rm -rf /etc/xrdp
```

### 4.5 重新安装 XRDP 与桌面相关组件

```bash
sudo apt update
sudo apt install -y xrdp xorgxrdp xfce4 xfce4-goodies
```

### 4.6 设置登录后默认进入 XFCE

```bash
echo xfce4-session > ~/.xsession
chmod +x ~/.xsession
```

### 4.7 给 xrdp 证书读取权限

```bash
sudo adduser xrdp ssl-cert
```

### 4.8 启用并重启服务

```bash
sudo systemctl daemon-reload
sudo systemctl enable xrdp
sudo systemctl enable xrdp-sesman
sudo systemctl restart xrdp-sesman
sudo systemctl restart xrdp
```

### 4.9 放行 3389 端口

```bash
sudo ufw allow 3389/tcp
sudo ufw reload
```

### 4.10 检查服务和端口

```bash
sudo systemctl status xrdp --no-pager
sudo systemctl status xrdp-sesman --no-pager
sudo ss -lntp | grep 3389
sudo ufw status
```

正常时应看到：

- `xrdp.service` 为 `active (running)`
- `xrdp-sesman.service` 为 `active (running)`
- `sudo ss -lntp | grep 3389` 有输出
- `3389/tcp` 已放行

## 5. Windows 端的正确验证方法

### 5.1 先不要急着点远程桌面

先在 Windows PowerShell 中测试端口是否真正打通：

```powershell
Test-NetConnection 192.168.3.165 -Port 3389
```

### 5.2 如何解读结果

如果结果是：

```powershell
PingSucceeded : True
TcpTestSucceeded : False
```

说明：

- 网络通
- 香橙派 IP 没错
- 但远程桌面服务没有真正提供 3389 端口

这时不要反复尝试 `mstsc`，而应返回香橙派排查 `xrdp` 服务。

如果结果是：

```powershell
TcpTestSucceeded : True
```

说明：

- `xrdp` 已经正常工作
- 可以再打开 `mstsc` 连接

### 5.3 Windows 连接参数

- 计算机：香橙派局域网 IP，例如 `192.168.3.165`
- 用户名：香橙派用户名，例如 `HwHiAiUser`
- 密码：该用户登录密码

## 6. 本次排障过程中遇到的典型问题与对应解决办法

下面是本次真实遇到的问题，以及最终验证有效的处理办法。

### 6.1 问题一：Windows 能 ping 通，但远程桌面连不上

现象：

- `ping 192.168.3.165` 正常
- `Test-NetConnection 192.168.3.165 -Port 3389` 失败

原因判断：

- 这说明网络正常
- 故障不在 Windows 端
- 问题在香橙派本机：`xrdp` 没有正常监听 3389

处理方法：

在香橙派上优先检查：

```bash
sudo systemctl status xrdp --no-pager
sudo systemctl status xrdp-sesman --no-pager
sudo ss -lntp | grep 3389
sudo ufw status
```

---

### 6.2 问题二：`ufw` 已放行 3389，但 Windows 仍连不上

现象：

- `ufw status` 中已有 `3389/tcp ALLOW`
- 但 `Test-NetConnection` 仍为 `False`

原因判断：

- 防火墙不是主因
- 根因仍然是 `xrdp` 服务没有正常启动

处理方法：

继续查看服务状态，不要误以为只要防火墙放行就一定能连接。

---

### 6.3 问题三：`xrdp.service` 和 `xrdp-sesman.service` 处于 failed

现象：

```bash
Active: failed (Result: exit-code)
```

原因判断：

- 远程桌面服务本身挂掉
- 因此 3389 根本没有监听

处理方法：

先采用“卸载 + 删除 `/etc/xrdp` + 重装”的方式恢复默认状态，见本文第 4 节。

说明：

- 在实际排障中，直接修修补补旧配置，不如彻底重建来得稳定
- 特别是已经多次失败过的情况下，重建更省时间

---

### 6.4 问题四：只监听 IPv6 或误以为是 IPv4/IPv6 问题

现象：

- 曾出现 `ss -lntp6 | grep 3389` 有输出
- `ss -lntp4 | grep 3389` 没输出

这类现象容易让人误以为是“xrdp 只监听 IPv6 导致 Windows 不能连”。

实际结论：

- 是否监听 IPv4/IPv6 需要结合 `Test-NetConnection` 和服务状态综合判断
- 如果 `xrdp.service` 本身是 failed，优先先修服务
- 不要一开始就盲改 `address=` 或强行修改监听参数

---

### 6.5 问题五：手工执行 `sudo /usr/sbin/xrdp -ns` 会卡住，但 systemd 启动失败

现象：

- 手工前台执行 `sudo /usr/sbin/xrdp -ns` 时程序不退出
- 但 `sudo systemctl restart xrdp` 却失败

原因判断：

- 说明 `xrdp` 程序本体并非完全损坏
- 更可能是“以服务用户身份启动时”出了问题
- `systemd` 服务使用的是 `User=xrdp`

这个现象非常重要，它帮助我们把问题范围缩小到了：

- 不是网络问题
- 不是主配置语法明显错误
- 而是 `xrdp` 作为服务用户启动时无法正常完成初始化

---

### 6.6 问题六：`sudo -u xrdp /usr/sbin/xrdp -ns` 报日志启动失败

实际关键报错：

```text
The log reference is NULL - log not initialized properly
Could not start log
error opening log file [The log is not properly started]. quitting.
```

这一步是整个排障过程中最关键的转折点。

根因判断：

- `xrdp` 服务以 `xrdp` 用户运行
- 启动时需要写自己的日志文件
- 但 `/var/log/xrdp.log` 不存在，或权限不对
- 导致 `xrdp` 无法初始化日志系统，启动直接失败

这解释了为什么：

- 配置文件看起来正常
- 防火墙也放行了
- 但服务就是起不来

---

### 6.7 问题七：日志文件不存在或权限不正确

最终验证有效的修复方法如下：

```bash
sudo touch /var/log/xrdp.log
sudo chown xrdp:adm /var/log/xrdp.log
sudo chmod 640 /var/log/xrdp.log

sudo touch /var/log/xrdp-sesman.log
sudo chown root:adm /var/log/xrdp-sesman.log
sudo chmod 640 /var/log/xrdp-sesman.log

sudo systemctl restart xrdp-sesman
sudo systemctl restart xrdp
```

修复后实际验证结果：

- `xrdp.service` 恢复为 `active (running)`
- `xrdp-sesman.service` 恢复为 `active (running)`
- `sudo ss -lntp | grep 3389` 出现监听
- Windows 端 `Test-NetConnection 192.168.3.165 -Port 3389` 返回 `TcpTestSucceeded : True`

这说明：

- 本次最终根因不是配置文件内容被修改
- 而是 `xrdp` 日志文件的创建与权限问题

---

### 6.8 问题八：是否已经“永久解决”

准确结论是：

- 当前故障已经修复
- 但如果以后再次重装 `xrdp`、清理日志、做系统维护，日志文件权限问题仍可能复发

因此建议增加一层固化措施。

最初建议的固化方向是：

```bash
sudo install -o xrdp -g adm -m 640 /dev/null /var/log/xrdp.log
sudo install -o root -g adm -m 640 /dev/null /var/log/xrdp-sesman.log
printf '/var/log/xrdp.log {\n    missingok\n    create 0640 xrdp adm\n}\n\n/var/log/xrdp-sesman.log {\n    missingok\n    create 0640 root adm\n}\n' | sudo tee /etc/logrotate.d/xrdp-local
```

作用：

- 明确日志文件的正确属主与权限
- 在日志轮转后也尽量按正确权限重建文件

建议后续再做两次验证：

1. 服务重启验证
   ```bash
   sudo systemctl restart xrdp-sesman
   sudo systemctl restart xrdp
   sudo systemctl status xrdp --no-pager
   sudo ss -lntp | grep 3389
   ```

2. 整机重启验证
   ```bash
   sudo reboot
   ```

重启后再检查：

```bash
sudo systemctl status xrdp --no-pager
sudo ss -lntp | grep 3389
```

Windows 再测：

```powershell
Test-NetConnection 192.168.3.165 -Port 3389
```

---

### 6.9 问题九：手工修好后，系统重启后再次失效

这是本次排障中最后定位出来的“真正持久化问题”。

现象：

- 手工执行修复命令后，`xrdp` 可以立刻恢复
- Windows 端 `Test-NetConnection 192.168.3.165 -Port 3389` 也能返回 `True`
- 但香橙派重启后，远程桌面再次失效
- 再次检查发现：
  - `/run/xrdp` 不存在
  - `/var/log/xrdp.log` 不存在
  - `/var/log/xrdp-sesman.log` 被错误地重建为 `root root` 或权限不对
  - `xrdp.service` / `xrdp-sesman.service` 再次 failed

这说明：

- 仅仅“把当前系统修好”还不够
- 还必须让系统在每次开机和每次服务启动前，自动创建正确的运行目录和日志文件

---

### 6.10 问题十：最终验证通过的持久化修复方案

最终验证通过的方案包含两层：

1. 用 `tmpfiles` 规则在系统运行期创建：
   - `/run/xrdp`
   - `/var/log/xrdp.log`
   - `/var/log/xrdp-sesman.log`
2. 用 `systemd override` 在每次服务启动前再次强制修正日志文件

#### 6.10.1 创建 tmpfiles 规则

```bash
sudo mkdir -p /etc/tmpfiles.d
cat <<'EOF' | sudo tee /etc/tmpfiles.d/xrdp.conf
d /run/xrdp 0755 xrdp xrdp -
f /var/log/xrdp.log 0640 xrdp adm -
f /var/log/xrdp-sesman.log 0640 root adm -
EOF
```

作用：

- 开机后自动准备 `/run/xrdp`
- 确保 `xrdp` 两个关键日志文件存在且权限正确

#### 6.10.2 为 `xrdp` 主服务添加 override

```bash
sudo mkdir -p /etc/systemd/system/xrdp.service.d
cat <<'EOF' | sudo tee /etc/systemd/system/xrdp.service.d/override.conf
[Service]
ExecStartPre=
ExecStartPre=/bin/sh /usr/share/xrdp/socksetup
ExecStartPre=/usr/bin/install -o xrdp -g adm -m 640 /dev/null /var/log/xrdp.log
EOF
```

作用：

- 保留原本的 `socksetup`
- 在每次启动 `xrdp` 前，重新按正确权限创建 `/var/log/xrdp.log`

#### 6.10.3 为 `xrdp-sesman` 添加 override

```bash
sudo mkdir -p /etc/systemd/system/xrdp-sesman.service.d
cat <<'EOF' | sudo tee /etc/systemd/system/xrdp-sesman.service.d/override.conf
[Service]
ExecStartPre=/usr/bin/install -o root -g adm -m 640 /dev/null /var/log/xrdp-sesman.log
EOF
```

作用：

- 在每次启动 `xrdp-sesman` 前，重新按正确权限创建 `/var/log/xrdp-sesman.log`

#### 6.10.4 应用并立即验证

```bash
sudo systemd-tmpfiles --create /etc/tmpfiles.d/xrdp.conf
sudo systemctl daemon-reload
sudo systemctl restart xrdp-sesman
sudo systemctl restart xrdp
sudo systemctl status xrdp --no-pager
sudo systemctl status xrdp-sesman --no-pager
sudo ss -lntp | grep 3389
ls -ld /run/xrdp
ls -l /var/log/xrdp.log /var/log/xrdp-sesman.log
```

成功标志：

- `xrdp.service` 为 `active (running)`
- `xrdp-sesman.service` 为 `active (running)`
- `3389` 正在监听
- `/run/xrdp` 存在且属主为 `xrdp:xrdp`
- `/var/log/xrdp.log` 为 `xrdp:adm`
- `/var/log/xrdp-sesman.log` 为 `root:adm`

#### 6.10.5 重启后再次验证

执行：

```bash
sudo reboot
```

重启后重新 SSH 登录，再执行：

```bash
sudo systemctl status xrdp --no-pager
sudo systemctl status xrdp-sesman --no-pager
sudo ss -lntp | grep 3389
ls -ld /run/xrdp
ls -l /var/log/xrdp.log /var/log/xrdp-sesman.log
```

Windows 再验证：

```powershell
Test-NetConnection 192.168.3.165 -Port 3389
```

本次实际验证结果：

- 重启后 `xrdp.service` 正常为 `active (running)`
- `xrdp-sesman.service` 正常为 `active (running)`
- `3389` 正常监听
- `/run/xrdp` 正常存在
- 两个日志文件权限保持正确
- Windows 端 `TcpTestSucceeded : True`

因此，这一套 `tmpfiles + systemd override` 才是本次最终验证通过的“持久化修复方案”。

### 6.11 问题十一：XRDP 端口正常，但登录后黑屏或闪退

在后续排障中，又出现了新的现象：

- Windows 端 `Test-NetConnection 192.168.3.165 -Port 3389` 返回 `True`
- 说明 `xrdp` 服务和 3389 端口都已经正常
- 但远程桌面登录后黑屏，或者一进入桌面马上闪退

这说明问题已经不在 `xrdp` 服务层，而在“桌面会话启动层”。

---

### 6.12 问题十二：`startxfce4` 段错误，根因是 Ascend / Python / ROS 环境污染

这一步是 XRDP 闪退问题的最终关键定位。

日志中的关键信息包括：

```text
Segmentation fault (core dumped) startxfce4
```

并且在 `~/.xsession-errors` 里持续看到大量环境变量被带入桌面会话，例如：

- `LD_LIBRARY_PATH`
- `PYTHONPATH`
- `ASCEND_TOOLKIT_HOME`
- `ASCEND_OPP_PATH`
- `ASCEND_AICPU_PATH`
- `TOOLCHAIN_HOME`

同时已经确认这些变量来自：

- `~/.bashrc`
- `/etc/profile`

实际检查结果中，至少存在：

- `~/.bashrc` 里导出了自定义 `LD_LIBRARY_PATH`
- `/etc/profile` 中全局导出了整套 Ascend 环境

由此可以得出结论：

- XRDP 桌面会话在启动时继承了 Ascend / Python / 动态库环境
- 这些变量改变了图形桌面程序的运行环境
- 最终导致 `startxfce4` / `xfwm4` 等程序崩溃

也就是说，这不是单纯的 XRDP 配置问题，而是：

- **开发环境变量全局注入，污染了图形桌面环境**

---

### 6.13 问题十三：为什么“干净启动脚本”仍然可能没有立刻解决

虽然已经尝试通过 `~/.xsession` 启动一个干净脚本，但日志仍然显示 Ascend 变量被带入了会话。

这说明：

- 污染源在更早阶段就已经进入登录环境
- 单纯在 `.xsession` 中 `unset` 变量，并不一定能彻底阻止桌面会话初始化过程中继承全局环境

所以最终正确的长期方案不是“反复在 XRDP 启动时清理变量”，而是：

- **从源头移除全局污染**

---

### 6.14 问题十四：最终建议的长期稳定方案

最终建议采用“按需加载开发环境”的方式，而不是把开发环境写入全局登录流程。

核心思路：

1. `XRDP` / 图形桌面登录使用干净系统环境
2. Ascend / Python / ROS 等开发环境只在需要时手动加载
3. 全局环境文件不要无条件导出 `LD_LIBRARY_PATH`、`PYTHONPATH`、`ASCEND_*`

这才是避免 XRDP 黑屏 / 闪退反复出现的长期稳定方案。

---

### 6.15 问题十五：永久修复方法

#### 6.15.1 先备份环境文件

```bash
cp ~/.bashrc ~/.bashrc.bak_$(date +%Y%m%d_%H%M%S)
sudo cp /etc/profile /etc/profile.bak_$(date +%Y%m%d_%H%M%S)
```

#### 6.15.2 注释掉 `~/.bashrc` 中自定义的 Ascend 动态库导出

```bash
sed -i 's#^export LD_LIBRARY_PATH=/home/HwHiAiUser/Ascend/acllib/lib64#\# export LD_LIBRARY_PATH=/home/HwHiAiUser/Ascend/acllib/lib64#' ~/.bashrc
```

#### 6.15.3 注释掉 `/etc/profile` 中全局 Ascend 环境


## 10. 本次排障的最终结论

本次远程桌面连接过程中，真正出现的问题可以分成两层：

第一层是表面现象：

- Windows 能 `ping` 通香橙派
- 但远程桌面连接不上
- `Test-NetConnection` 对 3389 失败

第二层是实际根因：

- `xrdp` 服务启动失败
- 3389 没有监听
- 最终定位为 `xrdp` 日志文件不存在或权限不正确，导致服务用户 `xrdp` 无法初始化日志系统

最终验证有效的解决办法是：

1. 重建 XRDP 环境
2. 设置 `~/.xsession`
3. 确保 `xrdp` 在 `ssl-cert` 组中
4. 创建并修正 `/var/log/xrdp.log` 与 `/var/log/xrdp-sesman.log` 的权限
5. 重启服务并验证 3389

最终成功标志为：

- `xrdp.service` = `active (running)`
- `xrdp-sesman.service` = `active (running)`
- `sudo ss -lntp | grep 3389` 有输出
- Windows `Test-NetConnection 192.168.3.165 -Port 3389` 返回 `TcpTestSucceeded : True`

最终稳定方案为：

- 重建 XRDP 基础环境
- 修复 `~/.xsession`
- 确保 `xrdp` 属于 `ssl-cert`
- 修复并固化日志文件权限
- 使用 `tmpfiles + systemd override` 解决系统重启后复发的问题
- 将 Ascend / Python / ROS 等开发环境从全局登录环境中拆出，改为按需手动加载

到此，香橙派远程桌面功能恢复正常，可继续用于 RViz、图片查看与图形化调试。

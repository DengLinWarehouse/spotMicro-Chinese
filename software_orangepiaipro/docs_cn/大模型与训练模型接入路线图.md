# SpotMicro 后续接入大模型与训练模型路线图

> 文档目的：将“后续接入大模型/训练模型”的分析结论固化为可执行路线，供后续立项、实施、验收和复盘使用。  
> 适用对象：负责 SpotMicro 机器狗后续扩展的系统工程、算法、嵌入式与集成维护人员。  
> 适用范围：`spotMicro-Chinese` 仓库当前已有的 ROS1 版本、Orange Pi 迁移版本、Standalone Python 版本。  
> 文档基线：2026-03。

---

## 1. 结论先行

### 1.1 推荐结论

这套 SpotMicro 仓库后续接入大模型或训练模型，**建议继续推进，但必须采用分层接入方案**：

1. **保留现有仓库作为执行层和技术验证层**
2. **把模型放在感知层、任务层、规划层**
3. **低层步态、逆运动学、标定、PWM 输出保持确定性控制**
4. **不要让大模型直接输出 12 路舵机命令**
5. **如果要做学习控制，优先训练输出 `cmd_vel`、姿态目标或足端目标，而不是直接输出 PWM**

### 1.2 不推荐结论

以下方向不建议作为当前平台的主攻路线：

- 让 LLM 直接闭环控制舵机
- 在当前硬件条件下做端到端 12 路 PWM 学习控制
- 把本仓库本身当作“训练平台主体”
- 为了接模型而持续投入大量时间打磨 ROS1 老环境

### 1.3 平台定位

当前 SpotMicro 更适合被定义为：

**“四足机器人低层执行与系统集成验证平台”**

而不是：

- 最终产品底座
- 现代化训练平台
- 高动态四足控制研究平台

## 2. 当前仓库能力与边界

## 2.1 当前已有能力

从仓库结构与代码实现看，当前系统已经具备以下基础能力：

| 能力 | 当前状态 | 关键位置 |
| --- | --- | --- |
| 低层步态/状态机控制 | 已具备 | `software_orangepiaipro/spot_micro_motion_cmd/` |
| 舵机标定与角度到 PWM 映射 | 已具备 | `spot_micro_motion_cmd/config/spot_micro_motion_cmd.yaml`、`software_standalone/spotmicro_standalone/hardware/servo_driver.py` |
| PCA9685 I2C 驱动 | 已具备 | `software_orangepiaipro/ros-i2cpwmboard/` |
| 键盘命令输入 | 已具备 | `spot_micro_keyboard_command/scripts/spotMicroKeyboardMove.py` |
| 基础 RViz / TF / 机体可视化 | 已具备 | `spot_micro_rviz/` |
| 激光雷达 + Hector SLAM 演示 | 已具备 | `spot_micro_launch/launch/motion_control_and_hector_slam.launch` |
| 纯 Python 无 ROS 控制栈 | 已具备 | `software_standalone/spotmicro_standalone/` |
| Mock 硬件后端 | 已具备 | `software_standalone/spotmicro_standalone/hardware/board.py` |

## 2.2 当前缺失能力

以下能力当前仓库没有形成可直接复用的完整链路：

| 能力 | 当前状态 | 对后续模型路线的影响 |
| --- | --- | --- |
| IMU 接入与姿态融合 | 缺失或仅文档提及 | 不利于学习控制与导航闭环 |
| 摄像头/视觉处理管线 | 缺失 | 视觉模型需自行补齐 |
| 足端接触/关节编码器/电流反馈 | 缺失 | 直接学习控制风险高 |
| 真实里程计与定位闭环 | 缺失 | 导航训练和策略评估难度高 |
| 成体系仿真环境 | 缺失 | Sim2Real 路线成本高 |
| 安全仲裁/命令仲裁/看门狗 | 较弱 | 多模型并存时风险高 |
| 训练数据记录规范 | 缺失 | 后续监督学习与离线评估效率低 |

## 2.3 核心工程事实

当前系统的低层执行链是：

```text
高层命令 -> 步态/姿态状态机 -> 逆运动学 -> 关节角 -> proportional -> PWM -> PCA9685 -> 舵机
```

其中：

- ROS1 版输入接口集中在 `/stand_cmd`、`/idle_cmd`、`/walk_cmd`、`/angle_cmd`、`/cmd_vel`
- Standalone 版输入接口集中在 `KeyboardCommandState -> Command -> Controller`

这意味着：

**现有仓库天然适合作为“确定性执行层”，并不天然适合作为“模型直接闭环执行层”。**

## 3. 仓库中的关键模型接入点

## 3.1 ROS1 版控制接入点

`software_orangepiaipro/spot_micro_motion_cmd/src/spot_micro_motion_cmd.cpp` 中已经固定了控制入口：

- `/stand_cmd`
- `/idle_cmd`
- `/walk_cmd`
- `/angle_cmd`
- `/cmd_vel`

这些接口已经足够承载大多数“高层模型接入”的需求。

### 推荐理解

- 任务模型：发模式命令
- 视觉模型：发目标偏差或速度命令
- 雷达/规划模型：发 `cmd_vel`
- LLM：发结构化高层动作，不发底层舵机值

## 3.2 ROS1 版状态输出点

现有可用于观察、记录和后处理的输出包括：

- `body_state`
- `lcd_state`
- `lcd_vel_cmd`
- `lcd_angle_cmd`
- `/tf`
- `/tf_static`
- 由命令积分得到的伪 `odom`

注意：

**当前 odom 不是传感器实测里程计，而是对命令的积分，不应当作为高可信训练标签或定位真值。**

## 3.3 Standalone 版模型接入点

`software_standalone/spotmicro_standalone/runtime/teleop.py` 的控制链更适合后续模型接入：

```text
KeyboardInput -> Command -> Controller -> SpotMicroStickFigure -> ServoDriver -> ServoBoard
```

其中最关键的替换点是：

- `interfaces/keyboard.py`
- `runtime/teleop.py`
- `pupper_src/Controller.py`

这意味着可以用：

- 网络输入
- UDP 输入
- 模型推理输出
- 录制回放输入

去替换原始键盘输入，而不用碰低层舵机映射。

## 3.4 现有过渡性接口

仓库中还存在一个轻量包：

- `software_orangepiaipro/extensions/packages/spot_micro_simple_command/`

它本质上说明系统作者已经尝试过绕过复杂状态机，直接通过“简单姿态命令 + IK + proportional”控制机器人。这是后续做模型中间层的一个重要信号：

**高层模型不一定非要直接改主状态机，可以先做中间命令节点。**

## 4. 推荐分层架构

建议把后续系统明确分成 4 层。

## 4.1 第 1 层：执行层

职责：

- 舵机标定
- IK
- 步态/状态机
- 角度限幅
- 舵机输出
- 急停

典型模块：

- `spot_micro_motion_cmd`
- `ros-i2cpwmboard`
- `software_standalone/spotmicro_standalone/hardware/servo_driver.py`

推荐控制频率：

- **50 Hz**

约束：

- 不接 LLM
- 不接大延迟模型
- 不接受未经检查的 12 维执行器输出

## 4.2 第 2 层：控制仲裁层

职责：

- 多来源命令仲裁
- 模式切换
- 超时停车
- 速度/姿态限幅
- 命令源优先级管理

建议新增模块：

- `spot_micro_command_arbiter`
- `spot_micro_safety_supervisor`

推荐运行频率：

- **10 到 20 Hz**

## 4.3 第 3 层：感知与智能层

职责：

- 摄像头模型推理
- 雷达特征提取
- 局部规划
- 语义决策
- 大模型任务编排

建议新增模块：

- `spot_micro_perception`
- `spot_micro_navigation`
- `spot_micro_mission`

推荐运行频率：

- 视觉模型：**5 到 15 Hz**
- 雷达局部逻辑：**5 到 10 Hz**
- LLM：**0.2 到 1 Hz**

## 4.4 第 4 层：离线训练与数据层

职责：

- 数据记录
- 标注
- 回放
- 训练
- 评估
- 模型版本管理

建议新增模块：

- `spot_micro_data_tools`
- `spot_micro_eval`

## 5. 技术路线对比

## 5.1 路线 A：LLM 做高层任务决策

### 方案定义

LLM 只负责任务理解、任务拆解和模式编排，不直接输出舵机命令。

### 推荐链路

```text
文本/语音/上位机指令
  -> LLM
  -> 结构化任务命令
  -> mission manager
  -> /stand_cmd /walk_cmd /idle_cmd /cmd_vel
```

### 适合场景

- 语音交互
- 巡逻/跟随/靠近/停止等模式切换
- 多步骤任务编排
- 汇报演示型功能

| 方案 | 原理概要 | 适用条件 | 性能预期 | 实现复杂度 | 调试难度 | 实机落地风险 |
| --- | --- | --- | --- | --- | --- | --- |
| 路线 A | LLM 只做高层任务决策 | 需要任务交互、语义控制 | 中 | 中 | 中 | 低 |

### 失效条件

- 让 LLM 直接输出 50 Hz 连续控制量
- 不做超时回退
- 不做规则校验和限幅

### 推荐结论

**推荐作为第一个大模型方向。**

---

## 5.2 路线 B：视觉模型输出目标/偏差，传统控制器执行

### 方案定义

视觉模型负责“看见什么”，传统控制器负责“怎么走”。

### 推荐链路

```text
相机 -> 检测/分类/跟踪模型 -> 目标偏差/目标状态 -> 速度控制器 -> /cmd_vel
```

### 适合场景

- 目标跟随
- 手势触发动作
- 指定目标靠近
- 视觉条件触发站立/行走/转向

| 方案 | 原理概要 | 适用条件 | 性能预期 | 实现复杂度 | 调试难度 | 实机落地风险 |
| --- | --- | --- | --- | --- | --- | --- |
| 路线 B | 感知模型输出目标状态，传统控制器输出 `cmd_vel` | 视觉交互、目标跟随 | 中高 | 中 | 中 | 中 |

### 失效条件

- 相机标定不准
- 不做丢目标保护
- 直接把检测框位置映射到关节命令

### 推荐结论

**推荐作为第一个训练模型方向。**

---

## 5.3 路线 C：雷达/地图 + 传统规划或 learned local planner

### 方案定义

利用现有雷达接入能力，做局部避障、建图与后续导航扩展。

### 推荐链路

```text
雷达 -> 预处理/建图/定位 -> 局部规划 -> /cmd_vel
```

### 当前仓库支撑

- 已有 `rplidar_ros`
- 已有 `hector_mapping`
- 已有 `lidar_link`
- 已有 `slam.launch` 和 `motion_control_and_hector_slam.launch`

但要明确：

**当前仓库有 SLAM 演示，不等于已有完整导航闭环。**

| 方案 | 原理概要 | 适用条件 | 性能预期 | 实现复杂度 | 调试难度 | 实机落地风险 |
| --- | --- | --- | --- | --- | --- | --- |
| 路线 C1 | 传统雷达 + 局部规划 | 室内慢速避障 | 中 | 中高 | 高 | 中 |
| 路线 C2 | 学习型局部规划输出 `cmd_vel` | 已有较好状态输入和安全壳 | 中 | 高 | 高 | 高 |

### 失效条件

- 把命令积分 odom 当真值
- 用漂移地图做强依赖路径规划
- 无碰撞边界和速度上限

### 推荐结论

- 先做 **局部避障**
- 后做 **巡线/绕障/墙跟随**
- 最后再考虑 learned planner

---

## 5.4 路线 D：模型输出姿态目标、足端目标或关节目标

### 方案定义

模型不直接输出 PWM，而是输出更高层、更有物理意义的目标：

- 机身姿态
- 足端轨迹
- 关节角

随后继续利用现有 IK、标定、PWM 映射层执行。

### 推荐链路

```text
模型 -> body pose / foot target / joint target -> IK -> proportional -> PWM
```

| 方案 | 原理概要 | 适用条件 | 性能预期 | 实现复杂度 | 调试难度 | 实机落地风险 |
| --- | --- | --- | --- | --- | --- | --- |
| 路线 D | 模型输出物理目标，保留现有执行层 | 有较好的观测、数据与安全壳 | 中 | 高 | 高 | 中高 |

### 失效条件

- 没有观测闭环
- 不做姿态可达域与足端可达域检查
- 不做异常回退

### 推荐结论

可以作为后续研究方向，但**不建议作为第一阶段主路线**。

---

## 5.5 路线 E：模型直接输出 12 路 PWM 或 proportional

### 方案定义

模型直接输出：

- 12 路 PWM
- 12 路 proportional
- 12 维关节直接动作

### 工程问题

当前平台存在以下硬约束：

- 舵机为 PWM 位置控制，不是高性能力控执行器
- 无关节编码器闭环
- 无足端力反馈
- 无高可信状态估计
- 有机械间隙、死区、温升和供电波动

| 方案 | 原理概要 | 适用条件 | 性能预期 | 实现复杂度 | 调试难度 | 实机落地风险 |
| --- | --- | --- | --- | --- | --- | --- |
| 路线 E | 模型直接控制 12 路执行器 | 当前平台不适合 | 低 | 高 | 很高 | 很高 |

### 失效条件

- 地面摩擦变化
- 负载变化
- 舵机温升
- 电池电压波动
- 接触状态变化

### 推荐结论

**不推荐。**

## 6. 推荐路线图

推荐按 6 个阶段推进。

## 阶段 0：执行层稳定化

### 目标

- 稳定站立
- 稳定慢走
- 稳定转向
- 统一标定参数
- 明确唯一舵机控制源

### 实施项

1. 固化 `spot_micro_motion_cmd.yaml` 与 standalone `default.yaml`
2. 明确 `servo_max_angle_deg`、`center_angle_deg`、`range` 为唯一可信参数源
3. 建立急停和超时回退规则
4. 统一记录命令和状态日志

### 成功标准

- 实机连续运行 5 分钟无异常抖动
- 切换 `stand -> walk -> stand -> idle` 无异常
- 同一台机器在 ROS 版与 standalone 版均能完成基础动作验证

---

## 阶段 1：统一高层命令接口

### 目标

把后续所有模型接入都收敛到统一接口，避免模型各自直接碰低层。

### 推荐统一命令结构

```text
mode: idle | stand | walk | angle_cmd
vx: m/s
vy: m/s
yaw_rate: rad/s
roll: rad
pitch: rad
source: keyboard | vision | lidar | llm | planner | replay
timestamp: s
timeout: s
```

### ROS 版建议

新增中间节点：

- `spot_micro_command_arbiter`
- `spot_micro_safety_supervisor`

由它们统一发布：

- `/stand_cmd`
- `/idle_cmd`
- `/walk_cmd`
- `/angle_cmd`
- `/cmd_vel`

### Standalone 版建议

新增模块：

- `interfaces/network.py`
- `interfaces/replay.py`
- `runtime/command_arbiter.py`

### 成功标准

- 键盘、脚本、模型输出都能经过同一仲裁层
- 任一命令源掉线后机器人能在 `0.3 ~ 1.0 s` 内减速并回退到安全状态

---

## 阶段 2：接入视觉模型

### 推荐切入点

先实现最简单且最有展示价值的能力：

- 目标检测
- 人体/手势识别
- 目标跟随

### 推荐结构

```text
camera
  -> detector / tracker
  -> target_state
  -> visual_follower
  -> command_arbiter
  -> /cmd_vel
```

### 模型输出建议

- `target_detected`
- `target_class`
- `target_confidence`
- `target_center_x`
- `target_distance`

### 控制建议

\[
yaw\_rate = k_{yaw} \cdot e_x
\]

\[
v_x = k_v \cdot e_d
\]

其中：

- `e_x` 为目标水平偏差
- `e_d` 为目标距离误差

### 参数初值建议

- `k_yaw: 0.6 ~ 1.2`
- `k_v: 0.1 ~ 0.3`
- `max_vx: 0.15 ~ 0.25 m/s`
- `max_yaw_rate: 0.15 ~ 0.25 rad/s`

### 成功标准

- 目标跟随成功率 > 80%
- 丢目标后 0.5 s 内减速到 0
- 误跟随率 < 10%

---

## 阶段 3：接入雷达与局部避障

### 推荐切入点

不要一上来做完整自主导航，先做：

- 近距离障碍物检测
- 碰撞方向禁止
- 局部速度裁剪

### 推荐结构

```text
scan
  -> lidar_safety_filter
  -> command_arbiter
  -> /cmd_vel
```

### 实现建议

定义扇区：

- 前方危险区：`[-25°, +25°]`
- 左右近障区：`[25°, 70°]`、`[-70°, -25°]`

定义阈值：

- 急停阈值：`0.20 ~ 0.25 m`
- 强减速阈值：`0.35 ~ 0.50 m`

速度裁剪建议：

- 障碍在正前方：裁剪 `vx`
- 障碍在左前方：裁剪左转与左移
- 障碍在右前方：裁剪右转与右移

### 成功标准

- 正前方固定障碍零碰撞
- 紧急停车响应 < 0.3 s
- 无模型条件下也能实现基础安全层

---

## 阶段 4：接入 LLM 任务管理器

### 推荐定位

LLM 只负责：

- 任务解释
- 任务分解
- 行为切换
- 异常解释

LLM 不负责：

- 高频控制
- 实时防撞
- 低层执行

### 推荐结构

```text
user request
  -> LLM
  -> mission manager
  -> structured action
  -> command arbiter
  -> execution layer
```

### 建议的结构化输出

```json
{
  "mode": "walk",
  "goal_type": "follow_target",
  "target": "person",
  "speed_limit": 0.15,
  "safety_policy": "indoor_safe",
  "timeout_s": 30
}
```

### 成功标准

- LLM 输出全部可被规则校验
- 非法命令被拦截
- LLM 故障或超时不影响底层安全运行

---

## 阶段 5：训练速度策略或局部规划策略

### 推荐目标

优先训练输出：

- `vx`
- `vy`
- `yaw_rate`

而不是关节或 PWM。

### 推荐训练数据

- `camera / lidar / state`
- 当前命令源
- 输出 `cmd_vel`
- 成功/失败标签
- 障碍距离特征

### 推荐训练方式

- 监督学习 imitation
- 小规模离线 RL 微调
- 回放评估优先于实机直上

### 推荐结构

```text
obs -> learned local policy -> cmd_vel -> safety supervisor -> execution layer
```

### 成功标准

- 推理延迟 < 100 ms
- 实机连续运行 5 分钟无跌倒
- 相对规则基线有明确收益

## 8. 安全与风险控制

## 8.1 必须具备的安全壳

后续只要接入模型，就必须补齐以下保护：

1. 命令超时停车
2. 速度限幅
3. 姿态限幅
4. 模型输出合法性检查
5. 命令源优先级仲裁
6. 手动急停
7. 模型异常时回退到 `stand` 或 `idle`

## 8.2 风险表

| 风险项 | 严重度 | 发生概率 | 触发条件 | 缓解措施 |
| --- | --- | --- | --- | --- |
| LLM 直接控低层导致失控 | 高 | 中 | LLM 直接输出连续控制 | LLM 只输出高层结构化命令 |
| 视觉模型误检引发撞击 | 高 | 中 | 误识别、丢目标 | 加置信度门限、超时停车、速度上限 |
| 雷达导航依赖伪 odom | 中高 | 高 | 把命令积分当真值 | 先做局部避障，不承诺全局导航 |
| learned policy 实机不稳 | 高 | 高 | 观测不足、无安全壳 | 先训 `cmd_vel`，再考虑姿态层 |
| 多命令源同时控制舵机 | 高 | 中 | 多节点同时发布低层命令 | 建立 command arbiter |
| 训练数据无统一标准 | 中 | 高 | 每个实验各自记录 | 建立统一日志格式与命名规则 |

## 11. 明确推荐与不推荐

## 11.1 明确推荐

当前约束下，推荐主路线为：

1. 先稳定低层执行层
2. 再做命令仲裁层
3. 先接视觉模型和雷达安全层
4. 再接 LLM 任务层
5. 最后再考虑 learned planner 或姿态策略

## 11.2 次选方案

如果 ROS1 环境维护成本过高，则建议：

- 用 `software_standalone` 作为主试验底座
- 只把需要雷达、RViz、bag 的部分留在 ROS1 版

## 11.3 不推荐方案

以下路线不推荐：

- 端到端输出 PWM
- 让 LLM 直接控制舵机
- 在当前平台上优先做 RL 步态而不先补观测和安全壳
- 把大量时间投入到 ROS1 老包适配，而不是投入到接口和分层改造

---

## 12. 最终判断

对当前 SpotMicro 平台，后续接入大模型/训练模型的最佳策略不是“替换底层”，而是“叠加高层”。

最值得优先做的顺序是：

1. **视觉模型 -> `cmd_vel`**
2. **雷达安全层 -> `cmd_vel` 裁剪**
3. **LLM -> 任务编排**
4. **learned local planner -> `cmd_vel`**
5. **姿态/足端策略 -> IK**

最不值得优先做的是：

1. **LLM -> PWM**
2. **RL -> 12 路舵机**
3. **不做安全壳直接实机闭环**

因此，本平台后续智能化的合理目标应定义为：

**“在 SpotMicro 低层执行验证平台之上，逐步构建可落地的感知、规划、任务管理和模型部署能力。”**
